## 第一阶段：认知建立

### 第 1 课：Claude Code 是什么，它和"聊天 AI"有什么本质区别

**课前一句话**：绝大多数人把 Claude Code 用成了"高级版 ChatGPT"，这门课要帮你看到它真正的边界在哪里。

#### 1.1 两种 AI 工具的本质差异

普通聊天 AI（Claude.ai 对话框、ChatGPT）做的是：
- 你说一句，它回一句
- 它没有"手"——无法读你的文件、改你的代码、跑命令
- 每次对话基本上都是从零开始

Claude Code 和 Cowork 做的是：
- 它能**读你机器上的真实文件**
- 它能**执行终端命令**（CLI）或**操作本地文件夹**（Cowork）
- 它会**记住这次任务的上下文**，多步骤连续推进
- 它可以**调用外部工具**（MCP 连接器、浏览器、数据库）

一句话概括：聊天 AI 是"顾问"，Claude Code / Cowork 是能动手的"执行者"。

#### 1.2 Claude Code 怎么工作的（内部原理，给非技术用户）

2026 年 3 月，Anthropic 的源码意外泄漏，技术社区从中发现了很多原来不为人知的细节。了解这些不是为了炫技，而是帮你理解"为什么要这样用它"。

**System Prompt 是动态组装的**

Claude Code 启动的时候，它的"系统指令"不是一段固定文字，而是由程序实时拼装出来的：
- 静态部分（身份定义、安全规则）走缓存，节省成本
- 动态部分（当前目录信息、你的 CLAUDE.md 内容、本次会话上下文）每次注入

这就是为什么 CLAUDE.md 写得好不好，会直接影响 Claude 在你项目里的表现。

**它有一个"情绪检测"系统**

源码中有一段专门用来检测用户沮丧情绪的逻辑——检测 "wtf"、"this sucks"、"不对啊"、"又错了" 这类词。有意思的是，这个系统用的是**正则表达式而不是 AI 判断**，因为工程师认为：这种高频触发的检测，用 AI 太贵了，用简单规则就够了。

这个设计细节告诉我们一件事：**Claude Code 团队在每一处都考虑了成本和效率的平衡**。你用它的时候，也应该有同样的意识。

**它有"上下文压缩"机制**

当会话内容达到上下文窗口的 87% 时，Claude Code 会自动触发压缩策略，把之前的对话压缩成摘要继续推进。这意味着超长任务里，早期的细节可能会"变淡"。后面第 9 课会讲怎么应对这个问题。

#### 1.3 Claude Code 适合谁

| 人群 | 最适合的入口 | 核心价值 |
|------|------------|---------|
| 开发者 | Claude Code CLI | 代码审查、重构、自动化脚本、仓库维护 |
| 产品经理 | Cowork | 需求文档、竞品分析、原型说明、数据整理 |
| 运营/市场 | Cowork | 内容生产、数据报告、资料归类、多文件处理 |
| 研究人员 | Cowork + CLI | 文献整理、数据分析、报告生成 |
| 创业者/独立开发 | CLI + Cowork 组合 | 从想法到产品的全链路 AI 协作 |

#### 1.4 学这门课之前先问自己三个问题

1. **我最想用 Claude Code 解决什么问题？** 越具体越好，不要说"提高效率"，要说"每周要处理 30 个竞品报告，现在手动要 3 天"。
2. **我更倾向于在终端工作，还是文件夹/界面工作？** 这决定了你应该先学 CLI 还是 Cowork。
3. **我现在最常用什么工具？** 知道自己的工具栈，才能判断哪些地方可以接入 MCP。

> **本课落地动作**：写下你的回答，保存好——这是后续所有课程的起点。

## 第二阶段：快速上手

### 第 2 课：安装、登录与环境配置

**课前一句话**：安装本身不难，难的是把"正确的上下文"配好——这才是影响你第一印象的关键。

#### 2.1 两条安装路径

**路径 A：Claude Code CLI（适合开发者和终端用户）**

前提条件：
- macOS / Linux / Windows（WSL 推荐）
- Node.js 环境
- Claude 账号（Pro 计划或 API Key）

基本步骤：
```bash
# 官方推荐安装方式（以当前文档为准）
npm install -g @anthropic-ai/claude-code

# 启动
claude
```

第一次启动后：
1. 完成登录/鉴权
2. 进入你的项目目录
3. 运行 `claude` 进入交互模式，或用 `claude "你的任务描述"` 单次执行

**路径 B：Cowork（适合知识工作者）**

前提条件：
- Claude Desktop（桌面端应用，不是网页版）
- Max 计划或 Team 计划（普通 Pro 计划功能受限）
- 目标文件夹的访问权限

启动步骤：
1. 打开 Claude Desktop
2. 在主界面找到 "Cowork" 入口
3. 创建 Project，关联文件夹
4. 开始第一个任务

**新手常见误区**：
- 用网页版 Claude 找 Cowork，找不到——Cowork 只在桌面端
- 用免费账号尝试长任务，中途断掉——Cowork 的持续任务需要付费计划
- 任务运行时关闭桌面端——任务会中断

#### 2.2 第一次配置：三件事先做好

**配置 1：设置工作目录（CLI）**

Claude Code CLI 默认在你启动命令时所在的目录工作。进错目录，后续所有输出都会出问题。

```bash
# 好习惯：进项目目录后再启动
cd /your/project/path
claude
```

**配置 2：设置 Project 和文件夹权限（Cowork）**

Cowork 的 Project 是"工作空间"的概念。一个 Project 可以关联：
- 本地文件夹（授权后 Claude 可读写）
- 上下文文件（你的工作说明、资料文档）
- Global Instructions（你的全局工作偏好）

第一次设置建议：
- 先用一个**不重要的测试文件夹**练手
- Global Instructions 里写："没有我的确认，不要删除任何文件"
- 第一周只做"读取和分析"任务，不让它修改文件

**配置 3：理解权限模式（CLI）**

Claude Code CLI 有三档权限：
- `default`：大多数操作需要你确认
- `acceptEdits`：文件修改自动通过，命令执行仍需确认
- `bypassPermissions`：全自动，高风险，新手不建议开

新手建议：先用默认模式，熟悉一两周后再考虑提权。

#### 2.3 验证任务：跑通你的第一个真实任务

**CLI 验证任务（⏱ 10 分钟）**

进入你自己的一个真实项目目录，输入：

```
帮我读一下这个项目的结构，告诉我：
1. 主要目录有什么
2. 入口文件在哪
3. 你觉得这个项目在做什么
先别改任何东西，只是告诉我你的理解。
```

观察：
- 它读文件准确吗？
- 分析结果具体吗（说出了文件名和路径）还是泛泛而谈？
- 有没有"瞎猜"的内容（没看到文件但说了内容）？

**Cowork 验证任务（⏱ 10 分钟）**

找一个包含 5-10 个文档的真实工作文件夹，让它：

```
读取这个文件夹里的文件，整理成一份要点清单：
每个文件的核心内容是什么，不要超过 3 句话。
先报告你读到了哪些文件，再给出汇总。
```

观察同上。

> **本课落地动作**：完成上面任意一个验证任务，截图保存，记下你觉得"它做得很好"和"它不太对"的地方各一条。

### 第 3 课：CLI vs Cowork——选错入口，再努力也是白费

**课前一句话**：这两个工具不是互相竞争，而是分工不同——关键是你要知道什么任务该交给谁。

#### 3.1 一张决策表

| 你想做的事 | 用 CLI | 用 Cowork |
|----------|--------|-----------|
| 读懂一个代码仓库 | ✅ 首选 | ❌ 不适合 |
| 改代码、修 Bug | ✅ 首选 | ❌ 不适合 |
| 跑终端命令、脚本 | ✅ 首选 | ❌ 不支持 |
| 整理本地文档文件夹 | ⚠️ 可以但别扭 | ✅ 首选 |
| 生成 PPT/Word/表格 | ❌ 不擅长 | ✅ 首选 |
| 处理竞品调研资料 | ❌ 不擅长 | ✅ 首选 |
| 定时任务、后台持续运行 | ✅（用脚本触发） | ✅（内置支持） |
| 需要打开浏览器完成任务 | ❌ | ✅（内置浏览器连接器） |
| 代码 + 文档一起处理 | ✅ + 共享目录协作 | ✅ + 读取代码文件夹 |

#### 3.2 两个场景对比

**场景一：写一份产品需求文档**

❌ 错误做法：打开 CLI，让它帮你写需求文档
→ 它会在终端输出一大段 Markdown，你还得手动复制粘贴

✅ 正确做法：打开 Cowork，关联你的"产品文档"文件夹，给它背景资料，让它直接生成文件
→ 文档生成在你的文件夹里，格式正确，可以直接打开编辑

**场景二：重构一个 Python 脚本**

❌ 错误做法：打开 Cowork，上传代码文件，让它改
→ 改完的代码在对话里，你还得手动复制回去；无法跑测试

✅ 正确做法：在项目目录里打开 CLI，让它读文件、改文件、跑测试
→ 修改直接落到文件里，还能立刻验证改对没有

#### 3.3 组合使用的时机

当任务横跨"文档处理"和"代码处理"时，可以组合使用：

**典型组合场景**：
- Cowork 先处理需求文档、竞品资料、设计说明
- 把整理好的上下文放进共享文件夹
- CLI 读取文件夹，根据这些上下文实现代码
- Cowork 再读取代码输出，生成用户文档

这个路径在第 12 课自动化工作流里会详细展开。

> **本课落地动作**：回顾你上周最耗时的 3 个工作任务，判断每个应该用 CLI、Cowork 还是组合。

## 第三阶段：核心工作流

### 第 4 课：5 步执行法——一次通过率倍增的关键

**课前一句话**：Claude Code 的问题往往不是能力不够，而是你给了它太大的自由度，或者太模糊的起点。

#### 4.1 为什么需要固定的执行流程

大多数人第一次用 Claude Code 时是这样的：
1. 脑子里有个任务
2. 直接打开，说"帮我做 XXX"
3. 它开始做，做到一半发现方向不对
4. 你打断它，重新说
5. 反复几次，花了 2 小时，得到的结果还没手动做的准

问题不是模型不行，是**指挥方式不对**。

固定流程的价值是：把自由度锁在正确的位置，减少你和 AI 的"方向对齐"成本。

#### 4.2 五步执行法

**第一步：读（理解现状）**

在开始任何修改之前，先让 Claude 读相关文件，说出它的理解。

```
先读一下 src/api/user.ts 和 src/services/auth.ts，
告诉我：
1. 这两个文件分别在做什么
2. 它们之间有什么关联
先别改，我需要确认你理解正确。
```

为什么这一步不能省：Claude 经常会在没完全理解结构的情况下开始修改，导致改了一个地方、破了另一个地方。

**第二步：出计划（确认范围）**

让它说出做法，你来审批，然后再执行。

```
好，理解准确。现在给我一个修改计划：
- 你打算改哪些文件
- 每个文件改什么
- 哪些地方你不确定，需要我确认

不要现在就改，先给我计划。
```

这一步能帮你发现 90% 的"跑偏"问题——在它动手之前。

**第三步：执行（范围内推进）**

计划确认后，再给执行指令。同时，主动限制不做什么。

```
计划没问题。按这个计划改，但注意：
- 不要重构其他文件
- 不要引入新的依赖
- 不要修改接口的入参和出参
```

**第四步：验证（确认结果正确）**

修改完成后，让它帮你验证，不要全靠眼看。

```
改完了，帮我做以下验证：
1. 读一遍改过的文件，说出你改了什么
2. 运行一下测试命令，看有没有报错
3. 告诉我这次改动有没有潜在风险
```

**第五步：总结（提取可复用知识）**

任务完成后，让它输出一份简短的总结。

```
任务完成，给我一个简短总结：
- 改了什么
- 为什么这样改
- 有没有遗留问题或待确认的地方
```

这个总结有两个用途：一是给你做记录，二是如果后续任务需要接续，这段总结可以作为新任务的上下文。

#### 4.3 "不做什么"清单：比目标更重要

一个任务描述里，禁止事项往往比目标更关键。

**通用禁止清单（可以直接复制用）：**

```
禁止事项：
- 不重构不相关的代码
- 不引入新的第三方依赖
- 不修改接口签名（入参、出参、类型）
- 不删除任何文件，除非我明确指定
- 不修改配置文件（.env、vite.config、tsconfig 等）
- 不向外部发送任何网络请求
- 遇到不确定的地方，先问我，不要自己决定
```

**经验：** 把你上次"它改错了"的地方，加进你的禁止清单。这个清单越用越准。

#### 4.4 一个完整案例

**任务：修复一个用户登录后跳转失败的 Bug**

```
# 第一步：读
读一下 src/pages/Login.tsx 和 src/router/index.ts，
告诉我登录成功后的跳转逻辑是怎么实现的。

# 第二步：出计划（等它回答后）
根据你的理解，给我一个修复计划。
只改跳转相关的逻辑，不要碰表单验证部分。

# 第三步：执行（计划确认后）
按计划修改。禁止事项：不改表单逻辑、不改路由配置。

# 第四步：验证
改完后读一遍改动，然后帮我想三个可能的边界情况，
比如"用户直接访问登录页但已经登录了"。

# 第五步：总结
给我一份两三句话的修改说明，我要粘贴进 commit message。
```

> **本课落地动作**：找一个你手头最近的真实任务，用 5 步执行法跑一遍，对比一下和之前的方式有什么不同。

### 第 5 课：Prompt 写法——Claude 听得懂的话，和你平时说的不一样

**课前一句话**：Anthropic 官方花了大量时间研究"如何和 Claude 说话"，他们的核心发现是：清晰、结构化、有上下文的指令，比聪明的措辞更重要。

#### 5.1 五个最有效的 Prompt 技巧

**技巧一：用 XML 标签组织结构**

Claude 对 XML 标签有很好的理解，用标签把不同类型的信息分开，可以大幅减少歧义。

```
# 不好的写法
帮我分析这段代码，找出 Bug，然后修复，用中文解释原因，别改其他地方

# 好的写法
<任务>分析下面这段代码，找出 Bug 并修复</任务>

<代码>
// 粘贴代码
</代码>

<要求>
- 用中文解释 Bug 的原因
- 只改有问题的部分
- 给我一个修改前后的对比
</要求>

<禁止>
不要改函数签名，不要引入新依赖
</禁止>
```

**技巧二：给"为什么"，不只给"什么"**

Claude 对"有动机的指令"反应比纯规则更好。解释你为什么要这样做，它会理解得更准。

```
# 不好：我需要你每次都先读文件再改
# 好：我需要你每次都先读文件再改，因为这个项目文件之间耦合很紧，
# 之前没读就改导致过几次事故
```

**技巧三：Few-shot 示例**

如果你有一个特定的输出格式要求，给它一个例子，比描述一百遍都有效。

```
<任务>给下面每个函数写注释</任务>

<示例格式>
/**
 * 计算用户的剩余额度
 * @param userId 用户 ID
 * @param month 查询月份 (YYYY-MM)
 * @returns 剩余额度（单位：元）
 */
</示例格式>

<代码>
// 你的代码
</代码>
```

**技巧四：长文档放前面**

Anthropic 测试发现：在给 Claude 一篇长文档后提问，把问题放在**文档后面**，性能比放前面高 30%。

```
# 低效写法
总结一下这篇文章的核心论点：[文章内容]

# 高效写法
[文章内容]

---
请总结上文的核心论点
```

**技巧五：让它思考后再输出**

对于复杂的判断类任务，让 Claude 先说思考过程，再给结论，准确率会明显更高。

```
帮我判断这段 SQL 有没有性能问题。
先列出你注意到的所有可能有问题的地方，
再根据这个分析给出你的结论和建议。
```

#### 5.2 常见的烂 Prompt 和改法

| 烂 Prompt | 问题 | 改法 |
|----------|------|------|
| "帮我优化代码" | 太模糊，不知道优化什么 | "帮我减少这段代码里的重复逻辑，保持功能不变" |
| "用最好的方式实现" | "最好"主观，会过度工程化 | "用最简单的方式实现，不考虑扩展性" |
| "帮我写完这个功能" | 边界不清，容易越界 | "实现登录表单的客户端验证，不涉及后端逻辑" |
| "改一下" | 什么都没说 | 给具体文件名、具体行数、具体期望结果 |
| 连续几段长对话后说"继续" | 上下文可能已经失真 | 新任务开新对话，重新给上下文 |

#### 5.3 一个结构化 Prompt 模板（可直接复制）

```
<背景>
[项目/任务的简短背景，3 句话以内]
</背景>

<任务>
[你想让它做什么，一句话概括]
</任务>

<输入>
[相关文件路径、代码片段、或文档内容]
</输入>

<要求>
- [要求 1]
- [要求 2]
- [输出格式要求]
</要求>

<禁止>
- [不要做的事 1]
- [不要做的事 2]
</禁止>

<示例>（可选）
[期望输出的一个例子]
</示例>
```

> **本课落地动作**：把你上周写过的一个 Prompt 用这个模板重写一遍，感受差异。

### 第 6 课：CLAUDE.md 与 Cowork 上下文——给 AI 装上"员工手册"

**课前一句话**：每次任务都从零解释上下文，效率极低，而且容易遗漏。把固定信息写进配置，是从"能用"到"好用"最重要的一步。

#### 6.1 两套配置系统的对应关系

| CLI（CLAUDE.md） | Cowork（上下文系统） | 作用 |
|-----------------|-------------------|------|
| `~/.claude/CLAUDE.md`（全局） | Global Instructions | 跨项目都生效的个人偏好 |
| 项目根目录 `CLAUDE.md` | Project Instructions + Context Files | 当前项目专属规则 |
| 子目录 `CLAUDE.md` | 文件夹内的上下文文件 | 更细粒度的模块规则 |

**简单类比**：
- Global Instructions = 公司全员培训手册
- Project Instructions = 部门工作规范
- 子目录/文件夹 = 具体岗位操作 SOP

#### 6.2 CLI：CLAUDE.md 标准结构

一份好用的 `CLAUDE.md` 通常包含以下六类信息：

```markdown
# 项目背景

[2-3 句话：这是什么项目，用来做什么，谁在用]

# 常用命令

\`\`\`bash
npm run dev       # 启动开发服务器
npm run build     # 构建生产版本
npm run test      # 运行测试
npm run lint      # 类型检查
\`\`\`

# 目录结构

[列出关键目录和它们的职责]

# 编码规范

- 语言：TypeScript 严格模式
- 格式化：使用 Prettier，不要手动调格式
- 注释：只在非自明的逻辑处写注释
- 命名：函数用 camelCase，组件用 PascalCase

# 禁止事项

- 不引入新的第三方依赖（需要先问我）
- 不修改 .env 文件
- 不改动 src/api/ 目录下的接口签名
- 不删除任何文件，除非我明确要求

# 当前任务上下文（定期更新）

[本周在做的功能/修复的问题，帮助 Claude 快速理解当前重点]
```

#### 6.3 Cowork：上下文配置实战

**Global Instructions 模板（适合大多数知识工作者）：**

```
你是我的工作助手。我的工作习惯如下：

输出偏好：
- 中文输出，专业词汇保留英文
- 结构化输出（用标题、列表而不是大段文字）
- 先结论，再解释
- 要点不超过 5 条，超过则归类

操作习惯：
- 没有我的确认，不要删除任何文件
- 任何写操作执行前，先告诉我你准备做什么
- 不确定的地方，直接问我，不要猜

当前工作重点：
[每周更新这里]
```

**Project Instructions 模板（以产品经理的竞品分析项目为例）：**

```
项目背景：
这个 Project 用于竞品分析工作。主要资料在"竞品资料"文件夹。

文件组织规范：
- 竞品资料/：原始截图和素材
- 分析报告/：整理后的分析文档
- 格式模板/：输出时参照这里的格式

输出要求：
- 每份分析报告参照"格式模板"里的结构
- 分析维度：功能、用户体验、定价、市场定位
- 数据来源必须注明（截图文件名或网址）

禁止事项：
- 不要在"竞品资料"文件夹写入任何新文件
- 报告草稿保存在"分析报告/草稿"子目录
```

#### 6.4 维护上下文的好习惯

1. **发现一次问题，立刻更新**：它做错了一件事，把禁止条件加进去，而不是每次都重新说
2. **每月回顾一次**：清掉已经过期的"当前任务上下文"，加入新的重点
3. **用 CLAUDE.md 记录"约定"**：和它多次来回讨论后定下来的规则，写进去，不然下次又得重新对齐

> **本课落地动作**：用今天学的模板，给你最常用的一个项目写一份 CLAUDE.md 或 Project Instructions。

## 第四阶段：进阶能力

### 第 7 课：Slash Commands——把高频动作变成一句话命令

**课前一句话**：高频任务的描述成本，是隐性的效率损耗。把它们收敛成命令，等于给团队建了一层轻量 SOP。

#### 7.1 三类控制工具的区别

| 工具 | 是什么 | 用来做什么 |
|------|--------|-----------|
| **Tools（工具）** | 内置的功能单元（读文件、写文件、跑命令） | Claude 自动调用，你感知不到 |
| **Slash Commands** | 你定义的可复用指令模板 | 用 `/命令名` 快速触发固定动作 |
| **Subagents** | 专注特定任务的子助手 | 复杂任务的拆分执行（第 10 课） |

#### 7.2 内置命令速查

| 命令 | 作用 |
|------|------|
| `/help` | 帮助 |
| `/clear` | 清空当前对话上下文 |
| `/compact` | 手动触发上下文压缩 |
| `/memory` | 查看/编辑当前记忆 |
| `/pr-comments` | 读取 PR 评论 |
| `/review` | 代码审查 |
| `/bug` | Bug 报告模板 |

#### 7.3 自定义命令：从零写一个 `/standup`

**场景**：每天早上 9 点，你需要整理一份站会消息——昨天做了什么、今天计划什么、有没有阻碍。以前要手动打开 Jira、看 git log、再写，现在可以一句话搞定。

**创建方式**：在 `.claude/commands/` 目录下创建文件 `standup.md`

```markdown
---
description: 生成今日站会消息
---

帮我整理一份站会消息，格式如下：

## 昨天
读取最近 24 小时内的 git commit 记录（用 git log --since="24 hours ago"），
列出主要完成的内容，每条不超过一句话。

## 今天
读取 TODO.md（如果存在），列出今日计划，没有则提示我手动填写。

## 阻碍
询问我是否有阻碍，如果有，格式化成一条描述。

输出格式：纯文本，适合直接粘贴进钉钉/飞书。
```

使用时：在 Claude Code 里输入 `/standup`，它会自动执行上面的逻辑。

#### 7.4 五个高频自定义命令范例

**1. `/commit` — 生成 git commit 消息**
```markdown
读取 `git diff --staged`，生成一条 commit 消息。
格式：<类型>(<范围>): <描述>
类型：feat/fix/refactor/docs/chore
描述：中文，不超过 50 字，说清楚做了什么。
```

**2. `/review` — 代码审查**
```markdown
读取 `git diff HEAD~1`，进行代码审查。
重点检查：逻辑错误、边界情况、潜在性能问题。
输出：按严重程度分级（必须改/建议改/可以考虑），每条附说明。
```

**3. `/doc` — 生成函数文档**
```markdown
读取 $ARGUMENTS 指定的文件，
为每个导出函数生成 JSDoc 注释。
要求：描述准确、参数类型正确、不超过 3 行。
```

**4. `/daily` — 日报生成**
```markdown
读取今日 git 提交和 TODO.md，生成工作日报。
格式：
- 完成事项（来自 git log）
- 未完成事项（来自 TODO.md 的未勾选项）
- 明日计划（来自 TODO.md 的下一步标注）
中文输出，简洁，适合内部汇报。
```

**5. `/debug` — 报错分析**
```markdown
分析 $ARGUMENTS 里的报错信息。
输出：
1. 报错原因（一句话）
2. 可能触发的代码位置
3. 三个可能的修复方向，按可能性排序
不要直接修改代码，先让我选方向。
```

> **本课落地动作**：把你最常重复描述的一个任务，写成一个自定义命令。

### 第 8 课：MCP 工具链——打通你常用的那些工具

**课前一句话**：Claude 本身很强，但真正的威力在于它能"伸出手"接触你的工具生态。MCP 就是这双手。

#### 8.1 MCP 是什么

MCP（Model Context Protocol）是 Anthropic 开发的开放协议，让 Claude 可以通过标准化接口连接外部工具和数据源。

你可以把它理解成：Claude 的"插件系统"，但不是装在 Claude 里，而是你在本地启动一个 MCP 服务器，Claude 通过协议连接它。

**已有官方/社区 MCP 服务器（部分列举）：**

| 工具 | 能做什么 |
|------|---------|
| Filesystem | 读写任意路径的文件（超越默认工作目录） |
| GitHub | 读 PR、Issues、代码，推送 commit |
| Slack | 读写消息频道，发送通知 |
| Notion | 读写 Notion 页面和数据库 |
| 飞书/钉钉 | 读写消息、文档（社区实现） |
| Browser | 打开网页、截图、操作浏览器（Playwright） |
| PostgreSQL / MySQL | 执行 SQL 查询 |
| Google Drive / Sheets | 读写云端文件 |

#### 8.2 配置 MCP 的基本步骤

1. 确认你想接入的工具有 MCP 服务器（搜索 `mcp server [工具名]`）
2. 按服务器文档安装依赖
3. 在 Claude Desktop 设置里添加 MCP 配置
4. 重启 Claude Desktop，验证连接

**配置示例（Claude Desktop 设置文件）：**

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/你/的/工作目录"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "你的_token"
      }
    }
  }
}
```

#### 8.3 最值得先接入的 3 个 MCP

**1. Filesystem MCP（开发者和知识工作者都推荐）**

最实用的第一步，让 Claude 可以读写你指定目录里的所有文件，不再受单一工作目录限制。

**2. GitHub MCP（开发者必装）**

直接在 Claude 里完成 PR review、Issue 整理、代码搜索，不用来回切换界面。

**3. Browser/Playwright MCP（研究和运营人员推荐）**

让 Claude 可以打开网页、截图、提取内容，用于竞品调研、数据采集、自动化测试。

#### 8.4 MCP 使用注意事项

- MCP 服务器运行在你的本地机器，需要保持运行才能使用
- 给 MCP 服务器的权限要最小化：只开放你真正需要的目录/资源
- 连接外部服务（GitHub、Slack）的 MCP 需要 Access Token，妥善保管
- 定期检查 MCP 服务器日志，发现异常立即断开

> **本课落地动作**：选一个你最常用的工具，找到对应的 MCP 服务器，尝试接入并完成一个小任务验证。

### 第 9 课：上下文管理——不让 AI 越跑越"忘事"

**课前一句话**：Claude 的上下文窗口是有限的，当你不主动管理它时，长任务里的 AI 会越来越像一个失忆的助手。

#### 9.1 理解上下文窗口

Claude Code 的上下文窗口（大约 20 万 token，约 15 万中文字）听起来很大，但在实际工作中会很快用完：
- 每次读一个大文件就消耗几千 token
- 来回对话本身也在累积
- 错误信息、命令输出都会占空间

当上下文占用达到 **87%** 时，Claude Code 会**自动压缩**——把之前的对话总结成摘要，腾出空间继续。

这个机制的问题是：**压缩是有损的**，早期对话里的细节可能被压掉。

#### 9.2 四个主动管理上下文的技巧

**技巧一：任务完成后用 `/clear` 开新对话**

一个任务一个对话。不要让所有任务都堆在同一个上下文里。

```
# 规则
修复一个 Bug → /clear
开始下一个功能 → 新对话，重新给上下文
```

**技巧二：用 CLAUDE.md 代替重复说明**

你每次都要重新解释的上下文，永远是 CLAUDE.md 里应该有的内容。

如果你在一次对话里说了超过 3 次同样的话，那就是该写进 CLAUDE.md 的信号。

**技巧三：长任务用"分段交付"模式**

把大任务切成多个里程碑，每个里程碑完成后让它输出总结，用总结开启下一段。

```
# 大任务：重构整个用户模块

# 对话 1：分析现有结构
分析 src/user/ 目录，给我一份重构建议报告。
[任务完成后，保存报告]

# 对话 2：执行重构（用报告作为上下文）
这是我们上次分析的重构建议：[粘贴报告]
按照第 2 条建议，先重构 UserService。
```

**技巧四：定期 `/compact` 手动压缩**

当对话较长但任务还没完成时，可以手动触发压缩，让 Claude 整理当前上下文，去掉无关内容。

#### 9.3 上下文重置的信号

当你观察到以下情况，应该考虑重置上下文：

- 它开始重复之前说过的分析
- 它"忘记"了你之前确认的约束条件
- 它给的建议和 20 分钟前的建议明显矛盾
- 回复质量明显下降，开始模糊化处理

> **本课落地动作**：回顾一次你觉得 Claude 后期"越来越差"的任务，判断是不是上下文管理问题。

## 第五阶段：Agent 架构

### 第 10 课：Multi-Agent——一个人指挥多个 AI 分工干活

**课前一句话**：单个 Agent 的瓶颈是它只有一个视角、一个线程。当任务真正复杂时，需要多个 Agent 各司其职。

#### 10.1 什么时候需要 Multi-Agent

**不需要 Multi-Agent 的情况（先做好单 Agent）：**
- 任务是线性的，一步一步来就行
- 任务范围在一个目录内
- 你还在熟悉 Claude Code 阶段

**适合 Multi-Agent 的情况：**
- 任务可以拆成互相独立的子任务，并行处理
- 单一 Agent 的上下文窗口装不下所有需要的信息
- 需要一个"质检员"角色来检查主 Agent 的输出
- 任务有明确的分工边界（研究/撰写/审查/发布）

#### 10.2 三种主要的 Multi-Agent 架构

**架构 1：主-从（Orchestrator + Workers）**

一个协调者 Agent 拆分任务，分发给多个执行者 Agent，最后汇总结果。

```
主 Agent：
"我需要写一份竞品分析报告，分析三家公司。
请分别启动三个子任务：
- 子任务 A：分析 A 公司产品功能
- 子任务 B：分析 B 公司产品功能
- 子任务 C：分析 C 公司产品功能
三个任务完成后，把结果汇总成一份对比报告。"
```

**架构 2：流水线（Pipeline）**

任务在多个 Agent 之间顺序流转，每个处理一段，结果传给下一个。

```
Agent 1（研究）→ Agent 2（撰写）→ Agent 3（审查）→ Agent 4（格式化）
```

**架构 3：主-质检（Writer + Reviewer）**

一个 Agent 生成内容，另一个专门检查，形成内部复核机制。

```
写作 Agent："写一份技术方案文档"
↓
审查 Agent："读取上面的文档，检查：
1. 技术方案是否有遗漏的边界情况
2. 语言是否清晰
3. 结论是否有足够支撑"
```

#### 10.3 Claude Code 内置的 Subagents

Claude Code 源码里有几个内置子 Agent 角色（从泄漏代码中提取）：

| 子 Agent | 职责 |
|---------|------|
| **Plan** | 复杂任务前做深度规划，输出执行计划 |
| **Explore** | 快速探索代码库，搜索关键词和文件 |
| **Task** | 执行具体的子任务，有独立的工具调用权限 |

在实际使用中，你可以通过描述任务来触发这些角色：

```
# 触发 Plan 模式
这是一个复杂的重构任务，请先做深度分析和规划，
列出完整的执行步骤和风险点，不要立刻开始执行。

# 触发 Explore 模式
快速扫描整个项目，找出所有和"用户认证"相关的文件和函数。

# 触发 Task 模式
把以下三个任务作为独立子任务执行：
1. 更新 README
2. 更新 CHANGELOG
3. 跑 lint 检查
```

#### 10.4 新手路径建议

**阶段 1**（前两周）：把单个 CLI 或 Cowork 用好，能稳定完成 5 步执行法。

**阶段 2**（第三四周）：加入 CLAUDE.md，开始写自定义 Slash Commands，减少重复描述。

**阶段 3**（第二个月）：接入 1-2 个 MCP，打通最常用的外部工具。

**阶段 4**（稳定后）：尝试简单的主-质检双 Agent 模式，再逐渐引入更复杂的协作。

不要在单工具都没跑稳的时候直接上多智能体，会适得其反。

> **本课落地动作**：找一个你觉得"光靠自己说太费劲"的复杂任务，画出一个简单的 Multi-Agent 分工图。

### 第 11 课：安全边界——危险能力怎么放权又不失控

**课前一句话**：Claude Code 能删文件、推代码、发请求，这些都是高价值能力——问题不是关闭它们，而是把触发条件和确认点设清楚。

#### 11.1 Claude Code 的"危险能力清单"

Claude Code 可以做到，但有风险的操作：

| 操作 | 风险级别 | 建议策略 |
|------|---------|---------|
| 删除文件 | 高 | 必须人工确认，先备份 |
| git push | 高 | 必须人工确认，只允许特定分支 |
| 安装依赖包 | 中 | 列出清单后人工确认 |
| 写入生产数据库 | 高 | 禁止，使用测试环境 |
| 发送外部网络请求 | 中-高 | 事先说明目标地址，人工确认 |
| 修改环境配置文件 | 高 | 禁止，除非专项任务 |
| 批量重命名/移动文件 | 中 | 先列出变更清单，确认后执行 |

#### 11.2 写进 CLAUDE.md 的安全边界模板

```markdown
# 安全边界

## 必须人工确认的操作（停下来等我说 OK 才能继续）
- git push（任何分支）
- 删除文件（哪怕是临时文件）
- 安装新的 npm/pip 依赖
- 修改 .env、.env.local 等环境配置文件
- 发送任何外部网络请求

## 禁止操作（不管什么情况都不能做）
- 访问 src/config/secrets.ts
- 操作 production 分支
- 执行 DROP TABLE 等破坏性 SQL

## 操作前必须完成
- 列出本次任务会修改的所有文件
- 如果涉及超过 5 个文件，先给我一个变更计划
```

#### 11.3 黄金操作习惯

**任务前：备份**
```bash
git stash  # 或 git checkout -b backup/[时间]
```

**任务后：检查 diff**
```bash
git diff   # 看改了什么
git status # 看有没有意外的文件变动
```

**发现问题时：恢复**
```bash
git stash pop     # 恢复 stash
git checkout .    # 放弃所有修改（慎用）
git revert [hash] # 撤销特定 commit
```

#### 11.4 一个关键原则

**不靠信任模型，靠最小化可恢复成本。**

真正稳定的 AI 编程习惯，不是让模型变得更可信，而是确保每一步操作都可以回退。在这个前提下，才值得逐步放开权限。

> **本课落地动作**：把今天的安全边界模板加进你的 CLAUDE.md。

### 第 12 课：自动化工作流——让 Claude Code 在后台替你盯着事

**课前一句话**：当你把触发条件、执行逻辑和安全边界都设置清楚后，就可以让 Claude Code 在你不在的时候自动完成某些工作。

#### 12.1 Hooks 机制：为工具调用设置钩子

Claude Code 源码中有一套 Hooks 系统，允许你在特定时机自动执行操作：

| Hook 类型 | 触发时机 | 使用场景 |
|----------|---------|---------|
| `PreToolUse` | 工具调用前 | 权限检查、日志记录、危险操作拦截 |
| `PostToolUse` | 工具调用后 | 结果验证、通知发送、后处理 |
| `Notification` | Claude 发送通知时 | 转发到钉钉/飞书/Slack |

**配置 Hook 示例（钉钉通知）：**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "node scripts/notify-dingtalk.js '$TOOL_OUTPUT'"
        }]
      }
    ]
  }
}
```

#### 12.2 三个可落地的自动化场景

**场景 A：每日代码健康巡检**

```bash
# 定时任务脚本（每天早上 8 点运行）
#!/bin/bash
cd /your/project
claude --no-confirm "
读取过去 24 小时的 git log 和当前 lint 报告，
生成一份简短的代码健康日报：
- 昨日提交摘要
- 当前 lint 警告数量
- 需要关注的风险点
输出到 reports/daily-$(date +%Y%m%d).md
"
```

**场景 B：PR 自动预审**

```bash
# 在 git pre-push hook 里加入
#!/bin/bash
claude "
读取 git diff main...HEAD，
进行代码预审：
1. 有没有明显的 Bug
2. 有没有遗忘的 TODO
3. 有没有敏感信息（token、密码）
结论：是否可以安全推送
如果有严重问题，输出 EXIT_FAIL
"
```

**场景 C：文档与代码同步检查（Cowork + CLI 协作）**

```
# Cowork 任务
读取 docs/ 文件夹里的 API 文档，
提取所有接口的参数和返回值，
生成一份"接口契约清单"保存到 reports/api-contract.md

# CLI 任务（接续）
读取 reports/api-contract.md 和 src/api/ 目录，
检查代码实现和文档描述是否一致，
不一致的地方列出来，不要自动修改。
```

#### 12.3 新手自动化路径

**阶段 1**：先用手动触发（自己输入命令），验证任务流程可靠。

**阶段 2**：把验证过的流程做成 Slash Command，一句话触发。

**阶段 3**：把稳定的 Slash Command 加入定时触发脚本。

**永远不要**：跳过手动验证阶段直接搞全自动化。没有验证过的流程自动化后出问题，排查成本是手动的十倍。

> **本课落地动作**：选一个你每周都手动做的重复任务，用 Slash Command 把它半自动化。

### 第 14 课：实战案例 B——开发者的代码仓库维护流程

**场景描述**：你是一名后端开发者，独立维护一个中型 Node.js 项目，面临：代码文档缺失、技术债务积累、PR review 质量参差不齐。

#### 工作流设计

**CLAUDE.md 配置（关键部分）：**

```markdown
# 项目背景
一个 Node.js RESTful API 服务，接 PostgreSQL，约 30 个接口，
无全职测试人员，靠代码审查保证质量。

# 最重要的规则
1. 每次修改前先读相关文件
2. 改动超过 3 个文件的任务，先给计划
3. 不能修改 src/db/migrations/ 里的文件
4. 不能改动 package-lock.json（除非专门升级依赖任务）

# 当前技术债务（持续更新）
- 用户模块缺少 JSDoc 注释
- tests/ 里的集成测试覆盖率 < 30%
- src/utils/ 有几个废弃函数没清理
```

**Slash Commands 套装：**

```markdown
# .claude/commands/review.md
读取 `git diff main...HEAD`
进行代码审查，重点关注：
1. 是否有 SQL 注入风险（重点！）
2. 错误处理是否完整
3. 日志是否足够排查问题
输出：问题列表，按严重程度排序（阻断/警告/建议）

# .claude/commands/doc.md
为 $ARGUMENTS 文件里所有导出的函数补全 JSDoc 注释。
要求：参数类型准确，返回值说明清晰，不超过 5 行。
改完后展示修改摘要，等我确认再写入文件。

# .claude/commands/debt.md
读取 CLAUDE.md 里的"当前技术债务"列表，
结合代码现状，给出本周优先清理哪 1-2 项的建议，
说明理由（影响范围、清理难度、收益）。
```

**周度维护工作流（每周五下午）：**

```
第一步：/debt → 选出本周要处理的技术债务项
第二步：5 步执行法完成清理任务
第三步：/doc [修改的文件] → 补充文档
第四步：/review → 最终自检
第五步：生成 commit 消息，推代码
```

## 附录 A：常见报错与排查手册

| 现象 | 可能原因 | 解决方法 |
|------|---------|---------|
| CLI 启动后立刻报错 | 安装不完整或 Node 版本不符 | 重新安装，确认 Node 版本 |

## 附录 D：源码启示录——从泄漏代码看懂 Claude 真正的工作方式

2026 年 3 月 31 日，Anthropic 因为打包失误，意外将 Claude Code 的完整 TypeScript 源码发布到 npm。技术社区从 51 万行代码里挖出了很多有价值的细节，这里整理其中对使用者最有启发的几条。

**启示 1：你的 CLAUDE.md 真的是"配置文件"**

源码显示，Claude Code 启动时，系统提示词是动态组装的：固定规则部分走缓存，而你写的 CLAUDE.md 内容会被每次注入。这意味着 CLAUDE.md 改一个字，对话行为就会真实改变——它不是装饰，是生效的配置。

**启示 2：87% 上下文占用时会自动压缩**

这个阈值是代码里写死的。压缩是有损的，早期对话细节可能消失。所以超长任务里，越重要的约束越应该写在 CLAUDE.md 里，而不是只在对话里说过一次。

**启示 3：它在检测你的情绪**

代码里有一个正则表达式系统，专门匹配用户的沮丧词汇（"怎么又错了"、"不对啊"等），在这种情况下调整回复策略。这告诉我们：当任务卡住时，换一种更平静、结构化的表述，往往比反复催促效果更好。

**启示 4：工具调用有安全分级**

源码里每个工具都有明确的"危险等级"标注，开发者必须显式声明哪些操作是安全的。这和我们建议你在 CLAUDE.md 里写禁止事项的逻辑完全一致——系统层面和使用者层面都在做同样的事情。

**启示 5：记忆是主动管理的**

源码中的 `autoDream` 机制会在你空闲 24 小时后，自动把本次会话的零散观察整合成结构化知识。这告诉我们：不要试图通过不断重复来让它"记住"，而是用 CLAUDE.md 做主动的、持久的上下文注入。
