# Claude Code & Cowork 从入门到精通

> 一套结合理论与实践的中文课程，面向想把 Claude Code 和 Cowork 真正用进日常工作的中国用户。
>
> **课程理念**：先建立正确认知，再跑通真实任务，最后系统化成可复用的个人工作流。

## 第一阶段：认知建立

### 第 1 课：Claude Code 是什么，它和"聊天 AI"有什么本质区别

**课前一句话**：绝大多数人把 Claude Code 用成了"高级版 ChatGPT"，这门课要帮你看到它真正的边界在哪里。

#### 1.1 两种 AI 工具的本质差异

普通聊天 AI（Claude.ai 对话框、ChatGPT）做的是：
- 你说一句，它回一句
- 它没有"手"——无法读你的文件、改你的代码、跑命令
- 每次对话基本上都是从零开始

Claude Code 和 Cowork 做的是：
- 它能**读你机器上的真实文件**
- 它能**执行终端命令**（CLI）或**操作本地文件夹**（Cowork）
- 它会**记住这次任务的上下文**，多步骤连续推进
- 它可以**调用外部工具**（MCP 连接器、浏览器、数据库）

一句话概括：聊天 AI 是"顾问"，Claude Code / Cowork 是能动手的"执行者"。

#### 1.2 Claude Code 怎么工作的（内部原理，给非技术用户）

先用官方可以长期成立的心智模型来理解 Claude Code：

**它不是一个聊天框，而是一套"带工具的工作代理"**

Claude Code 每次处理任务，本质上都在做四件事：
- 读取当前上下文：你的提示词、当前目录、`CLAUDE.md`、规则文件、可用工具
- 判断接下来要不要调用工具：读文件、搜代码、跑命令、调用 MCP、启动子 Agent
- 在权限模式约束下执行动作
- 把结果写回当前会话，并在必要时更新 memory

这就是为什么同样一句"帮我修这个 Bug"，在不同项目、不同权限模式、不同 `CLAUDE.md` 下，表现会差很多。

**它有两套长期记忆机制**

- `CLAUDE.md` / `.claude/rules/`：你写给它的长期指令
- Auto Memory：Claude 在多次协作后，自己总结出来的经验和偏好

官方现在明确把这两者都定义为"context"，不是强制执行配置。也就是说，写得越具体、越短、越一致，遵循度越高；写得越模糊、越冲突，效果越不稳定。

**它会主动管理上下文，但你不能把这件事完全交给它**

Claude Code 会在长会话里做上下文整理、压缩、重组，避免任务中途彻底失忆。但压缩天然是有损的，所以真正重要的规则不要只说在对话里，应该写进 `CLAUDE.md`、rules 或 skill 里。后面第 9 课会系统讲这个问题。

#### 1.3 Claude Code 适合谁

| 人群 | 最适合的入口 | 核心价值 |
|------|------------|---------|
| 开发者 | Claude Code CLI | 代码审查、重构、自动化脚本、仓库维护 |
| 产品经理 | Cowork | 需求文档、竞品分析、原型说明、数据整理 |
| 运营/市场 | Cowork | 内容生产、数据报告、资料归类、多文件处理 |
| 研究人员 | Cowork + CLI | 文献整理、数据分析、报告生成 |
| 创业者/独立开发 | CLI + Cowork 组合 | 从想法到产品的全链路 AI 协作 |

#### 1.4 学这门课之前先问自己三个问题

1. **我最想用 Claude Code 解决什么问题？** 越具体越好，不要说"提高效率"，要说"每周要处理 30 个竞品报告，现在手动要 3 天"。
2. **我更倾向于在终端工作，还是文件夹/界面工作？** 这决定了你应该先学 CLI 还是 Cowork。
3. **我现在最常用什么工具？** 知道自己的工具栈，才能判断哪些地方可以接入 MCP。

> **本课落地动作**：写下你的回答，保存好——这是后续所有课程的起点。

### 第 2 课：安装、登录与环境配置

**课前一句话**：安装本身不难，难的是把"正确的上下文"配好——这才是影响你第一印象的关键。

#### 2.1 两条安装路径

**路径 A：Claude Code CLI（适合开发者和终端用户）**

前提条件：
- macOS / Linux / Windows
- Claude 账号（按你使用的入口，可能是 Pro / Max / Team / Enterprise / API）
- 如果使用 npm 安装方式，需要 Node.js 环境

推荐安装顺序：
```bash
# macOS（官方推荐之一）
brew install anthropics/tap/claude-code

# Windows（官方推荐之一）
winget install Anthropic.ClaudeCode

# 跨平台通用备选
```

前提条件：

**新手常见误区**：
- 还没分清 CLI、本地桌面、云端 web session、remote control 的差别
- 一上来就开高权限或 auto mode，没建立回退习惯
- 在错误目录里启动 Claude，导致它读了不该读的项目

#### 2.2 第一次配置：三件事先做好

**配置 1：设置工作目录（CLI）**

Claude Code CLI 默认在你启动命令时所在的目录工作。进错目录，后续所有输出都会出问题。

```bash
# 好习惯：进项目目录后再启动
cd /your/project/path
claude
```

**配置 2：设置 Project 和文件夹权限（Cowork / Desktop）**

Cowork 的 Project 是"工作空间"的概念。一个 Project 可以关联：
- 本地文件夹（授权后 Claude 可读写）
- 上下文文件（你的工作说明、资料文档）
- Global Instructions（你的全局工作偏好）

第一次设置建议：
- 先用一个**不重要的测试文件夹**练手
- Global Instructions 里写："没有我的确认，不要删除任何文件"
- 第一周只做"读取和分析"任务，不让它修改文件

**配置 3：理解权限模式（CLI / Desktop / Web 通用）**

Claude Code 当前常见的权限模式是：
- `default`：只自动读文件；改文件、跑命令等需要你确认
- `acceptEdits`：普通文件编辑自动通过，命令和敏感写入仍需确认
- `plan`：只做分析、阅读、规划，不改代码
- `auto`：让分类器在后台帮你审动作，减少频繁确认
- `dontAsk`：只允许预先放行的工具，其他一律拒绝
- `bypassPermissions`：几乎跳过所有检查，只适合隔离环境

新手建议：
1. 第一周优先用 `default`
2. 第二周开始把复杂任务先放进 `plan`
3. 对已验证过的中长任务，再尝试 `acceptEdits`
4. `auto` 和 `bypassPermissions` 放到后面，且一定配合 sandbox / worktree / git 检查点

### 第 3 课：平台地图——CLI、Desktop、Web、IDE、Cowork 怎么选

**课前一句话**：不是只有一个 Claude Code。入口选错了，体验会直接下降一个档次。

#### 3.1 一张平台决策表

| 你想做的事 | CLI | IDE 插件 | Desktop / Web | Cowork |
|----------|-----|---------|--------------|--------|
| 读懂一个代码仓库 | ✅ 首选 | ✅ 很顺手 | ⚠️ 可做但不优 | ❌ |
| 改代码、修 Bug、跑测试 | ✅ 首选 | ✅ 首选 | ⚠️ 适合轻改 | ❌ |
| 边看代码边对话 | ⚠️ | ✅ 最佳 | ❌ | ❌ |
| 整理本地文档文件夹 | ⚠️ 可以但别扭 | ❌ | ✅ | ✅ 首选 |
| 写报告、需求、总结 | ⚠️ | ❌ | ✅ | ✅ 首选 |
| 浏览器辅助任务 | ⚠️ 通过 MCP | ⚠️ 通过 MCP | ✅ 更自然 | ✅ |
| 后台任务、计划任务 | ✅ 脚本化强 | ⚠️ | ✅ 云端/调度 | ✅ |
| 多目录、多系统协作 | ✅ | ✅ | ✅ | ✅ |

#### 3.2 五个入口的本质区别

- CLI：最适合开发者，控制力最强，脚本化最好
- IDE：最适合边改边看，特别适合 review、重构、逐文件迭代
- Desktop：最适合本地文件夹、跨目录协作、可视化权限切换
- Web：最适合远程会话、跨设备访问、轻量处理
- Cowork：最适合非代码工作流，尤其是文档、资料、研究类任务

#### 3.3 两个典型错误

**错误一：把文档生产线放进 CLI**

CLI 当然也能生成文档，但它天然是"终端优先"。如果你的目标是直接产出文件、交付材料、汇总资料，Desktop / Cowork 往往更顺手。

**错误二：把仓库级重构放进纯文档入口**

大规模代码修改往往需要读目录、改文件、跑测试、看 diff、回退变更。这套动作在 CLI 或 IDE 里更稳定。

#### 3.4 一个更成熟的组合路径

典型组合场景：
- Cowork / Desktop 先整理需求文档、竞品资料、研究输入
- 把结果沉淀到共享目录或项目文档
- CLI / IDE 读取这些上下文，实现代码或脚本
- Desktop / Cowork 再读取产出，生成面向业务的交付材料

## 第三阶段：核心工作流

### 第 4 课：5 步执行法——一次通过率倍增的关键

**课前一句话**：Claude Code 的问题往往不是能力不够，而是你给了它太大的自由度，或者太模糊的起点。

#### 4.1 为什么需要固定的执行流程

大多数人第一次用 Claude Code 时是这样的：
1. 脑子里有个任务
2. 直接打开，说"帮我做 XXX"
3. 它开始做，做到一半发现方向不对
4. 你打断它，重新说
5. 反复几次，花了 2 小时，得到的结果还没手动做的准

问题不是模型不行，是**指挥方式不对**。

固定流程的价值是：把自由度锁在正确的位置，减少你和 AI 的"方向对齐"成本。

#### 4.2 五步执行法

**第一步：读（理解现状）**

在开始任何修改之前，先让 Claude 读相关文件，说出它的理解。

```
先读一下 src/api/user.ts 和 src/services/auth.ts，
告诉我：
1. 这两个文件分别在做什么
2. 它们之间有什么关联
先别改，我需要确认你理解正确。
```

为什么这一步不能省：Claude 经常会在没完全理解结构的情况下开始修改，导致改了一个地方、破了另一个地方。

**第二步：出计划（确认范围）**

让它说出做法，你来审批，然后再执行。

```
好，理解准确。现在给我一个修改计划：
- 你打算改哪些文件
- 每个文件改什么
- 哪些地方你不确定，需要我确认

不要现在就改，先给我计划。
```

这一步能帮你发现 90% 的"跑偏"问题——在它动手之前。

**第三步：执行（范围内推进）**

计划确认后，再给执行指令。同时，主动限制不做什么。

```
计划没问题。按这个计划改，但注意：
- 不要重构其他文件
- 不要引入新的依赖
- 不要修改接口的入参和出参
```

**第四步：验证（确认结果正确）**

修改完成后，让它帮你验证，不要全靠眼看。

```
改完了，帮我做以下验证：
1. 读一遍改过的文件，说出你改了什么
2. 运行一下测试命令，看有没有报错
3. 告诉我这次改动有没有潜在风险
```

**第五步：总结（提取可复用知识）**

任务完成后，让它输出一份简短的总结。

```
任务完成，给我一个简短总结：
- 改了什么
- 为什么这样改
- 有没有遗留问题或待确认的地方
```

这个总结有两个用途：一是给你做记录，二是如果后续任务需要接续，这段总结可以作为新任务的上下文。

#### 4.3 "不做什么"清单：比目标更重要

一个任务描述里，禁止事项往往比目标更关键。

**通用禁止清单（可以直接复制用）：**

```
禁止事项：
- 不重构不相关的代码
- 不引入新的第三方依赖
- 不修改接口签名（入参、出参、类型）
- 不删除任何文件，除非我明确指定
- 不修改配置文件（.env、vite.config、tsconfig 等）
- 不向外部发送任何网络请求
- 遇到不确定的地方，先问我，不要自己决定
```

**经验：** 把你上次"它改错了"的地方，加进你的禁止清单。这个清单越用越准。

#### 4.4 一个完整案例

**任务：修复一个用户登录后跳转失败的 Bug**

```
# 第一步：读
读一下 src/pages/Login.tsx 和 src/router/index.ts，
告诉我登录成功后的跳转逻辑是怎么实现的。

# 第二步：出计划（等它回答后）
根据你的理解，给我一个修复计划。
只改跳转相关的逻辑，不要碰表单验证部分。

# 第三步：执行（计划确认后）
按计划修改。禁止事项：不改表单逻辑、不改路由配置。

# 第四步：验证
改完后读一遍改动，然后帮我想三个可能的边界情况，
比如"用户直接访问登录页但已经登录了"。

# 第五步：总结
给我一份两三句话的修改说明，我要粘贴进 commit message。
```

### 第 5 课：Prompt 写法——Claude 听得懂的话，和你平时说的不一样

**课前一句话**：Anthropic 官方花了大量时间研究"如何和 Claude 说话"，他们的核心发现是：清晰、结构化、有上下文的指令，比聪明的措辞更重要。

#### 5.1 五个最有效的 Prompt 技巧

**技巧一：用 XML 标签组织结构**

Claude 对 XML 标签有很好的理解，用标签把不同类型的信息分开，可以大幅减少歧义。

```
# 不好的写法
帮我分析这段代码，找出 Bug，然后修复，用中文解释原因，别改其他地方

# 好的写法
<任务>分析下面这段代码，找出 Bug 并修复</任务>

<代码>
// 粘贴代码
</代码>

<要求>
- 用中文解释 Bug 的原因
- 只改有问题的部分
- 给我一个修改前后的对比
</要求>

<禁止>
不要改函数签名，不要引入新依赖
</禁止>
```

**技巧二：给"为什么"，不只给"什么"**

Claude 对"有动机的指令"反应比纯规则更好。解释你为什么要这样做，它会理解得更准。

```
# 不好：我需要你每次都先读文件再改
# 好：我需要你每次都先读文件再改，因为这个项目文件之间耦合很紧，
# 之前没读就改导致过几次事故
```

**技巧三：Few-shot 示例**

如果你有一个特定的输出格式要求，给它一个例子，比描述一百遍都有效。

```
<任务>给下面每个函数写注释</任务>

<示例格式>
/**
 * 计算用户的剩余额度
 * @param userId 用户 ID
 * @param month 查询月份 (YYYY-MM)
 * @returns 剩余额度（单位：元）
 */
</示例格式>

<代码>
// 你的代码
</代码>
```

**技巧四：长文档放前面**

Anthropic 测试发现：在给 Claude 一篇长文档后提问，把问题放在**文档后面**，性能比放前面高 30%。

```
# 低效写法
总结一下这篇文章的核心论点：[文章内容]

# 高效写法
[文章内容]

---
请总结上文的核心论点
```

**技巧五：让它思考后再输出**

对于复杂的判断类任务，让 Claude 先说思考过程，再给结论，准确率会明显更高。

```
帮我判断这段 SQL 有没有性能问题。
先列出你注意到的所有可能有问题的地方，
再根据这个分析给出你的结论和建议。
```

#### 5.2 常见的烂 Prompt 和改法

| 烂 Prompt | 问题 | 改法 |
|----------|------|------|
| "帮我优化代码" | 太模糊，不知道优化什么 | "帮我减少这段代码里的重复逻辑，保持功能不变" |
| "用最好的方式实现" | "最好"主观，会过度工程化 | "用最简单的方式实现，不考虑扩展性" |
| "帮我写完这个功能" | 边界不清，容易越界 | "实现登录表单的客户端验证，不涉及后端逻辑" |
| "改一下" | 什么都没说 | 给具体文件名、具体行数、具体期望结果 |
| 连续几段长对话后说"继续" | 上下文可能已经失真 | 新任务开新对话，重新给上下文 |

#### 5.3 一个结构化 Prompt 模板（可直接复制）

```
<背景>
[项目/任务的简短背景，3 句话以内]
</背景>

<任务>
[你想让它做什么，一句话概括]
</任务>

<输入>
[相关文件路径、代码片段、或文档内容]
</输入>

<要求>
- [要求 1]
- [要求 2]
- [输出格式要求]
</要求>

<禁止>
- [不要做的事 1]
- [不要做的事 2]
</禁止>

<示例>（可选）
[期望输出的一个例子]
</示例>
```

### 第 6 课：CLAUDE.md、Rules、Auto Memory——给 AI 装上"员工手册"

**课前一句话**：每次任务都从零解释上下文，效率极低，而且容易遗漏。把固定信息写进配置，是从"能用"到"好用"最重要的一步。

#### 6.1 先建立一个新认知

Claude Code 现在有四种常见的长期上下文来源：

| 载体 | 作用 | 适合放什么 |
|------|------|-----------|
| `~/.claude/CLAUDE.md` | 全局偏好 | 语言偏好、输出风格、通用禁令 |
| 项目根目录 `CLAUDE.md` | 项目级规则 | 架构说明、命令、目录、禁止事项 |
| `.claude/rules/` | 更细粒度规则 | 某类文件、某种任务、某个子目录的专门规则 |
| Auto Memory | 长期协作记忆 | 你反复强调的习惯、团队偏好、历史经验 |

**关键区别：**
- `CLAUDE.md` 和 rules 是你主动写的
- Auto Memory 是 Claude 逐步总结的
- 它们都是 context，不是硬编码开关

#### 6.2 项目级 `CLAUDE.md` 标准结构

一份好用的 `CLAUDE.md` 通常包含以下七类信息：

```markdown
# 项目背景

[2-3 句话：这是什么项目，用来做什么，谁在用]

# 常用命令

\`\`\`bash
npm run dev       # 启动开发服务器
npm run build     # 构建生产版本
npm run test      # 运行测试
npm run lint      # 类型检查
\`\`\`

# 目录结构

[列出关键目录和它们的职责]

# 编码规范

- 语言：TypeScript 严格模式
- 格式化：使用 Prettier，不要手动调格式
- 注释：只在非自明的逻辑处写注释
- 命名：函数用 camelCase，组件用 PascalCase

# 禁止事项

- 不引入新的第三方依赖（需要先问我）
- 不修改 .env 文件
- 不改动 src/api/ 目录下的接口签名
- 不删除任何文件，除非我明确要求

# 当前重点（定期更新）

[本周在做的功能/修复的问题，帮助 Claude 快速理解当前重点]

# 协作约定

- 复杂任务先出计划，再执行
- 修改后先自检，再总结
- 不确定的地方先提问，不要自己猜
```

#### 6.3 `.claude/rules/`：把项目约束拆得更细

当一个项目越来越大时，不要把所有东西都堆进根目录 `CLAUDE.md`。更好的做法是：

- 根目录 `CLAUDE.md` 放全局项目规则
- `.claude/rules/frontend.md` 放前端规则
- `.claude/rules/backend.md` 放后端规则
- `.claude/rules/review.md` 放审查标准

这样做的好处是：冲突更少，维护更容易，也更方便团队协作。

#### 6.4 Auto Memory：让 Claude 真正"越用越懂你"

Auto Memory 适合承接这类信息：
- 你长期坚持的输出格式
- 团队反复强调的工程习惯
- 某个项目里一再出现的约束和偏好

不适合放进 memory 的内容：
- 本周临时需求
- 很快过期的版本号、分支名、排期
- 带有密钥、凭证、敏感数据的内容

经验法则：
- "这周的任务"放 `CLAUDE.md`
- "这个团队一直这样做"放 memory
- "只影响一个模块"放 rules

#### 6.5 Cowork / Desktop：上下文配置实战

**Global Instructions 模板（适合大多数知识工作者）：**

```
你是我的工作助手。我的工作习惯如下：

输出偏好：
- 中文输出，专业词汇保留英文
- 结构化输出（用标题、列表而不是大段文字）
- 先结论，再解释
- 要点不超过 5 条，超过则归类

操作习惯：
- 没有我的确认，不要删除任何文件
- 任何写操作执行前，先告诉我你准备做什么
- 不确定的地方，直接问我，不要猜

当前工作重点：
[每周更新这里]
```

**Project Instructions 模板（以产品经理的竞品分析项目为例）：**

```
项目背景：
这个 Project 用于竞品分析工作。主要资料在"竞品资料"文件夹。

文件组织规范：
- 竞品资料/：原始截图和素材
- 分析报告/：整理后的分析文档
- 格式模板/：输出时参照这里的格式

输出要求：
- 每份分析报告参照"格式模板"里的结构
- 分析维度：功能、用户体验、定价、市场定位
- 数据来源必须注明（截图文件名或网址）

禁止事项：
- 不要在"竞品资料"文件夹写入任何新文件
- 报告草稿保存在"分析报告/草稿"子目录
```

#### 6.6 维护上下文的好习惯

1. **发现一次问题，立刻更新**：它做错了一件事，把禁止条件加进去，而不是每次都重新说
2. **每月回顾一次**：清掉已经过期的"当前重点"，加入新的协作约定
3. **把信息放对地方**：项目事实进 `CLAUDE.md`，长期偏好进 memory，局部规则进 rules
4. **避免互相冲突**：同一件事不要在全局、项目、rules 里写三种版本

> **本课落地动作**：用今天学的模板，给你最常用的一个项目写一份 CLAUDE.md 或 Project Instructions。

## 第四阶段：进阶能力

### 第 7 课：Skills 与内置命令——把高频动作变成一句话能力

**课前一句话**：Claude Code 现在不只是"命令驱动"，而是"命令 + skill + subagent"三层协作。真正高频的动作，应该被做成可复用能力。

#### 7.1 三类控制工具的区别

| 工具 | 是什么 | 用来做什么 |
|------|--------|-----------|
| **Tools（工具）** | Claude 调用的能力单元 | 读文件、写文件、跑命令、连 MCP |
| **Built-in Commands** | 官方内置入口 | `/clear`、`/compact`、`/context` 等 |
| **Skills** | 你定义的可复用能力包 | 一个目录里放 prompt、规则、脚本、模板 |
| **Subagents** | 专注特定任务的子助手 | 复杂任务拆分与并行执行 |

#### 7.2 先学这些内置命令

| 命令 | 作用 |
|------|------|
| `/help` | 帮助 |
| `/clear` | 清空当前对话上下文 |
| `/compact` | 手动触发上下文压缩 |
| `/context` | 查看当前上下文和可用资源 |
| `/memory` | 查看/编辑当前记忆 |
| `/agents` | 查看或管理子 Agent |
| `/permissions` | 查看或调整权限模式 |
| `/doctor` | 诊断环境问题 |
| `/init` | 初始化项目协作文件 |
| `/security-review` | 做安全导向的代码审查 |
| `/schedule` | 创建计划任务 |
| `/bug` | Bug 报告模板 |

说明：
- 旧版资料里常见的 `/review` 现在不适合作为主教学入口
- 自定义 slash commands 仍然可用，但官方当前更推荐把复杂高频任务做成 skill

#### 7.3 Skill 是什么，为什么比单个命令更强

一个 skill 往往至少包含四样东西：
- 一个明确任务说明
- 一组输入输出约束
- 可复用模板 / 示例
- 必要时附带脚本、资源文件、检查逻辑

当任务跨多步、需要模板、需要脚本、需要固定目录结构时，skill 比单一命令更稳。

#### 7.4 从零做一个 `standup` skill

**场景**：每天早上 9 点，你需要整理一份站会消息——昨天做了什么、今天计划什么、有没有阻碍。以前要手动打开 Jira、看 git log、再写，现在可以一句话搞定。

**创建方式**：在 `.claude/skills/standup/` 目录下准备一个 skill 说明文件（团队也可以放到共享仓库）

```markdown
# standup skill

任务：生成今日站会消息

## 昨天
读取最近 24 小时内的 git commit 记录（用 git log --since="24 hours ago"），
列出主要完成的内容，每条不超过一句话。

## 今天
读取 TODO.md（如果存在），列出今日计划，没有则提示我手动填写。

## 阻碍
询问我是否有阻碍，如果有，格式化成一条描述。

输出格式：纯文本，适合直接粘贴进钉钉/飞书。
```

使用方式：
- 可以在会话里直接说"使用 standup skill"
- 也可以保留一个兼容入口，把它封装成 `/standup`

#### 7.5 五个高频 skill / 命令范例

**1. `/commit` — 生成 git commit 消息**
```markdown
读取 `git diff --staged`，生成一条 commit 消息。
格式：<类型>(<范围>): <描述>
类型：feat/fix/refactor/docs/chore
描述：中文，不超过 50 字，说清楚做了什么。
```

**2. `/security-review` — 安全审查**
```markdown
读取 `git diff HEAD~1`，进行安全导向审查。
重点检查：敏感信息、命令注入、SQL 注入、权限绕过、危险默认值。
输出：按严重程度分级（阻断/警告/建议），每条附说明。
```

**3. `/doc` — 生成函数文档**
```markdown
读取 $ARGUMENTS 指定的文件，
为每个导出函数生成 JSDoc 注释。
要求：描述准确、参数类型正确、不超过 3 行。
```

**4. `/daily` — 日报生成**
```markdown
读取今日 git 提交和 TODO.md，生成工作日报。
格式：
- 完成事项（来自 git log）
- 未完成事项（来自 TODO.md 的未勾选项）
- 明日计划（来自 TODO.md 的下一步标注）
中文输出，简洁，适合内部汇报。
```

**5. `/debug` — 报错分析**
```markdown
分析 $ARGUMENTS 里的报错信息。
输出：
1. 报错原因（一句话）
2. 可能触发的代码位置
3. 三个可能的修复方向，按可能性排序
不要直接修改代码，先让我选方向。
```

> **本课落地动作**：把你最常重复描述的一个任务，优先做成 skill；如果它很简单，再做成 slash command。

### 第 8 课：MCP 工具链——打通你常用的那些工具

**课前一句话**：Claude 本身很强，但真正的威力在于它能"伸出手"接触你的工具生态。MCP 就是这双手。

#### 8.1 MCP 是什么

MCP（Model Context Protocol）是 Anthropic 开发的开放协议，让 Claude 可以通过标准化接口连接外部工具和数据源。

你可以把它理解成：Claude 的"插件系统"，但不是装在 Claude 里，而是你在本地启动一个 MCP 服务器，Claude 通过协议连接它。

**已有官方/社区 MCP 服务器（部分列举）：**

| 工具 | 能做什么 |
|------|---------|
| Filesystem | 读写任意路径的文件（超越默认工作目录） |
| GitHub | 读 PR、Issues、代码，推送 commit |
| Slack | 读写消息频道，发送通知 |
| Notion | 读写 Notion 页面和数据库 |
| 飞书/钉钉 | 读写消息、文档（社区实现） |
| Browser | 打开网页、截图、操作浏览器（Playwright） |
| PostgreSQL / MySQL | 执行 SQL 查询 |
| Google Drive / Sheets | 读写云端文件 |

#### 8.2 配置 MCP 的基本步骤

1. 确认你想接入的工具有 MCP 服务器（搜索 `mcp server [工具名]`）
2. 按服务器文档安装依赖
3. 在 Claude Desktop 设置里添加 MCP 配置
4. 重启 Claude Desktop，验证连接

**配置示例（Claude Desktop 设置文件）：**

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/你/的/工作目录"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "你的_token"
      }
    }
  }
}
```

#### 8.3 最值得先接入的 3 个 MCP

**1. Filesystem MCP（开发者和知识工作者都推荐）**

最实用的第一步，让 Claude 可以读写你指定目录里的所有文件，不再受单一工作目录限制。

**2. GitHub MCP（开发者必装）**

直接在 Claude 里完成 PR review、Issue 整理、代码搜索，不用来回切换界面。

**3. Browser/Playwright MCP（研究和运营人员推荐）**

让 Claude 可以打开网页、截图、提取内容，用于竞品调研、数据采集、自动化测试。

#### 8.4 MCP 使用注意事项

- MCP 服务器运行在你的本地机器，需要保持运行才能使用
- 给 MCP 服务器的权限要最小化：只开放你真正需要的目录/资源
- 连接外部服务（GitHub、Slack）的 MCP 需要 Access Token，妥善保管
- 定期检查 MCP 服务器日志，发现异常立即断开

> **本课落地动作**：选一个你最常用的工具，找到对应的 MCP 服务器，尝试接入并完成一个小任务验证。

### 第 9 课：上下文与记忆管理——不让 AI 越跑越"忘事"

**课前一句话**：Claude 的上下文窗口是有限的，当你不主动管理它时，长任务里的 AI 会越来越像一个失忆的助手。

#### 9.1 理解上下文窗口

不要死记具体 token 数字，更重要的是知道它为什么会很快被吃满：
- 每次读一个大文件、报错堆栈、测试输出，都会迅速占用上下文
- 任务越长，早期信息越容易被压缩、摘要化、边缘化
- 子 Agent、MCP、长日志、超长文档都会进一步放大这个问题

Claude Code 会自动做上下文整理，但这个过程天然是有损的。结论很简单：

**不要把关键规则只说在对话里。**

#### 9.2 四个主动管理上下文的技巧

**技巧一：任务完成后用 `/clear` 开新对话**

一个任务一个对话。不要让所有任务都堆在同一个上下文里。

```
# 规则
修复一个 Bug → /clear
开始下一个功能 → 新对话，重新给上下文
```

**技巧二：用 `CLAUDE.md`、rules、memory 分担重复说明**

你每次都要重新解释的上下文，不应该一直堆在会话里。

判断方法：
- 重复 3 次以上的项目约束，写进 `CLAUDE.md`
- 长期有效的个人偏好，写进 memory
- 只影响某一类任务的规则，写进 `.claude/rules/`

**技巧三：长任务用"分段交付"模式**

把大任务切成多个里程碑，每个里程碑完成后让它输出总结，用总结开启下一段。

```
# 大任务：重构整个用户模块

# 对话 1：分析现有结构
分析 src/user/ 目录，给我一份重构建议报告。
[任务完成后，保存报告]

# 对话 2：执行重构（用报告作为上下文）
这是我们上次分析的重构建议：[粘贴报告]
按照第 2 条建议，先重构 UserService。
```

**技巧四：善用 `/compact` 和 `/context`**

当对话较长但任务还没完成时：
- 用 `/compact` 主动压缩上下文
- 用 `/context` 检查当前会话到底加载了哪些信息
- 发现上下文污染时，及时 `/clear`

#### 9.3 上下文重置的信号

当你观察到以下情况，应该考虑重置上下文：

- 它开始重复之前说过的分析
- 它"忘记"了你之前确认的约束条件
- 它给的建议和 20 分钟前的建议明显矛盾
- 回复质量明显下降，开始模糊化处理

> **本课落地动作**：回顾一次你觉得 Claude 后期"越来越差"的任务，判断问题是出在上下文、memory，还是规则放错了地方。

## 第五阶段：Agent 架构

### 第 10 课：Subagents 与 Agent Teams——一个人指挥多个 AI 分工干活

**课前一句话**：Claude Code 现在已经不是单线程助手了。真正复杂的任务，要学会把规划、探索、执行、审查拆给不同 Agent。

#### 10.1 什么时候需要 Subagents / Agent Teams

**不需要多 Agent 的情况（先做好单 Agent）：**
- 任务是线性的，一步一步来就行
- 任务范围在一个目录内
- 你还在熟悉 Claude Code 阶段

**适合多 Agent 的情况：**
- 任务可以拆成互相独立的子任务，并行处理
- 单一 Agent 的上下文窗口装不下所有需要的信息
- 需要一个"质检员"角色来检查主 Agent 的输出
- 任务有明确的分工边界（研究/撰写/审查/发布）

#### 10.2 Claude Code 里最值得先学的三种用法

**用法 1：Plan Mode**

复杂任务先进入 `plan`，只做分析和规划，不做改动。

```
这是一个复杂的重构任务。
先进入规划模式：
- 读相关文件
- 输出完整步骤
- 标出风险点和待确认问题
先不要改任何代码。
```

**用法 2：Subagents**

把不同类型的子任务分给不同 Agent，例如：
- 一个负责探索代码库
- 一个负责实现
- 一个负责审查风险

**用法 3：Agent Teams**

让 Claude 把一个大任务拆成多个后台子任务，再汇总结果。它特别适合：
- 大仓库代码搜索
- 跨多个文档的综合分析
- 研究、撰写、审查的流水线协作

#### 10.3 三种主要的多 Agent 架构

**架构 1：主-从（Orchestrator + Workers）**

一个协调者 Agent 拆分任务，分发给多个执行者 Agent，最后汇总结果。

```
主 Agent：
"我需要写一份竞品分析报告，分析三家公司。
请分别启动三个子任务：
- 子任务 A：分析 A 公司产品功能
- 子任务 B：分析 B 公司产品功能
- 子任务 C：分析 C 公司产品功能
三个任务完成后，把结果汇总成一份对比报告。"
```

**架构 2：流水线（Pipeline）**

任务在多个 Agent 之间顺序流转，每个处理一段，结果传给下一个。

```
Agent 1（研究）→ Agent 2（撰写）→ Agent 3（审查）→ Agent 4（格式化）
```

**架构 3：主-质检（Writer + Reviewer）**

一个 Agent 生成内容，另一个专门检查，形成内部复核机制。

```
写作 Agent："写一份技术方案文档"
↓
审查 Agent："读取上面的文档，检查：
1. 技术方案是否有遗漏的边界情况
2. 语言是否清晰
3. 结论是否有足够支撑"
```

#### 10.4 新手路径建议

**阶段 1**（前两周）：把单个 CLI 或 Cowork 用好，能稳定完成 5 步执行法。

**阶段 2**（第三四周）：加入 `CLAUDE.md`、rules，开始做 skill，减少重复描述。

**阶段 3**（第二个月）：接入 1-2 个 MCP，打通最常用的外部工具。

**阶段 4**（稳定后）：先学 `plan` 和简单 subagent，再逐渐引入 agent teams。

不要在单工具都没跑稳的时候直接上多智能体，会适得其反。

> **本课落地动作**：找一个你觉得"光靠自己说太费劲"的复杂任务，画出一个简单的 Agent Team 分工图。

### 第 11 课：安全边界——危险能力怎么放权又不失控

**课前一句话**：Claude Code 能删文件、推代码、发请求，这些都是高价值能力——问题不是关闭它们，而是把触发条件和确认点设清楚。

#### 11.1 Claude Code 的"危险能力清单"

Claude Code 可以做到，但有风险的操作：

| 操作 | 风险级别 | 建议策略 |
|------|---------|---------|
| 删除文件 | 高 | 必须人工确认，先备份 |
| git push | 高 | 必须人工确认，只允许特定分支 |
| 安装依赖包 | 中 | 列出清单后人工确认 |
| 写入生产数据库 | 高 | 禁止，使用测试环境 |
| 发送外部网络请求 | 中-高 | 事先说明目标地址，人工确认 |
| 修改环境配置文件 | 高 | 禁止，除非专项任务 |
| 批量重命名/移动文件 | 中 | 先列出变更清单，确认后执行 |

#### 11.2 先理解官方的权限控制思路

真正稳定的权限控制不是一句"别乱改"，而是四层叠加：
- 权限模式：`default`、`acceptEdits`、`plan`、`auto`、`dontAsk`、`bypassPermissions`
- allow / ask / deny 规则：哪些动作默认允许，哪些必须问，哪些直接禁
- sandbox / worktree：把危险操作放进可隔离、可回退环境
- 你自己的项目规则：`CLAUDE.md`、rules、hooks

课程建议：
- 新手默认 `default`
- 高风险改动先 `plan`
- 已验证流程再 `acceptEdits` 或 `auto`
- `bypassPermissions` 只在隔离环境使用

#### 11.3 写进 `CLAUDE.md` 的安全边界模板

```markdown
# 安全边界

## 必须人工确认的操作（停下来等我说 OK 才能继续）
- git push（任何分支）
- 删除文件（哪怕是临时文件）
- 安装新的 npm/pip 依赖
- 修改 .env、.env.local 等环境配置文件
- 发送任何外部网络请求

## 禁止操作（不管什么情况都不能做）
- 访问 src/config/secrets.ts
- 操作 production 分支
- 执行 DROP TABLE 等破坏性 SQL

## 操作前必须完成
- 列出本次任务会修改的所有文件
- 如果涉及超过 5 个文件，先给我一个变更计划
```

#### 11.4 黄金操作习惯

**任务前：备份**
```bash
git checkout -b backup/[时间]
# 或用 worktree 开一个隔离分支
```

**任务后：检查 diff**
```bash
git diff   # 看改了什么
git status # 看有没有意外的文件变动
```

**发现问题时：恢复**
```bash
git restore .
git restore --staged .
git revert [hash]
```

#### 11.5 一个关键原则

**不靠信任模型，靠最小化可恢复成本。**

真正稳定的 AI 编程习惯，不是让模型变得更可信，而是确保每一步操作都可以回退。在这个前提下，才值得逐步放开权限。

> **本课落地动作**：把今天的安全边界模板加进你的 CLAUDE.md。

### 第 12 课：Hooks、Schedule 与 Auto Mode——让 Claude 在后台替你盯着事

**课前一句话**：当你把触发条件、执行逻辑和安全边界都设置清楚后，就可以让 Claude Code 在你不在的时候自动完成某些工作。

#### 12.1 Hooks：把关键节点自动化

Hooks 是官方公开能力，不是"泄漏彩蛋"。它允许你在特定时机自动插入检查、通知、后处理：

| Hook 类型 | 触发时机 | 使用场景 |
|----------|---------|---------|
| `PreToolUse` | 工具调用前 | 权限检查、日志记录、危险操作拦截 |
| `PostToolUse` | 工具调用后 | 结果验证、通知发送、后处理 |
| `Notification` | Claude 发送通知时 | 转发到钉钉/飞书/Slack |

**配置 Hook 示例（钉钉通知）：**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "node scripts/notify-dingtalk.js '$TOOL_OUTPUT'"
        }]
      }
    ]
  }
}
```

#### 12.2 `/schedule`：把稳定任务变成计划任务

适合被调度的任务有三个特征：
- 输入来源稳定
- 输出格式固定
- 失败代价可控

优先考虑这几类：
- 每日代码健康巡检
- 周报 / 日报生成
- 文档一致性检查
- 依赖更新预审

#### 12.3 三个可落地的自动化场景

**场景 A：每日代码健康巡检**

```bash
# 计划任务 / 定时任务脚本（每天早上 8 点运行）
#!/bin/bash
cd /your/project
claude --no-confirm "
读取过去 24 小时的 git log 和当前 lint 报告，
生成一份简短的代码健康日报：
- 昨日提交摘要
- 当前 lint 警告数量
- 需要关注的风险点
输出到 reports/daily-$(date +%Y%m%d).md
"
```

**场景 B：PR 自动预审**

```bash
# 在 git pre-push hook 里加入
#!/bin/bash
claude "
读取 git diff main...HEAD，
进行代码预审：
1. 有没有明显的 Bug
2. 有没有遗忘的 TODO
3. 有没有敏感信息（token、密码）
结论：是否可以安全推送
如果有严重问题，输出 EXIT_FAIL
"
```

**场景 C：文档与代码同步检查（Cowork + CLI 协作）**

```
# Cowork 任务
读取 docs/ 文件夹里的 API 文档，
提取所有接口的参数和返回值，
生成一份"接口契约清单"保存到 reports/api-contract.md

# CLI 任务（接续）
读取 reports/api-contract.md 和 src/api/ 目录，
检查代码实现和文档描述是否一致，
不一致的地方列出来，不要自动修改。
```

**场景 D：Auto Mode 处理长任务**

当任务经过验证、权限边界清楚、你又不想频繁点确认时，可以尝试 `auto mode`。

适合：
- 大量安全读操作
- 已验证过的重构套路
- 标准化日报、巡检、整理任务

不适合：
- 首次尝试的新流程
- 涉及生产环境、数据库、敏感配置
- 你还没准备好 diff review 和回退机制的时候

#### 12.4 新手自动化路径

**阶段 1**：先用手动触发（自己输入命令），验证任务流程可靠。

**阶段 2**：把验证过的流程做成 skill 或 slash command，一句话触发。

**阶段 3**：把稳定的流程交给 `/schedule` 或脚本调度。

**阶段 4**：对低风险、重复性高的任务，再尝试 `auto mode`。

**永远不要**：跳过手动验证阶段直接搞全自动化。没有验证过的流程自动化后出问题，排查成本是手动的十倍。

> **本课落地动作**：选一个你每周都手动做的重复任务，先做成 skill，再决定要不要接 `/schedule`。

### 第 14 课：实战案例 B——开发者的代码仓库维护流程

**场景描述**：你是一名后端开发者，独立维护一个中型 Node.js 项目，面临：代码文档缺失、技术债务积累、PR review 质量参差不齐。

#### 工作流设计

**CLAUDE.md 配置（关键部分）：**

```markdown
# 项目背景
一个 Node.js RESTful API 服务，接 PostgreSQL，约 30 个接口，
无全职测试人员，靠代码审查保证质量。

# 最重要的规则
1. 每次修改前先读相关文件
2. 改动超过 3 个文件的任务，先给计划
3. 不能修改 src/db/migrations/ 里的文件
4. 不能改动 package-lock.json（除非专门升级依赖任务）

# 当前技术债务（持续更新）
- 用户模块缺少 JSDoc 注释
- tests/ 里的集成测试覆盖率 < 30%
- src/utils/ 有几个废弃函数没清理
```

**Skills / Commands 套装：**

```markdown
# .claude/commands/security-review.md
读取 `git diff main...HEAD`
进行安全导向审查，重点关注：
1. 是否有 SQL 注入风险（重点！）
2. 错误处理是否完整
3. 日志是否足够排查问题
输出：问题列表，按严重程度排序（阻断/警告/建议）

# .claude/commands/doc.md
为 $ARGUMENTS 文件里所有导出的函数补全 JSDoc 注释。
要求：参数类型准确，返回值说明清晰，不超过 5 行。
改完后展示修改摘要，等我确认再写入文件。

# .claude/commands/debt.md
读取 CLAUDE.md 里的"当前技术债务"列表，
结合代码现状，给出本周优先清理哪 1-2 项的建议，
说明理由（影响范围、清理难度、收益）。
```

**周度维护工作流（每周五下午）：**

```
第一步：/debt → 选出本周要处理的技术债务项
第二步：5 步执行法完成清理任务
第三步：/doc [修改的文件] → 补充文档
第四步：/security-review → 最终自检
第五步：生成 commit 消息，推代码
```

### 第 15 课：实战案例 C——知识工作者的研究资料处理系统

**场景描述**：你是一名市场研究分析师，每周需要处理大量文章、报告和数据，整理成简报发给管理层。以前完全手动，准备用 Cowork 建立可复用的"研究资料处理系统"。

#### 系统设计

**文件结构：**
```
市场研究/
├── 原始资料/
│   ├── 2026-W14/     ← 本周收集

│   ├── 周报/

│   ├── 周报模板.md
│   └── 专题报告模板.md
└── 知识库/           ← 提炼出的长期洞察
```

**Global Instructions（关键部分）：**
```
研究分析的输出要求：
- 先事实，再分析，再判断
- 数据必须注明来源（文件名或网址）
- 不确定的内容用「待核实」标注，不要猜测

```

**周报生成工作流：**

```
第一阶段：资料入库（每天随手做）
把收集到的文章、截图放进"原始资料/2026-W14"

第二阶段：批量摘要（周五上午）
读取"原始资料/2026-W14"里的所有文件，
为每份资料生成 3-5 句话摘要：
- 核心观点
- 关键数据
- 来源信息
输出到"原始资料/2026-W14/摘要汇总.md"

第三阶段：生成周报
基于"摘要汇总.md"，参照"模板/周报模板.md"，
生成本周市场简报：
- 本周重要动态（3-5 条）
- 值得关注的趋势（2-3 条）
- 建议管理层关注的事项（1-2 条）
保存到"简报/周报/2026-W14-市场简报.md"

第四阶段：知识沉淀
读取本周简报，判断哪些洞察有长期价值，
提炼后追加到"知识库/行业洞察.md"
```

建立这个系统三个月后，你会发现：
- 知识库里积累了大量提炼过的行业洞察
- 每周周报的质量越来越稳定（因为有了历史参照）
- 新员工接手时，系统本身就是最好的培训材料

## 附录 A：常见报错与排查手册

| 现象 | 可能原因 | 解决方法 |
|------|---------|---------|
| CLI 启动后立刻报错 | 安装不完整或环境不匹配 | 重新安装，确认安装方式，并运行 `/doctor` |
| Cowork 任务中途停止 | 桌面端被关闭或电脑睡眠 | 任务运行时保持电脑唤醒 |
| Claude 读不到文件 | 工作目录设置错误 | 检查启动时所在目录 |
| 回答越来越不准确 | 上下文太长 | `/clear` 开新对话，重给上下文 |
| 同样的任务结果差异很大 | 上下文不稳定 | 把关键约束写进 CLAUDE.md |
| 它改了不该改的文件 | 禁止事项没写清楚 | 更新 CLAUDE.md，加入禁止事项 |
| MCP 连接失败 | 服务器未启动或配置错误 | 检查 MCP 服务器日志和配置文件 |

## 附录 D：实现细节旁注——哪些值得参考，哪些不要当成产品承诺

2026 年 3 月 31 日，Anthropic 因为打包失误，意外将 Claude Code 的完整 TypeScript 源码发布到 npm。技术社区从中挖出了一些有意思的实现细节。

这些细节**可以帮助理解设计思路**，但不应该直接当成长期稳定的产品承诺。更稳的学习路径始终是：先学官方文档，再把源码观察当作补充。

**旁注 1：`CLAUDE.md` 确实很重要，但它更像高优先级 context，不是强制配置项**

这意味着：写得具体、无冲突、贴近项目事实，效果最好；不要把它当成万能开关。

**旁注 2：Claude Code 确实会主动整理上下文**

这意味着：超长任务里，越关键的信息越应该沉淀到 `CLAUDE.md`、rules、skill，而不是只留在会话里。

**旁注 3：安全分级和权限拦截是产品核心，不是边角料**

这说明 Claude Code 的护城河不只是模型本身，还包括权限模式、sandbox、hooks、allow/ask/deny 规则这套工程能力。

**旁注 4：实现细节会变，但使用原则不会变**

真正稳定的原则只有几个：先计划，后执行；重要规则持久化；高风险操作要可回退；自动化必须先手动验证。

### D. 课程实施时重点吸收的结论

- `CLAUDE.md`、rules、memory 都应被讲成 context 管理体系，而不是单一配置技巧
- `plan`、`acceptEdits`、`auto`、`bypassPermissions` 要和安全边界一起讲，不能拆开
- subagents 的价值不只是“多智能体”，更是“上下文隔离”和“职责收敛”
- hooks、schedule、MCP 要被讲成“工作流搭建能力”，不是孤立功能点
- 泄漏源码只保留为附录背景，不作为核心知识来源
