## 这节课你要拿到什么

- 一张自动化适配评分表（填完就知道该不该自动化）
- 一份必须人工确认的动作清单
- 一套包含停止开关、日志和人工接管的回退机制

## 不要先追求全自动，先做正确分工（⏱ 5 分钟）

任何流程都先分成 3 层：

| 层级 | 说明 | 举例 |
|------|------|------|
| 自动执行层 | 低风险、可重复、出错可回退 | 信息收集、格式整理、草稿生成 |
| 人工确认层 | 涉及判断、对外影响、不可逆 | 审批、外发消息、正式发布 |
| 禁止自动层 | 高危、高损失、无法回退 | 支付、删除生产数据、批量覆盖 |

> 💡 先把你手头的流程按这 3 层分一遍，自动化的边界就清楚了。

✅ 完成标准：你能把自己的一个工作流程拆成这 3 层

## 跟着做：用评分表判断该不该自动化（⏱ 10 分钟）

给一个流程打分，满足越多越适合自动化：

| 维度 | 说明 | 是/否 | 你的情况 |
|------|------|-------|---------|
| 高频 | 一周至少发生 3 次 | | |
| 规则明确 | 完成标准能写成规则 | | |
| 输入稳定 | 数据来源固定，格式一致 | | |
| 风险可控 | 出错可回退，不会造成不可逆损失 | | |
| 可观察 | 有日志和结果可看 | | |

**评分规则：**
- 4-5 个"是"：适合自动化，可以开始搭建
- 3 个"是"：可以做半自动（自动生成草稿 + 人工确认）
- 1-2 个"是"：不建议自动化，手动或用简单工具辅助即可

### 动手填表（⏱ 5 分钟）

打开飞书多维表格或 Notion，把你想自动化的流程列出来，逐个打分：

| 流程 | 高频 | 规则明确 | 输入稳定 | 风险可控 | 可观察 | 得分 | 建议 |
|------|------|---------|---------|---------|--------|------|------|
| 每日日报整理 | ✅ | ✅ | ✅ | ✅ | ✅ | 5 | 可自动化 |
| 客户消息回复 | ✅ | ⚠️ | ✅ | ❌ | ✅ | 3 | 半自动 |
| 合同审批 | ❌ | ❌ | ❌ | ❌ | ✅ | 1 | 不自动化 |

✅ 完成标准：你至少给 3 个流程打了分，并且知道哪些适合自动化

## 哪些动作必须人工确认

下面这些动作默认都要确认，没有例外：

| 动作 | 为什么必须确认 | 确认方式建议 |
|------|--------------|-------------|
| 对外发送消息 | 发错内容影响信任 | 先生成草稿 → 人工审核 → 确认发送 |
| 改生产数据 | 不可逆，影响业务 | 先在测试环境验证 → 人工确认 → 执行 |
| 删除或覆盖文件 | 不可逆，可能丢失数据 | 先备份 → 人工确认 → 执行 |
| 下单、支付、扣费 | 涉及资金 | 必须人工确认金额和对象 |
| 批量执行不可逆动作 | 错误会被成倍放大 | 先小批量测试 → 人工确认 → 分批执行 |

> ⚠️ 任何一个想绕过人工确认，都要有充分理由和完整的回退机制。

## 跟着做：设计你的回退机制（⏱ 15 分钟）

回退机制最少要有 3 个组件：

### 1. 停止开关

一旦发现输出异常，能立刻停止，不继续放大错误。

**怎么实现：**
- Coze：在工作流里加"条件判断"节点，异常时走"停止"分支
- 飞书多维表格：设置自动提醒，当某个字段出现异常值时通知你
- 自建脚本：加一个开关变量，设为 false 时整个流程不执行

### 2. 日志记录

至少记录这 5 项：

| 记录项 | 说明 | 记录工具 |
|--------|------|---------|
| 执行时间 | 什么时候跑的 | 飞书多维表格（自动记录） |
| 触发来源 | 谁/什么触发的 | 飞书多维表格 |
| 执行内容 | 做了什么 | 飞书多维表格 |
| 输出结果 | 输出到哪里、内容是什么 | 飞书多维表格 |
| 异常情况 | 是否报错、是否需要人工介入 | 飞书多维表格 |

> 推荐用飞书多维表格做日志，因为它支持自动记录时间、筛选异常、生成统计图表。Notion Database 也可以。

### 3. 人工接管点

流程卡住时，人可以无缝接手，不用重来。

**设计方法：**
- 每个关键步骤的输出都保存到可访问的位置（不要只存在内存里）
- 流程中断时，记录"做到哪一步了"
- 人工接手后，能从断点继续，不需要从头开始

✅ 完成标准：你的回退机制包含停止开关、日志记录和人工接管点这 3 个组件

## 一个健康的自动化流程长什么样

以"每日客户反馈整理"为例：

| 步骤 | 执行方式 | 工具 | 说明 |
|------|---------|------|------|
| 1. 收集原始反馈 | 自动 | Coze（定时抓取） | 从客服系统、社群、邮件收集 |
| 2. 分类和提取要点 | 自动 | Coze（文本处理） | 按主题分类，提取关键信息 |
| 3. 生成整理报告 | 自动 | Coze → 飞书文档 | 输出到草稿区 |
| 4. 人工确认 | 人工 | 飞书文档 | 检查分类是否准确、要点是否遗漏 |
| 5. 写入正式报告 | 人工确认后自动 | 飞书文档 | 确认后自动归档 |
| 6. 记录日志 | 自动 | 飞书多维表格 | 记录执行时间、条数、异常情况 |

**关键：不要把第 4 步删掉。** 人工确认是整个流程的安全阀。

## 📋 案例：小赵从半自动升级到可控自动

小赵，电商客服主管，团队每天收到 200+ 条客户咨询，之前全靠人工分拣和回复。

### 他的升级过程

| 阶段 | 做了什么 | 自动化程度 | 效果 |
|------|---------|-----------|------|
| 第 1 个月 | 用 Coze 搭了消息分类 Bot，只做分类不回复 | 半自动（分类自动，回复手动） | 分拣时间从 2 小时降到 30 分钟 |
| 第 2 个月 | 加了常见问题的回复草稿生成 | 半自动（草稿自动，发送手动） | 回复速度提升 50%，但仍需人工确认 |
| 第 3 个月 | 对"物流查询"类问题开放自动回复（最低风险） | 部分自动 | 物流类问题 80% 自动处理，客服专注复杂问题 |
| 第 4 个月 | 加了异常检测：连续 3 条回复被用户追问时自动停止 | 可控自动 | 自动回复准确率 92%，异常时自动转人工 |

> "最大的教训是第 2 个月差点直接开全自动回复。幸好先跑了一周草稿模式，发现有 15% 的回复不准确。如果直接发出去，客户投诉就来了。" —— 小赵

### 小赵的回退机制

| 组件 | 实现方式 |
|------|---------|
| 停止开关 | Coze 工作流里加了条件节点：连续 3 条被追问 → 停止自动回复 → 通知人工 |
| 日志记录 | 飞书多维表格记录每条自动回复的内容、用户反应、是否需要人工介入 |
| 人工接管 | 自动回复停止后，对话自动转给在线客服，附带之前的对话摘要 |

## 常见翻车

### 翻车 1：没设停止开关，错误被成倍放大

❌ 错误做法：自动化流程跑了一晚上，因为数据源格式变了，生成了 200 条格式错误的消息并全部发出。

为什么错：没有异常检测和停止机制，错误被自动重复执行。

✅ 正确做法：在 Coze 工作流里加条件判断——如果输出格式不符合预期，停止执行并通知你。飞书多维表格设置异常值提醒。

### 翻车 2：没有人工接管点，流程卡住只能重来

❌ 错误做法：自动化流程在第 3 步卡住了，但前 2 步的结果没有保存，只能从头跑。

为什么错：中间结果只存在内存里，流程中断后全部丢失。

✅ 正确做法：每个关键步骤的输出都写入飞书文档或多维表格。流程中断时，人工从断点继续，不需要重来。

### 翻车 3：为了"全自动"删掉人工确认步骤

❌ 错误做法：觉得人工确认太慢，把所有确认步骤都去掉，追求"全自动"。

为什么错：省掉的 5 分钟确认时间，可能换来 5 小时的事故处理时间。

✅ 正确做法：只对最低风险的动作（如信息收集、格式整理）去掉确认。涉及对外发送、数据修改的动作，永远保留人工确认。
