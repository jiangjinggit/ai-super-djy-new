- 一份"不做什么"清单模板

## 什么是执行型工作台

执行型工作台是按任务启动的智能体，适合处理变化快、上下文重、需要交互式推进的任务。

代表产品：
- Claude Code：Anthropic 出品，擅长代码、文档、命令行协作，支持项目级上下文
- Kiro：AI IDE，集成了代码编辑、终端、文件管理，适合开发场景
- Cursor / Windsurf：AI 代码编辑器，适合日常开发辅助

> 截至 2026 年 3 月，功能和价格可能变化，请以官网为准。

### 它不只是写代码

在真实项目里，代码只是交付的一部分。执行型工作台还适合：

| 任务类型 | 举例 |
|----------|------|
| 阅读理解 | 读项目结构、理解模块关系 |
| 问题排查 | 查找 Bug 根因、分析报错日志 |
| 文档整理 | 补全注释、生成 API 文档、写变更说明 |
| 发布准备 | 生成发布说明、检查版本号、输出测试结论 |
| 知识提取 | 从代码里提取业务规则、整理技术决策记录 |

## 跟着做：掌握 5 步标准执行流程（⏱ 5 分钟）

以后给执行型工作台提任务，用这个固定顺序：

### 第 1 步：先读相关文件

让它先阅读相关代码/文档，总结现状。不要一上来就让它动手改。

### 第 2 步：输出执行计划

让它列出打算做什么、改哪些文件、不改哪些文件。你确认后再继续。

### 第 3 步：动手执行

按计划修改代码/文档。

### 第 4 步：跑验证

改完后运行测试、编译检查或人工审查。

### 第 5 步：输出变更说明和风险点

列出改了什么、为什么这么改、有什么潜在风险。

> ⚠️ 如果你跳过前两步直接让它改，返工率通常会很高。

✅ 完成标准：你能背出这 5 步的顺序，并且知道为什么前两步不能跳

## 跟着做：填写任务描述模板（⏱ 10 分钟）

### 模板

```text
目标：
[要解决什么问题]

范围：
[允许改哪些文件，不允许碰哪些文件]

上下文：
[相关模块、相关报错、关联逻辑]

执行要求：
1. 先阅读相关文件并总结现状
2. 给出简短计划
3. 再实施修改
4. 修改后运行验证
5. 输出风险和后续建议

不做什么：
[明确列出不允许的操作]

验收标准：
[什么算完成]
```

### 填写示例：修复一个日期显示 Bug

```text
目标：
修复课程页面的日期显示问题，当前显示为 "Invalid Date"

范围：
允许改：src/components/lesson-page/ 目录下的文件
不允许碰：src/content/ 目录、其他页面组件

上下文：
- 用户反馈课程页面底部的"最后更新时间"显示为 "Invalid Date"
- 可能是日期格式解析问题
- 相关组件：LessonMarkdown.tsx

执行要求：
1. 先阅读 LessonMarkdown.tsx，找到日期显示的代码
2. 给出修复计划
3. 实施修改
4. 确认页面正常显示日期
5. 输出改了什么、为什么这么改

不做什么：
- 不重构整个组件
- 不改日期数据的来源格式
- 不引入新的日期处理库

验收标准：
课程页面底部正确显示"最后更新：2026-03-31"格式的日期
```

✅ 完成标准：你用这个模板写了一个真实任务的描述，包含"不做什么"部分

## 一个很重要的习惯：写清"不做什么"（⏱ 5 分钟）

这会大幅减少模型过度发挥。常见的"不做什么"包括：

| 类别 | 示例 |
|------|------|
| 范围限制 | 不重构整个模块、不改无关文件 |
| 依赖限制 | 不引入新依赖、不升级现有依赖版本 |
| 接口限制 | 不改接口协议、不改数据库表结构 |
| 风格限制 | 不改代码风格、不重命名现有变量 |

> 💡 经验法则：如果你不写"不做什么"，模型会按自己的理解"顺便"做很多你没要求的事。

## 📋 案例：小张用执行型工作台修复跨文件 Bug

小张，前端开发，接到一个 Bug：用户在课程页面切换主题时，部分组件的颜色没有跟着变。

### 他的操作过程

| 步骤 | 做了什么 | 耗时 |
|------|---------|------|
| 写任务描述 | 用模板写清目标、范围、不做什么 | 5 分钟 |
| 第 1 步：读文件 | 让 Claude Code 读 ThemeProvider 和相关组件 | 2 分钟 |
| 第 2 步：看计划 | Claude Code 列出 3 个需要改的文件和原因 | 1 分钟 |
| 确认计划 | 小张确认计划合理，同意执行 | 1 分钟 |
| 第 3 步：执行 | Claude Code 修改 3 个文件 | 3 分钟 |
| 第 4 步：验证 | 运行编译检查，切换主题测试 | 3 分钟 |
| 第 5 步：总结 | Claude Code 输出变更说明和风险点 | 1 分钟 |
| 总计 | | 16 分钟 |

> "关键是第 2 步。以前我直接说'帮我修'，它经常改错文件。现在让它先出计划，我确认了再改，一次通过率高了很多。" —— 小张

### 对比：如果不用模板

小张之前试过直接说"主题切换有 Bug，帮我修一下"，结果：
- Claude Code 把整个 ThemeProvider 重构了
- 引入了一个新的 CSS-in-JS 库
- 改了 8 个文件，其中 5 个不需要改
- 花了 40 分钟排查它引入的新问题

## 常见翻车

### 翻车 1：目标没写清楚，模型过度发挥

❌ 错误做法：直接说"这个模块有点乱，帮我整理一下"

为什么错：目标太模糊，模型会按自己的理解重构整个模块，改动范围远超预期。

✅ 正确做法：写清楚"整理 XX 文件的函数顺序，不改逻辑、不改命名、不拆文件"。

### 翻车 2：没写"不做什么"，模型引入新依赖

❌ 错误做法：让它修一个日期格式问题，没说不能引入新库。结果它装了 dayjs 并改了 12 个文件的导入。

为什么错：模型倾向于用"更好的方案"解决问题，如果你不限制，它会自作主张。

✅ 正确做法：在任务描述里明确写"不引入新依赖，用现有的 Date API 解决"。

### 翻车 3：跳过"先读再改"，直接让它动手

❌ 错误做法：跳过第 1、2 步，直接说"改这个文件的第 30 行"

为什么错：它不了解上下文，可能改对了这一行但破坏了其他关联逻辑。

✅ 正确做法：即使你知道问题在哪，也让它先读相关文件、出计划，确认后再改。多花 3 分钟，少返工 30 分钟。

## 新手最实用的 3 类任务

| 任务 | 为什么适合新手 | 怎么开始 |
|------|--------------|---------|
| 小范围 Bug 修复 | 目标明确、范围小、容易验证 | 用模板写清 Bug 描述和范围限制 |
| 文档整理和说明补全 | 低风险、不影响功能、练习模板使用 | 让它先读代码再生成文档草稿 |
| 在已有结构里补测试 | 结构已有、只需补充、容易验证 | 指定测试框架和覆盖范围 |

暂时不要一开始就让它：
- 全量重构（范围太大，无法验证）
- 设计复杂架构（需要人的判断力）
- 在不熟悉的项目里大面积改代码（上下文不足，容易出错）
