## 这节课你要拿到什么

- 一张权限分层表（L1/L2/L3），直接套用到你的工作台
- 一个"先只读、后写入"的 4 周部署计划
- 一份部署前自检清单

## 什么是长期工作台

长期工作台是一种后台持续运行的智能体系统，适合处理高频、重复、流程清晰的任务。

代表产品：
- OpenClaw：支持多渠道接入（飞书、企微、Slack 等）、统一路由、后台长期运行
- Coze（扣子）：可视化搭建，支持定时触发和插件调用，新手门槛更低

> 截至 2026 年 3 月，功能和价格可能变化，请以官网为准。

### 长期工作台 vs 执行型工作台

| 维度 | 长期工作台（OpenClaw 型） | 执行型工作台（Claude Code 型） |
|------|-------------------------|-------------------------------|
| 运行方式 | 后台持续运行 | 按任务启动 |
| 适合任务 | 高频重复、流程固定 | 高变化、强上下文 |
| 典型场景 | 日报汇总、消息归类、巡检 | Bug 修复、文档整理、代码审查 |
| 核心优势 | 多渠道接入、长时间在线 | 上下文理解强、交互式推进 |
| 核心风险 | 权限没收紧 → 错误被放大 | 规则缺失 → 输出漂移 |

## 这类工作台最适合什么任务

只要任务满足下面 4 条，就适合先放进去：

- ✅ 高频重复（一周至少 3 次）
- ✅ 输入来源固定（不需要每次重新指定）
- ✅ 结果格式稳定（摘要、表格、通知等）
- ✅ 允许先生成草稿（不需要直接对外发送）

推荐起手任务：

| 任务 | 输入 | 输出 | 推荐工具 |
|------|------|------|---------|
| 日报汇总 | 多个渠道的消息 | 一份结构化摘要 | OpenClaw / Coze |
| 消息归类 | 群聊消息 | 按主题分类的列表 | OpenClaw / 飞书机器人 |
| 素材同步 | RSS / 网页 | 整理后存入知识库 | Coze + 飞书文档 |
| 巡检摘要 | 监控数据 | 异常项列表 | OpenClaw / Coze |
| 内容抓取整理 | 指定网站 | 格式化的内容卡片 | Coze |

## 跟着做：给你的工作台设计权限分层（⏱ 15 分钟）

这是本课最重要的部分。任何长期运行的系统，权限设计必须先于能力扩展。

### L1：只读层（第 1 周就可以开）

| 能力 | 说明 |
|------|------|
| 读取网页 | 仅限指定 URL 列表 |
| 读取文档 | 仅限指定目录 |
| 读取消息 | 仅限指定群/频道 |

### L2：草稿层（第 2 周稳定后开）

| 能力 | 说明 |
|------|------|
| 生成摘要 | 输出到草稿区，不直接发布 |
| 生成待办 | 写入个人待办列表 |
| 写入草稿区 | 仅限草稿目录，不碰正式文档 |

### L3：人工确认层（第 3 周以后，确认稳定再开）

| 能力 | 说明 | 确认方式 |
|------|------|---------|
| 发送消息 | 发到群/频道 | 必须人工确认后发送 |
| 回写正式文档 | 写入正式知识库 | 必须人工确认后写入 |
| 触发外部动作 | 调用第三方 API | 必须人工确认后执行 |

**关键原则：任何高风险动作都必须停在 L3 等人工确认。**

### 动手填你的权限表（⏱ 10 分钟）

打开飞书文档或 Notion，复制下面这张表，填上你的实际情况：

| 能力 | 是否开放 | 当前层级 | 说明 |
|------|---------|---------|------|
| 读文件 | 是 | L1 | 仅限 XX 目录 |
| 写草稿 | 是 | L2 | 仅限草稿目录 |
| 发消息 | 否/需确认 | L3 | 必须人工确认 |
| 执行命令 | 否/限定 | L3 | 仅允许白名单命令 |
| ... | | | |

✅ 完成标准：你的权限表至少有 5 行，每行都标明了层级和限制条件

## 跟着做：制定 4 周部署计划（⏱ 10 分钟）

### 第 1 周：只读 + 记录（L1）

- 只做信息收集和摘要
- 输出到自己能看到的位置（个人文档、测试群）
- 每天检查输出质量，记录到飞书多维表格

⏱ 每天投入：10 分钟检查 + 5 分钟记录

### 第 2 周：加草稿能力（L2）

- 开放草稿写入
- 加归档和分类功能
- 继续记录，关注格式是否稳定

⏱ 每天投入：10 分钟检查 + 5 分钟记录

### 第 3 周：加通知（L3 试运行）

- 加通知功能，但只发给自己或测试群
- 每条通知都要人工确认后再发
- 记录确认通过率（如果低于 90%，说明还不够稳定）

⏱ 每天投入：15 分钟（含确认操作）

### 第 4 周以后：逐步开放

- 在确认日志、回退、人工确认都稳定后，再逐步开放更强能力
- 每次只开放一个新能力，观察一周再开下一个

✅ 完成标准：你有一份 4 周计划，写清楚每周开放什么能力、怎么验证

## 📋 案例：小李用长期工作台做每日素材收集

小李，自媒体运营，每天要从 8 个公众号和 3 个行业网站收集素材，整理成选题列表。之前每天花 1 小时手动操作。

### 他的部署过程

| 阶段 | 做了什么 | 权限层级 | 效果 |
|------|---------|---------|------|
| 第 1 周 | 用 Coze 搭了素材抓取 Bot，只读取指定网站 | L1 | 每天自动收集，但格式不太稳定 |
| 第 2 周 | 调整了输出格式，加了分类标签，写入飞书草稿 | L2 | 格式稳定了，每天只需 10 分钟检查 |
| 第 3 周 | 加了飞书机器人通知，每天早上推送选题列表到自己 | L3（仅自己） | 通知准确率 95%，偶尔有重复 |
| 第 4 周 | 修复重复问题后，开放推送到团队群（仍需确认） | L3（团队） | 团队反馈好，每天省 45 分钟 |

> "最关键的决定是前两周只开 L1 和 L2。如果第一天就开推送，第三天肯定翻车。" —— 小李

## 常见翻车

### 翻车 1：一上来就开写入权限

❌ 错误做法：第一天就让工作台自动写入正式文档、自动发消息到团队群

为什么错：没有经过测试期，输出格式和内容质量都不稳定。自动写入正式文档后，团队看到的是未经确认的内容。

✅ 正确做法：前两周只开 L1（只读）和 L2（草稿），确认输出稳定后再开 L3。

### 翻车 2：没设日志，出问题后查不到原因

❌ 错误做法：工作台跑了两周，某天输出突然变差，但不知道是输入变了还是模型变了。

为什么错：没有记录每次执行的输入、输出和异常情况，无法定位问题。

✅ 正确做法：从第一天起就用飞书多维表格记录每次执行的关键信息（输入来源、输出条数、是否异常、人工修改次数）。

### 翻车 3：权限没有分层，所有能力一次性全开

❌ 错误做法：觉得"反正都要用到"，一次性把读取、写入、发送、执行命令全部开放。

为什么错：后台长期运行的系统，一旦有过大权限，错误会被自动放大。比如一个格式 Bug 导致每天自动发 10 条错误消息。

✅ 正确做法：严格按 L1 → L2 → L3 的顺序逐步开放，每个层级至少稳定运行一周再升级。
