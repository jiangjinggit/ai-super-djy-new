## 学习目标

- 建立 AI 编程的最小治理框架
- 分清本地执行、云端执行、企业接入的风险差异
- 学会给权限、Secrets、外部请求、Git 操作和审计设置边界
- 能写出一版可执行的团队治理清单

## 适合谁

- 已经开始在真实仓库里使用 AI 编程的人
- 准备把个人使用扩展到团队的人
- 需要回答“能不能安全推广”的技术负责人

## 前置条件

- 了解自己的默认工具入口和模型分工
- 至少做过一次真实改动和一次验证
- 接受“AI 编程的最大风险通常不是写错一行代码，而是流程失控”

## 核心知识

### 1. 治理不是企业附加题，而是所有人的基础题

AI 编程工具现在能做的事情已经很多：

- 读仓库
- 改多文件
- 跑命令
- 发外部请求
- 安装依赖
- 生成 PR
- 在部分场景里做异步委托

这意味着风险也不再是“回答错了一个知识点”，而是：

- 改错仓库
- 泄露敏感信息
- 误发请求
- 把错误改动送进主分支

所以治理必须从第一天就有。

### 2. 三种执行方式，三种风险结构

| 执行方式 | 主要风险 | 默认建议 |
| --- | --- | --- |
| 本地执行 | 误改文件、误跑命令、误动 Git | 用权限模式、规则文件、人工确认点控制 |
| 云端执行 | 数据出边界、环境不一致、结果难追踪 | 先做任务分级，明确哪些仓库和任务可委托 |
| 企业接入 | 账号权限、知识库污染、组织级误推广 | 强制规则、审计、审批、灰度上线 |

### 3. 最小权限边界怎么定

建议先把任务分成 3 档：

#### 低风险

- 读代码
- 总结目录
- 草拟测试
- 生成文档

这类任务可以更开放一些。

#### 中风险

- 改局部代码
- 跑测试
- 调整配置但不触生产
- 生成 PR 草稿

这类任务需要明确计划和验证。

#### 高风险

- 删文件
- 改 CI/CD
- 推远端分支
- 动生产配置
- 访问 Secrets
- 发外部请求

这类任务必须有人工确认点，默认不允许静默执行。

### 4. 你真正要管的是五件事

#### 1. 权限

谁能读、谁能写、谁能执行命令、谁能发请求、谁能访问生产资源。

#### 2. 数据

哪些仓库、配置、日志、客户数据可以进入 AI 工作流，哪些绝对不行。

#### 3. 审计

至少要能追踪：

- 谁发起了任务
- 任务改了什么
- 是否跑过验证
- 谁批准了高风险动作

#### 4. 成本

不要只看 token 单价，还要看：

- 返工次数
- 等待时间
- 人工 review 时间
- 异步失败重跑成本

#### 5. 回退

没有回退，就不要开高权限。

最小回退包括：

- 备份分支
- 变更 diff
- 手动验证步骤
- 快速恢复手段

### 5. 一份能落地的治理清单

| 类别 | 最小规则 |
| --- | --- |
| 仓库分级 | 把仓库分成可公开试验、团队内部、敏感核心三档 |
| 任务分级 | 低 / 中 / 高风险任务分别定义默认权限 |
| Secrets | 默认不可见，只有白名单任务能访问 |
| 外部请求 | 默认禁止，显式允许才可执行 |
| Git | 允许本地 diff，不默认允许推远端 |
| 审计 | 高风险任务必须留执行记录和验收记录 |

## Demo

### 给一条 AI 编程流程补齐治理边界

场景：团队要让 AI 参与“补测试 + 生成 PR 说明 + 初步 Review”。

可以这样补边界：

1. **权限**  
   只允许读仓库、改测试文件、生成 PR 说明，不允许改生产配置

2. **数据**  
   不读取 `.env`、不读取生产凭证文件

3. **外部请求**  
   默认禁止联网，除非任务明确要求查官方文档

4. **审计**  
   每次任务要记录：提示词、改动文件、测试结果、最终审批人

5. **回退**  
   所有变更都要能通过 Git 一键回退

这个 Demo 的重点是：  
治理不是“把 AI 禁掉”，而是让它在清楚的边界内稳定发挥价值。

## 作业

给你的主流程补一份最小治理规则，至少写出下面 5 行：

1. 哪类任务默认允许  
2. 哪类任务默认禁止  
3. 哪类任务必须人工确认  
4. 哪类数据不允许进入  
5. 出问题时怎么回退

## 常见误区

1. **把治理理解成“大公司才需要”**  
个人开发者同样会遇到误删、误改、误推送的问题。

2. **只管权限，不管回退**  
权限再细，如果出了事没有恢复路径，风险仍然很高。

3. **只看 token 成本，不看总成本**  
返工和审核时间往往比模型调用费更贵。

4. **高风险动作靠“模型自己判断”**  
这是最不该外包给模型的一类决策。

## 验收标准

- 能说清本地、云端、企业接入三种执行方式的风险差异
- 能把任务至少分成低 / 中 / 高风险三档
- 能写出一版覆盖权限、数据、审计、成本、回退的最小治理清单
- 能指出自己当前流程里最危险的 1-2 个薄弱点
