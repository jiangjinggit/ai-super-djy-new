## 学习目标

- 建立 AI 编程从 `Autocomplete -> Chat -> Agent -> Cloud Agent -> Spec-driven` 的演进认知
- 分清 `CLI Agent / IDE Agent / Cloud Agent / Review Bot` 四类主流入口
- 理解“选工具”与“选模型”是两层不同决策
- 能把 `Claude Code / Codex / Cursor / Gemini / Kiro` 正确放进同一张坐标图

## 适合谁

- 想系统进入 AI 编程，但现在脑子里只有一堆工具名的人
- 已经在用某个 AI 编程产品，但说不清它跟别家的本质差异的人
- 需要给自己或团队做工具选型的人

## 前置条件

- 知道代码仓库、分支、测试、PR 是什么
- 至少体验过一次 AI 写代码或解释代码
- 不要求会高级 Prompt，但要愿意按流程做任务拆解

## 核心知识

### 1. AI 编程不是一件事，而是一条演进线

过去 3 年，AI 编程大致经历了 5 个阶段：

| 阶段 | 典型形态 | 解决什么问题 | 主要局限 |
| --- | --- | --- | --- |
| Autocomplete | 代码补全插件 | 减少重复敲字 | 只能改局部，不理解完整任务 |
| Chat | IDE 里的聊天框 | 问问题、要解释、要小片段代码 | 会讲不会做，执行链路断开 |
| Agent | 能读仓库、改文件、跑命令 | 把任务推进到“可执行” | 仍需要人盯着边界 |
| Cloud Agent | 云端异步执行、并行跑任务 | 降低等待成本，适合委托型任务 | 数据、审计、回退要求更高 |
| Spec-driven | 先写规格、再生成实现 | 降低多人协作中的跑偏风险 | 对团队流程要求更高 |

这条演进线的核心变化，不是模型变聪明了一点，而是产品开始接管更多“执行链路”。

### 2. 现在主流产品到底在竞争什么

很多人以为这些产品都在做同一件事，差别只是模型厂商不同。实际上，主战场已经变成 4 个维度：

1. **入口**：终端、IDE、云端任务台，还是规格驱动工作台  
2. **执行深度**：只会写片段，还是能读目录、改多文件、跑验证  
3. **上下文系统**：是否支持项目规则、记忆、规范文件、子任务拆分  
4. **治理能力**：权限、审计、回退、团队协作能不能落地

所以现在选工具，不能只问一句“哪个模型强”，而要先问：

- 我的任务是在编辑器里盯着做，还是可以丢出去异步做？
- 我需要一个聪明助手，还是需要一个可治理的执行系统？
- 我更缺单次输出质量，还是更缺一条稳定流程？

### 3. 四类主流入口的任务边界

| 入口 | 代表产品 | 最适合 | 不适合 |
| --- | --- | --- | --- |
| CLI Agent | Claude Code、Gemini CLI、Codex CLI | 本地仓库、小步快跑、强监督、改完马上验 | 想把很多任务丢后台异步处理 |
| IDE Agent | Cursor、通义灵码、TRAE、CodeBuddy IDE | 边看代码边改、上下文切换频繁、团队日常开发 | 需要大规模并行委托的任务 |
| Cloud Agent | Codex、Cursor Background Agents | 可拆分、等待成本高、能异步回收结果的任务 | 高敏感、强依赖本地环境的仓库 |
| Spec-driven / Workflow IDE | Kiro、部分企业平台 | 需求明确、多人协作、需要先写规格再实现 | 极小改动、一次性脚手架需求 |

第一层判断通常不是“我喜欢哪个产品”，而是“我的任务属于哪个入口”。

### 4. 选工具不等于选模型

把这两个概念拆开：

| 层级 | 你在选什么 | 例子 |
| --- | --- | --- |
| 工具层 | 入口、工作流、规则、治理、协作方式 | Claude Code、Cursor、Kiro、通义灵码 |
| 模型层 | 推理质量、成本、速度、上下文、中文能力 | Claude、GPT/Codex、Gemini、Qwen、DeepSeek |

同一个工具可以接多种模型；同一个模型家族也可以被多个工具调用。

真正成熟的 AI 编程配置通常是：

- 工具层：定一条默认入口
- 模型层：按任务分工配置主力、回退、跑量
- 流程层：定读、计划、执行、验证、总结

### 5. 一张够用的 AI 编程工具坐标图

你可以先用下面这张“任务视角”坐标图记住主流产品：

| 工具 | 主入口 | 核心气质 | 第一默认场景 |
| --- | --- | --- | --- |
| Claude Code | 终端 | 本地、强监督、规则丰富 | 仓库内真实改动与验证 |
| Codex | 云端 + CLI | 异步委托、并行、远程执行 | 可拆分、等待成本高的编码任务 |
| Cursor | IDE | 编辑器原生 Agent、协作顺滑 | 日常开发与 Review |
| Gemini CLI / Code Assist | 终端 + IDE | 开放入口、Google 生态、多模态 | 终端代理与云生态协同 |
| Kiro | IDE | 规格驱动、流程先行 | 需求到实现的团队型流程 |

这张表现在只解决一件事：让你知道这些工具不是同一种东西。

## Demo

### 同一个任务，三种路线的差异

任务：给现有 Web 项目增加“CSV 批量导入用户”的后台功能。

**路线 A：聊天式提问**

- 你把需求丢给聊天框
- 它给你一份实现方案和示例代码
- 你自己负责找文件、改代码、跑测试、修 bug

适合：纯学习、原型讨论、还没准备动仓库的时候。

**路线 B：IDE Agent**

- 在编辑器里让 Agent 读相关目录
- 它边解释边改文件
- 你实时审批、实时修正方向

适合：边看边改、上下文频繁跳转、你不想离开编辑器的时候。

**路线 C：CLI / Cloud Agent**

- 先给仓库级上下文和限制
- 让它读、出计划、执行、验证
- 对可并行子任务，再委托云端异步跑

适合：范围明确、需要改多文件、需要统一验证标准的时候。

**这个 Demo 的重点不是谁更强，而是谁更省你的总时间。**

## 常见误区

1. **把 AI 编程理解成“更聪明的代码补全”**  
这会让你一直停留在局部交互层，看不到工作流价值。

2. **用同一种工具硬吃所有任务**  
终端、IDE、云端、规格驱动各有边界，入口选错，返工会非常高。

3. **把工具评测当模型评测**  
一个工具不好用，可能是入口不适合，不一定是模型太差。

4. **只看一次生成效果，不看总拥有成本**  
一次输出漂亮，不等于真实开发最省时间。
