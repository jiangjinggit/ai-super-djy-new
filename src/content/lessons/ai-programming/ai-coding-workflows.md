## 学习目标

- 把 AI 编程从“单轮对话”升级成“可复用流程”
- 掌握个人开发、小团队、PR 评审、多 Agent 协作四类工作流
- 学会把 `读 -> 计划 -> 执行 -> 验证 -> 总结` 写成默认 SOP
- 能为自己的真实流程选一条第一默认路径

## 适合谁

- 已经会让 AI 写点代码，但流程仍然混乱的人
- 想把 AI 纳入需求、实现、测试、评审链路的人
- 需要把个人实践抽象成团队做法的人

## 前置条件

- 选过一条默认工具入口
- 对模型角色已有初步判断
- 愿意先从一条小流程跑通，而不是一上来改造全链路

## 核心知识

### 1. 工作流设计比 Prompt 重要

Prompt 写得再好，如果流程是乱的，最终也会表现成：

- 改动范围越来越飘
- 验证没人做
- AI 做了很多“看起来很忙”的动作
- 最后真正能进仓库的东西很少

所以工作流的目标不是“让 AI 更自由”，而是：

> 让 AI 在正确的地方有自由，在关键边界没有自由。

### 2. 四种最常见的 AI 编程工作流

#### A. 单兵开发流

适合：

- 个人开发者
- 小型需求
- 高频小改动

默认节奏：

1. 读相关文件  
2. 出计划  
3. 执行  
4. 验证  
5. 总结

这是最容易落地的一条主线，也是大多数人第一条应该跑稳的流程。

#### B. 结对编程流

适合：

- 你仍想牢牢掌握实现方向
- 但希望 AI 承担搜索、起草、改代码、补测试这些动作

人负责：

- 目标
- 验收标准
- 边界判断

AI 负责：

- 阅读与归纳
- 改动执行
- 自检
- 总结

#### C. PR 评审流

适合：

- 团队已开始规范评审
- 希望让 AI 先做一轮初筛

常见结构：

1. AI 先看 diff  
2. 标记高风险点  
3. 列出可能漏测和边界问题  
4. 人类 reviewer 重点看这些位置  
5. 必要时再让 AI 回头补测试或文档

#### D. 多 Agent / 异步协作流

适合：

- 任务可以拆分
- 你不想全程盯着执行
- 团队愿意为并行收益付治理成本

这条路线的关键不是“Agent 越多越好”，而是：

- 任务边界清晰
- 子任务彼此低耦合
- 最后有统一验收人

### 3. 一套默认 SOP

无论你用什么工具，都建议先用下面这套最小模板：

#### 第一步：读

让 AI 先读文件，说出理解，不要急着改。

#### 第二步：计划

明确：

- 会改哪些文件
- 不改哪些文件
- 哪些点不确定

#### 第三步：执行

在批准后执行，并明确禁止项。

#### 第四步：验证

至少做两类验证：

- AI 自检
- 命令 / 测试 / 手工验证

#### 第五步：总结

输出：

- 改了什么
- 为什么这样改
- 还有什么风险

### 4. 三套默认流程模板

| 场景 | 推荐默认栈 | 重点 |
| --- | --- | --- |
| 个人开发 | CLI Agent 或 IDE Agent + 主力模型 | 快速、可控、少折腾 |
| 小团队 | IDE Agent + PR Review + 明确规则 | 保证协作与评审质量 |
| 企业团队 | IDE / CLI / 云端混合 + 审批与审计 | 流程、权限、可追溯性 |

### 5. 什么时候该升到多 Agent

满足 3 条再考虑：

- 任务可明确拆分  
- 单 Agent 已经稳定  
- 你真的因为等待成本受限  

如果单 Agent 还没跑稳，就不要过早追求“多代理编排”。

## Demo

### 设计一条 `需求 -> 实现 -> 测试 -> PR -> Review` 流水线

以“新增 CSV 导入功能”为例：

1. **需求阶段**  
   用 Kiro / 文档工具先把规格和验收标准写清楚

2. **实现阶段**  
   用 Claude Code / Cursor / 通义灵码让 AI 读目录、出计划、执行改动

3. **测试阶段**  
   让 AI 补测试、列边界情况、跑命令验证

4. **PR 阶段**  
   让 AI 产出变更摘要、风险说明、测试结论

5. **Review 阶段**  
   用 AI 做第一轮审查，人类 reviewer 最终裁决

这条流程的核心不是用了多少工具，而是每一段都知道“AI 负责什么，人负责什么”。

## 作业

把你当前最常见的一条开发流程写成 5 列表：

| 当前步骤 | 谁在做 | AI 能不能接手 | 风险点 | 你准备怎么验证 |
| --- | --- | --- | --- | --- |

然后把其中 **最容易标准化的一步**，改造成 AI 默认流程。

建议先改这一类：

- 读仓库总结
- 改小 bug
- 补测试
- 生成 PR 说明

## 常见误区

1. **把 AI 编程理解成“让 AI 自己想怎么做”**  
没有流程约束，往往只会放大不稳定性。

2. **一开始就想做全链路自动化**  
最容易失败。先把单条流程跑稳，再谈自动化。

3. **只做执行，不做验证**  
这样最后所有质量责任都会回到人身上，而且更晚暴露问题。

4. **多 Agent 只是把混乱放大**  
任务边界没写清时，Agent 越多，最后越难收口。

## 验收标准

- 能描述至少 3 类 AI 编程工作流
- 能写出一套 `读 -> 计划 -> 执行 -> 验证 -> 总结` 的默认 SOP
- 能指出自己最适合先改造哪一段开发链路
- 能判断自己现在是否真的适合上多 Agent
