## 学习目标

- 建立国内 AI 编程工具的判断框架，而不是只靠“国外更强 / 国内更方便”的印象做选择
- 分清通义灵码、TRAE、CodeBuddy 这 3 条主要产品路线
- 理解中国用户的真实落地约束：中文、采购、网络、合规、云生态
- 带走一张 `国内 AI 编程工具选择表`

## 适合谁

- 在中国环境下要落地 AI 编程的个人开发者和团队负责人
- 需要考虑企业采购、权限、知识库、云厂商协同的人
- 想知道“国产工具是不是只适合入门”的人

## 前置条件

- 已经理解国外主流工具是怎么分入口和流程的
- 了解自己所在团队是否有采购、合规和数据边界要求
- 接受“能力上限”和“落地摩擦”是两套评价体系

## 核心知识

### 1. 中国用户的选型，必须多看四个现实约束

国外工具常常在产品创新上领先，但中国用户做真实选型时，通常还要额外看这 4 件事：

1. **中文体验**：代码注释、文档说明、团队沟通是否更顺  
2. **采购与支付**：个人是否能直接买，企业是否能走正式采购  
3. **网络与账号**：账号创建、支付方式、网络稳定性是否会影响日常使用  
4. **云生态与合规**：是否能接入阿里云、火山引擎、腾讯云，以及组织级权限控制

也就是说，国内工具不只是“能力的低配版”，而是另一条落地路线。

### 2. 通义灵码：阿里路线的关键词是“普及 + 企业接入”

通义灵码的典型特征是：

- IDE 入口成熟
- 面向国内开发者普及度高
- 与阿里生态、企业知识和组织级能力结合更自然
- 对中文文档、代码解释和团队推广更友好

如果你的团队特点是：

- 希望快速推开
- 已经在阿里云体系里
- 对企业采购和合规更敏感

那通义灵码通常是一个现实阻力较小的起点。

### 3. TRAE：字节路线更像“AI IDE 产品化”

TRAE 的亮点通常不只是模型，而是产品组织方式：

- 更完整的 AI IDE 体验
- 强调 Builder、项目级能力、MCP 和规则配置
- 中文界面与产品感更强
- 对“边开发边让 AI 参与多个环节”的体验打磨较多

它适合：

- 喜欢 IDE 主工作台
- 希望少折腾、多拿到成品体验
- 需要比较顺滑的中文交互

### 4. CodeBuddy：腾讯路线更强调“团队协作与企业治理”

CodeBuddy 这条线更值得看的是：

- IDE、CLI、团队协作入口并存
- 更容易纳入腾讯云和企业开发体系
- 更强调多角色、多阶段、组织级推广
- 对技术管理者来说，治理和接入常常比“单次最强效果”更重要

它适合：

- 团队规模不小
- 有统一开发规范
- 后续要和腾讯云或企业平台联动

### 5. 一张国内工具选择表

| 判断维度 | 通义灵码 | TRAE | CodeBuddy |
| --- | --- | --- | --- |
| 默认入口 | IDE / 企业开发场景 | AI IDE | IDE + CLI + 团队协作 |
| 最大优势 | 普及快、中文顺、企业接入稳 | 产品化体验强、AI IDE 感明显 | 团队治理、企业协作、腾讯云生态 |
| 适合谁 | 想快速推广的开发团队 | 重体验、重 IDE 主工作台的团队 | 需要组织级治理的团队 |
| 主要风险 | 容易只停留在辅助层 | 如果流程没设计好，容易把 AI IDE 当聊天框 | 个人开发者可能觉得“体系偏重” |

### 6. 国内工具什么时候会优先于国外工具

如果你满足下面任意 3 条，国内工具往往值得优先试：

- 团队主要工作语言是中文
- 需要正式采购和发票流程
- 需要国内云厂商协同
- 仓库和数据不适合轻易放到海外流程里
- 你的团队更重视可推广性，而不是前沿功能尝鲜

反过来，如果你最看重的是：

- 最新 Agent 工作流
- 最新异步委托能力
- 最前沿产品设计

那国外工具通常更值得先试。

## Demo

### 同一个中文业务需求，分别设计 3 条国内落地方案

需求：给电商中台加一个“活动价批量校验”功能。

**通义灵码方案**

- 以 IDE 内日常开发和团队推广为主
- 先做代码理解、批量修改建议、单仓库提效
- 重点在“让更多人先用起来”

**TRAE 方案**

- 以 AI IDE 为主工作台
- 让 AI 参与需求拆解、代码理解、实现与调试
- 重点在“把多个开发环节集中到一个工作台里”

**CodeBuddy 方案**

- 以团队协同、规则、流程接入为主
- 强化组织级的规范、审计、多人协作
- 重点在“长期可治理地推广”

这个 Demo 的重点是：  
同一个需求，不同工具路线解决的是不同层级的问题。

## 常见误区

1. **“国外一定更强，所以国产不用看”**  
这忽略了真实落地里的采购、合规、网络和团队推广成本。

2. **“国产更方便，所以能力也够了”**  
便利和上限不是一回事，复杂任务和前沿工作流仍要单独评估。

3. **只看个人体验，不看团队推广**  
一个高手用得爽，不代表十个人一起用也能跑顺。

4. **只看单次生成效果，不看接入能力**  
企业最关心的经常不是一段代码写得多漂亮，而是能不能纳入既有流程。

## 验收标准

- 能说清通义灵码、TRAE、CodeBuddy 各自更适合什么团队
- 能解释中国用户为什么需要一套不同于海外用户的判断框架
- 能根据自己的场景筛出 1-2 条优先试用路线
- 能区分“能力上限”和“落地摩擦”这两种评价维度
