## 学习目标

- 把前 7 课的地图收口成 30 天可执行路线
- 为个人开发者、小团队、企业技术管理者各设计一条默认方案
- 学会在 `国际优先 / 国内优先 / 混合栈` 三种路线里做选择
- 最终带走一张适合自己的 AI 编程落地表

## 适合谁

- 不想继续收藏工具，而是想真的开始落地的人
- 想在中国环境下用相对低风险的方式推进 AI 编程的人
- 需要给自己或团队做一个试运行计划的人

## 前置条件

- 已经完成工具入口、模型分工和治理边界的基本判断
- 愿意接受“先做小闭环，再扩规模”的推进方式
- 至少有 1 个真实仓库或真实开发任务可用于试运行

## 核心知识

### 1. 先选路线，不要先选全家桶

对中国用户来说，最实用的路线通常只有 3 条：

#### A. 国际优先

适合：

- 追求前沿工作流
- 英文资料和海外产品接受度高
- 对产品创新速度非常敏感

典型组合：

- 工具：Claude Code / Cursor / Codex / Kiro
- 模型：Claude / GPT-Codex / Gemini

优点：

- 前沿能力更新快
- Agentic Coding 范式通常领先

风险：

- 账号、支付、网络、企业合规摩擦更大

#### B. 国内优先

适合：

- 中文和企业接入更重要
- 团队需要正式采购、权限和生态协同
- 更重视推广成本和组织阻力

典型组合：

- 工具：TRAE / 通义灵码 / Qoder / CodeBuddy
- 模型：Qwen / DeepSeek / Kimi / GLM / 豆包 / MiniMax

优点：

- 中文顺、采购顺、接入顺

风险：

- 某些前沿工作流和产品创新节奏未必领先

#### C. 混合栈

适合：

- 既要兼顾前沿能力，又要控制中国环境下的落地摩擦

典型组合：

- 日常开发：国内 IDE 或团队工具
- 复杂任务 / 关键任务：国际高质量模型与前沿 Agent

优点：

- 兼顾效率、能力上限和现实约束

风险：

- 配置更复杂，需要更清晰的边界

### 2. 三类人群的默认方案

#### 个人开发者

目标不是“最全工具链”，而是 **最快形成稳定习惯**。

建议：

- 先选一个主入口：CLI Agent 或 IDE Agent
- 再选一个主力模型和一个回退模型
- 先跑通 1 条小闭环：读代码 -> 改小 bug -> 跑测试 -> 写总结

#### 小团队 / 创业团队

目标是 **既提效，又不把流程搞乱**。

建议：

- 先统一 1 个日常入口
- 给 PR 评审和测试补齐设置标准用法
- 不急着全团队多 Agent，先统一 SOP 和规则文件

#### 企业技术管理者

目标是 **可控推广**，不是先追最强单点效果。

建议：

- 先挑低风险团队做试点
- 先从低风险任务切入：读仓库、补测试、写文档、PR 摘要
- 先把权限、审计、知识边界和回退方案做出来

### 3. 一张 30 天路线图

| 时间 | 重点 | 交付物 |
| --- | --- | --- |
| 第 1 周 | 选入口 | 1 个主入口 + 1 个备选入口 |
| 第 2 周 | 选模型 | 1 个主力模型 + 1 个回退模型 + 1 条评测表 |
| 第 3 周 | 跑流程 | 1 条完整开发 SOP |
| 第 4 周 | 补治理 | 1 份最小治理清单 + 试运行结果 |

### 4. 一条你现在就能执行的默认策略

如果你不想纠结太久，可以先用下面这条：

#### 个人开发者默认版

- 主入口：你最顺手的 IDE Agent 或 CLI Agent
- 主力模型：1 个均衡模型
- 回退模型：1 个中文或低成本模型
- 第一任务：修 1 个小 bug 或补 1 组测试

#### 小团队默认版

- 主入口：统一一个 IDE Agent
- 共同规则：项目级规则文件 + PR 模板
- 第一任务：PR 说明、代码理解、测试补齐

#### 企业默认版

- 主入口：企业已采购或最易纳管的工具
- 第一步：限制在低风险任务灰度
- 第一任务：文档总结、代码解释、测试草稿、Review 初筛

### 5. 什么时候应该加码，什么时候应该降级

#### 可以加码的信号

- 一次通过率明显提升
- 团队成员愿意持续用
- 验证成本下降
- 治理规则已跑顺

#### 应该降级的信号

- AI 每次都要重讲规则
- 返工量没有下降
- 高风险动作边界不清
- 团队成员使用方式差异太大，无法沉淀标准

## Demo

### 给三类人设计 30 天落地方案

**A. 个人独立开发者**

- 第 1 周：只选一个主入口
- 第 2 周：只定一个主力模型和回退模型
- 第 3 周：跑通小 bug 修复工作流
- 第 4 周：加入测试补齐或 PR 总结

**B. 3-8 人小团队**

- 第 1 周：统一 IDE Agent 或 CLI Agent
- 第 2 周：统一 PR 模板和规则文件
- 第 3 周：让 AI 先承担代码理解、测试草稿、PR 摘要
- 第 4 周：评估是否进入 Review 初筛

**C. 企业技术负责人**

- 第 1 周：选试点团队和试点仓库
- 第 2 周：完成权限和数据边界划分
- 第 3 周：限定在低风险任务内试运行
- 第 4 周：复盘结果，决定是否扩大范围

## 常见误区

1. **一开始就想把所有工具都接进来**  
工具越多，初期噪音越大，越不容易跑出稳定习惯。

2. **把“试用很多产品”误当成“已经在落地”**  
没有真实仓库和真实任务，所有体验都只是试玩。

3. **团队推广先追求最强效果，不先做最小闭环**  
最后往往因为复杂度太高而放弃。

4. **明明应该混合栈，却执着于只选单一阵营**  
中国用户很多时候真正最优的就是混合方案。

## 验收标准

- 能说明自己更适合国际优先、国内优先还是混合栈
- 能写出一条未来 30 天可执行的路线
- 能明确自己的主入口、主力模型和第一个真实任务
- 能定义 30 天后复盘是否继续推进的判断标准
