## 核心知识

### 1. Claude Code 代表什么

Claude Code 代表的是一条非常清晰的路线：

- **本地仓库优先**
- **终端为主入口**
- **强监督、强规则、强权限边界**
- **适合一边看 diff、一边验结果**

它的价值不只是“会改代码”，而是把这些能力装进了同一条工作链：

- 读文件和目录
- 运行命令
- 根据 `CLAUDE.md`、Rules、Memory 读上下文
- 通过权限模式控制风险
- 必要时拆成子任务或连接工具链

它的默认姿势更像：

> 你坐在终端前，AI 是一个能执行的结对工程师，但你始终在回路里。

### 2. Codex 代表什么

Codex 这条线更强调：

- **把任务封装成委托**
- **支持云端异步处理**
- **适合并行、适合等待成本高的任务**
- **更接近“把任务交给编码代理排队推进”**

这条路线不是为了替代本地开发，而是为了改变“人必须一直盯着 AI”的成本结构。

它更适合的任务通常有这些特征：

- 可以被描述成比较完整的任务单
- 你愿意等待一段时间再回来验收
- 任务之间可以并行
- 本地环境依赖不是特别强，或可被云端环境复现

一句话区分：

- Claude Code 更像 **前台结对工程师**
- Codex 更像 **后台任务代理**

### 3. 两条路线的本质差异

| 维度 | Claude Code | Codex |
| --- | --- | --- |
| 默认场景 | 本地仓库、即时推进 | 云端委托、异步处理 |
| 工作节奏 | 读一段、计划一段、执行一段、马上验 | 把任务打包、委托出去、等待结果 |
| 人在回路中的位置 | 高频介入 | 低频介入、重点做验收 |
| 最适合的任务 | 高敏感、强依赖本地环境、改动风险高 | 可拆分、等待成本高、适合并行 |
| 核心能力焦点 | 权限、上下文、终端执行 | 任务委托、并行、云端回收 |

注意，这不是二选一，而是两条默认入口。

真正成熟的团队常见做法是：

- 高敏感仓库、核心改动：本地终端 Agent
- 中低风险、可排队任务：云端异步 Agent

### 4. 怎么选：看三件事

#### 第一件事：仓库和数据有多敏感

如果任务涉及：

- 客户代码
- 生产配置
- 内部密钥
- 合规要求严格的仓库

默认先选 **本地强监督路径**。

#### 第二件事：你愿不愿意为“等待”付钱

如果你正在推进一个复杂功能，且很多子任务可以并行：

- 文档整理
- 测试补齐
- 不同模块的静态检查
- 多个候选实现方案

那云端委托的价值就会非常高，因为它把你的等待时间释放出来。

#### 第三件事：本地环境依赖强不强

如果任务高度依赖：

- 本地数据库
- 内网服务
- 私有包仓库
- 难以复刻的开发环境

那就更适合本地终端 Agent。

### 5. 一份够用的判断模板

在每个任务开始前，先过这张表：

| 问题 | 如果答案是“是” | 默认选择 |
| --- | --- | --- |
| 任务需要读取高敏感仓库或 Secrets 吗？ | 是 | Claude Code 一类本地路径 |
| 任务依赖复杂本地环境或内网吗？ | 是 | Claude Code 一类本地路径 |
| 任务可以被完整描述并异步等待吗？ | 是 | Codex 一类云端路径 |
| 任务能拆成多个相互独立子任务吗？ | 是 | Codex 一类云端路径 |
| 你需要边看边调、频繁改方向吗？ | 是 | Claude Code 一类本地路径 |

## Demo

### 同一个真实任务，分别设计 Claude Code 路径和 Codex 路径

任务：给一个已有 SaaS 项目增加“导出订单报表”的功能。

**Claude Code 路径**

1. 先让它读取订单模块、导出模块、权限判断相关文件  
2. 让它先出计划，明确改哪些文件、不改哪些文件  
3. 你审批后执行  
4. 改完立刻跑测试和手动验证  
5. 最后看 diff 和总结

适合原因：

- 涉及权限与导出格式，风险不低
- 需要看清改动边界
- 你可能中途频繁修正方向

**Codex 路径**

1. 把任务拆成几个可委托子任务  
2. 例如：导出接口草案、前端下载交互、测试用例补齐、文档更新  
3. 并行委托后，等结果回来统一验收  
4. 最终只对需要保守处理的部分做人手收口

适合原因：

- 可以拆分
- 等待成本高
- 你更想把时间放在验收而不是全程盯执行

## 常见误区

1. **把 Codex 当成“更远程的聊天框”**  
它真正的价值在异步委托和并行处理，不只是把对话挪到云上。

2. **把 Claude Code 当成“只能做小改动”**  
它能处理复杂任务，但前提是你把上下文、计划和权限边界设清楚。

3. **用云端代理处理不该离开本地的仓库**  
只看效率，不看数据边界，是最容易出治理事故的做法。

4. **把所有任务都丢给本地终端 Agent**  
这样会让你成为系统瓶颈，浪费可并行任务带来的时间收益。
