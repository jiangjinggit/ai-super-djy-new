## 学习目标

- 建立“工具层”和“模型层”分开决策的习惯
- 理解代码任务的模型评估不能只看 benchmark
- 学会给代码任务配置 `主力模型 / 回退模型 / 跑量模型`
- 能用模型家族视角看 `Claude / GPT-Codex / Gemini / Qwen / DeepSeek / 混元`

## 适合谁

- 工具已经大致选定，但模型总在跟风切换的人
- 想降低返工率和总成本的人
- 需要给团队写默认模型策略的人

## 前置条件

- 已经区分“工具”和“模型”是两层不同配置
- 至少做过几次真实代码任务，不再满足于“看演示很强”
- 愿意用任务结果而不是品牌印象做判断

## 核心知识

### 1. 代码任务为什么不能只看 benchmark

很多 benchmark 只能回答“模型在特定题库上表现如何”，但真实代码任务还要看另外 4 件事：

1. **仓库理解能力**：能不能读懂你的目录、约束和历史代码  
2. **执行稳定性**：是不是经常在中途偏题、过度重构、漏改边界  
3. **验证成本**：你为了修它的输出，还要花多少人工时间  
4. **协作适配**：它在你选定的工具和工作流里表现是否稳定

所以真正该比较的是：

> 某个模型在我的真实仓库和真实任务里，能不能以可接受的返工率完成工作。

### 2. 最实用的四种模型角色

先不要问“谁最强”，先问“这个角色谁来担”。

| 角色 | 典型任务 | 你真正要看什么 |
| --- | --- | --- |
| 高质量主力 | 复杂功能、跨文件重构、关键 PR | 理解深度、稳定性、长链路表现 |
| 日常默认 | 中小功能、代码解释、测试补齐 | 速度、成本、整体均衡 |
| 低成本跑量 | 批量分析、预处理、初稿、静态整理 | 单价、吞吐、基本稳定性 |
| 回退模型 | 主力限流、风格不匹配、中文任务偏多 | 可得性、切换成本、兼容性 |

### 3. 按家族看主流代码模型

#### Claude 家族

- 优势：复杂工程理解、长链路推理、对代码和文档一起处理的稳定性高
- 适合：高质量主力、关键任务、复杂仓库理解
- 风险：成本通常不适合拿来跑大量低价值任务

#### GPT / Codex 家族

- 优势：编码代理生态强、工具调用成熟、CLI 与云端委托协同明显
- 适合：Agentic Coding、复杂任务委托、标准化编码流程
- 风险：如果任务描述不清，仍可能在大任务里走偏

#### Gemini 家族

- 优势：长上下文、多模态、Google 生态协同
- 适合：代码 + 文档 + 截图 / 原型混合任务，或终端与云生态联动任务
- 风险：要结合你所在工具里的实际表现，不要只看 API 能力

#### Qwen 家族

- 优势：中文场景、国内生态、开源和企业接入路径都比较清晰
- 适合：中文开发协作、国内云环境、团队推广
- 风险：具体效果要看你使用的是开源版本、商业 API，还是嵌入某个工具的默认模型

#### DeepSeek 家族

- 优势：价格低、国内 API 获取方便、适合批量跑量
- 适合：低成本分析、初步整理、批量辅助任务
- 风险：不要把低成本模型直接当关键路径主力

#### 混元 / 豆包等国内企业模型路线

- 优势：组织级接入、中文、云生态和企业治理
- 适合：内部工具链、敏感环境、团队统一部署
- 风险：要重点看团队流程适配，而不是只看单次演示

### 4. 一张够用的代码任务评测表

你真正需要的不是“全网最全模型榜”，而是一张能复用的任务表：

| 任务类型 | 默认模型 | 回退模型 | 跑量模型 | 验证方式 |
| --- | --- | --- | --- | --- |
| 修小 bug |  |  |  |  |
| 跨文件功能 |  |  |  |  |
| 测试补齐 |  |  |  |  |
| PR 评审 |  |  |  |  |
| 中文注释 / 文档 |  |  |  |  |

填这张表时，至少记录：

- 一次通过率
- 返工量
- 平均等待时间
- 总成本

### 5. 切模型的正确触发条件

不要因为“新模型发布了”就全仓切换。更合理的触发条件是：

1. 返工率持续升高  
2. 成本结构明显不合理  
3. 新任务类型出现，原模型不适配  
4. 主力模型在你的工具链里不稳定  
5. 团队开始进入更严格的治理或合规环境

## Demo

### 同一个任务，设计三档模型方案

任务：给业务后台新增“批量导出 + 权限过滤 + 下载记录”功能。

**方案 A：高质量主力**

- 主力模型：Claude / GPT-Codex 一类高质量模型
- 用于：理解权限边界、设计改动范围、处理复杂测试

**方案 B：日常默认**

- 默认模型：均衡型模型
- 用于：中等复杂度编码、重构局部模块、日常补齐

**方案 C：低成本跑量**

- 跑量模型：DeepSeek / 低成本模型
- 用于：列出受影响文件、初步摘要、批量生成测试框架初稿

这 3 档一起用，往往比“全程只用一个模型”更稳。

## 作业

从你的真实工作里选 3 类任务：

1. 小改动  
2. 中等复杂度功能  
3. 复杂、跨文件、风险高的改动

为每类任务各填：

- 默认模型
- 回退模型
- 是否需要跑量模型
- 你打算怎么验证结果

## 常见误区

1. **把模型家族和工具产品混成一层**  
结果常常是工具没换，但你以为已经完成了“重新选型”。

2. **高质量模型全场硬用**  
这通常会把预算烧在不需要高质量的地方。

3. **低成本模型直接承担关键路径任务**  
便宜不等于适合，关键任务更看返工总成本。

4. **每次看到新版本就重配工作流**  
你会在模型切换上花掉比真正收益更多的时间。

## 验收标准

- 能解释至少 4 种模型角色
- 能用家族视角描述主流编程模型，而不是只背版本号
- 能为自己 3 类高频代码任务配出默认模型和回退模型
- 能说清什么情况下才值得切模型
