## 学习目标

- 理解 Cursor、Google（Gemini CLI / Code Assist / Antigravity）、Kiro 代表的 3 条产品路线
- 分清 `IDE Agent`、`Google 多入口 Agent`、`Spec-driven IDE` 的核心差异
- 知道 Rules、Background Agents、Specs、Steering、Hooks 这些词解决的是哪类问题
- 能判断自己更适合“编辑器原生协作”还是“规格先行”

## 适合谁

- 已经知道 AI 编程要进入 IDE 层，但不知道 Cursor、Gemini、Kiro 怎么区分的人
- 想给团队选一条从需求到实现更稳定的流程的人
- 既关心开发体验，也关心上下文规则和长期治理的人

## 前置条件

- 用过至少一种编辑器内 AI 工具
- 理解需求、实现、PR、Review 在开发链路里的位置
- 接受“不是所有团队都适合一上来就多 Agent + 多规则”

## 核心知识

### 1. Cursor：把 Agent 做进编辑器工作台

Cursor 这条路线的特点不是“有聊天框”，而是让编辑器成为 Agent 的主战场。

它的代表能力通常落在 4 个地方：

- **Rules**：项目级和用户级约束，解决“AI 总是忘记团队习惯”  
- **Background Agents**：异步任务与并行推进，解决“人一直盯着太慢”  
- **Bugbot / Review 能力**：把 AI 拉进代码评审和问题定位环节  
- **多模型路由**：在一个编辑器里切换不同模型

Cursor 的核心优势是：

> 对于“我一天的大多数时间都在 IDE 里”的开发者，它降低了工具切换成本。

### 2. Google：Gemini CLI 只是入口之一

这一课里把它简称为“Gemini 路线”，但更准确地说，它指的是 Google 围绕 Gemini 模型构建的整条开发产品线，而不只是 `Gemini CLI` 一个终端产品。

Google 这条路线当前至少覆盖：

- `Gemini CLI`：开源终端 Agent，是 Google 在命令行里的入口
- `Gemini Code Assist`：IDE 与企业开发协作能力
- `Antigravity`：更偏 agent-first 的开发平台，把编辑器、终端、浏览器和多 Agent 编排放到同一套工作面里

这条路线的特点是：

- 不是单一 CLI，而是终端、IDE、Agent 平台多入口并行
- Google 生态协同明显
- 多模态和长上下文能力适合复杂代码库与说明资料一起处理
- 对愿意自己拼装终端工作流，或把任务交给多 Agent 异步推进的人更友好

它适合的不是“最轻的补全场景”，而是：

- 你想保留终端主控权
- 或者你希望在 IDE、终端、Agent 平台之间自由切换
- 又希望接入搜索、文档、云环境等更大的生态能力

### 3. Kiro：支持 vibe coding，但主打规格驱动

Kiro 不是只能先写规格。它同样支持直接聊天、直接改代码、先做 vibe coding 式探索。

但 Kiro 最值得关注的地方，不是“它也能聊天写代码”，而是它把下面这件事做成了主叙事：

1. 先写 `Specs`
2. 用 `Steering` 让项目规则和设计意图长期生效
3. 用 `Hooks` 在关键节点做自动校验和后处理

所以更准确的说法是：

- Kiro 也能做快速探索、边聊边改、先跑出一个原型
- 只是它真正和别家拉开差异的地方，是把 `Specs + Steering + Hooks` 做成了更完整的工程流程

这条路线明显更适合：

- 多人协作
- 需求经常变复杂
- 团队已经意识到“先把规格写清楚，后面返工会更少”

它不是“只能规格驱动”，也不是“更强聊天”，而是更像把 AI 编程嵌进软件工程流程。

### 4. 三条路线的边界

| 维度 | Cursor | Google / Gemini | Kiro |
| --- | --- | --- | --- |
| 默认入口 | IDE | CLI + IDE + Agent 平台 | IDE |
| 核心价值 | 编辑器内高频协作 | 多入口协同、长上下文、多 Agent 与 Google 生态连接 | 既能 vibe coding，也能把规格驱动与流程治理做深 |
| 最适合 | 日常开发、Review、边看边改 | 终端主控、长上下文、多 Agent 编排与云协同 | 团队流程、需求到实现闭环，也适合先快速探索再收敛成规格 |
| 最大风险 | 过度依赖编辑器入口，流程抽象不够清晰 | 产品线入口较多，理解不同界面和边界需要时间 | 规格没写好时，前期成本会显得偏高 |

### 5. 什么时候不要强上 Spec-driven

很多人一看到 Kiro 的 `Specs` 思路就很兴奋，但这里有个判断边界：

如果你的任务是：

- 小 bug
- 一次性脚本
- 快速原型
- 临时小改动

那先用 Cursor、CLI Agent，或者直接在 Kiro 里走 vibe 模式，通常更高效。

Spec-driven 更适合：

- 需求要跨多人协作
- 有明确验收标准
- 容易在实现阶段跑偏
- 后续要持续维护

## Demo

### 同一个需求，Cursor 路线 vs Kiro 的两种用法

需求：给内部管理后台做一个“导入客户名单并去重”的功能。

**Cursor 路线**

1. 直接在 IDE 里让 Agent 读相关目录  
2. 先补理解，再改前后端文件  
3. 过程中实时看 diff、频繁修方向  
4. 用 Review / Bugbot 再做一轮收口

适合：

- 需求已经比较清楚
- 你想快速进实现
- 需要强交互感

**Kiro 的 vibe 用法**

1. 先直接描述需求，让它快速起草界面、接口和数据处理
2. 边看代码边改，先把可运行版本做出来
3. 确认方向没错后，再决定要不要补规格

适合：

- 你还在探索方案
- 需要先快速做出原型
- 暂时没有多人协作和严格验收压力

**Kiro 的 spec 用法**

1. 先把功能规格写清楚  
2. 明确输入、校验、去重规则、异常处理、验收标准  
3. 用 Steering 固定项目约束  
4. 再让 Agent 按规格推进代码  
5. 用 Hooks 补验证、生成文档或触发检查

适合：

- 功能后续要长期维护
- 团队成员多
- 容易在边界处理上返工

## 作业

找你当前最典型的 3 类开发任务，判断它更适合：

- Cursor 路线
- Google / Gemini 路线
- Kiro 路线

并补一列：

`如果这个任务再扩大 3 倍规模，我还会保持这个选择吗？`

这能帮助你判断你是在为“当前便捷”选工具，还是在为“长期流程”选工具。

## 常见误区

1. **把 IDE Agent 当成“更贵的代码补全”**  
真正的价值在上下文联动、规则、异步任务和 Review。

2. **把开源 CLI 误解成“功能会更弱”**  
开放终端入口的价值恰恰在于可拼装、可连接、可纳入你的既有工程工具链。

3. **把 Kiro 误解成“只能先写 Spec”**  
它也支持 vibe coding；只是当任务开始涉及多人协作、验收、回归和长期维护时，Spec-driven 才是它更有优势的用法。

4. **把 Rules / Steering 写成空话**  
像“写得优雅一点”“尽量考虑性能”这类规则几乎没有执行价值。
