## 适合谁

- 已经有一个明确的 2-4 周目标，但推进总是断断续续的人
- 在做个人产品、课程、内容项目或小工具 MVP，需要每天往前推一点的人
- 愿意让 OpenClaw 先推进低风险任务，再由自己做关键判断的人

## 不适合谁

- 目标还没定清楚，只想让 AI “自己想该做什么”的人
- 期待 OpenClaw 在你不确认的情况下自动改核心代码、接支付、发正式版本的人
- 当前项目没有固定节奏，也没有意愿每周回顾一次系统输出的人

## 前置条件

- OpenClaw 已部署运行，并且已经能把结果推到飞书
- 你已经有一个明确目标，例如“14 天内做出可演示 MVP”或“本周把课程 landing page 上线”
- 你能接受把任务边界写死：哪些可以自动推进，哪些必须人工确认
- 最好已经有一个项目文件夹，用来存放 `PROJECT.md`、文案草稿和需求清单

---

## 先看结论

这节课不是“AI 自动替你创业”，而是做一件更现实的事：

- 你定义目标
- OpenClaw 每天推进一部分低风险高价值工作
- 它把结果和状态沉淀下来
- 你只处理真正需要判断的部分

最适合自动推进的任务类型，通常只有 5 类：

1. 资料整理
2. 需求草稿
3. 页面文案
4. 用户反馈归类
5. 下一步建议

如果你把高风险动作也默认放开，这节课大概率会出问题。

---

## 主工具路径

| 环节 | 工具 | 作用 |
|------|------|------|
| 目标定义 | 你 + `PROJECT.md` | 写清目标、周期、成功标准和不做什么 |
| 每日任务生成 | OpenClaw Cron | 每天早上生成 1-3 个可执行的小任务 |
| 低风险执行 | OpenClaw | 整理资料、更新文档、产出草稿、汇总反馈 |
| 状态沉淀 | `PROJECT.md` / daily notes | 记录已完成、阻塞、待确认决策 |
| 结果通知 | 飞书 | 每天只推送最关键的产出和下一步建议 |

## Step 1：先写目标卡，不要直接开跑

最容易失败的情况，是只写一句：

```text
帮我推进我的产品。
```

更可执行的写法是：

```markdown
# 本阶段目标卡

## 目标
14 天内做出一个可演示的 AI 小工具 MVP

## 成功标准
- 有一个可访问页面
- 能让 3 个真实用户试用
- 我能拿到 3 条有效反馈

## 本阶段不做
- 不接支付
- 不做复杂账号体系
- 不做移动端原生 App

## 我的时间上限
工作日每天最多投入 90 分钟

## 当前约束
- 只用 Trae 或 Claude Code
- 优先做静态前端验证版
- 如果要改核心逻辑，必须先提醒我确认
```

目标卡越具体，OpenClaw 越不会每天都重新猜你的方向。

---

## Step 2：把允许自动推进的任务类型写死

不要让系统默认“什么都可以做”。第一版只开放下面这些任务：

| 可自动推进 | 说明 |
|------------|------|
| 研究资料整理 | 整理竞品、用户反馈、参考案例 |
| 需求草稿 | 把零散想法收敛成 PRD、页面结构、验收清单 |
| 文案初稿 | landing page、介绍文案、FAQ、测试脚本 |
| 反馈归类 | 试用反馈、评论、飞书消息、问卷整理 |
| 优先级建议 | 产出下一步建议、风险提醒、待确认事项 |

默认不要放开的任务：

| 默认禁止 | 原因 |
|----------|------|
| 自动部署正式版本 | 风险高，容易误操作 |
| 自动接支付 / 改回调 | 涉及资金和稳定性 |
| 改数据库或核心业务逻辑 | 回滚成本高 |
| 无确认地批量改代码 | 很容易把项目带偏 |

可以直接写进 `AGENTS.md`：

```markdown
## MVP 推进任务边界

允许自动推进：
- 资料整理
- PRD / 页面结构草稿
- 文案初稿
- 用户反馈归类
- 优先级建议

必须先询问我确认：
- 改核心代码
- 正式部署
- 接支付
- 修改数据库结构
- 删除已有文件
```

---

## Step 3：建立一个最小状态板

不需要先做 Kanban 系统。第一版一个 `PROJECT.md` 就够了。

建议直接放在项目根目录，和 `README.md` 同级，例如：

```text
your-project/
├── README.md
├── PROJECT.md
├── docs/
└── src/
```

模板：

```markdown
# 项目状态

## 当前目标
[一句话]

## 已完成
- [事项]

## 当前阻塞
- [事项]

## 待确认决策
- [事项]

## 下一步建议
- [事项]
```

OpenClaw 每天推进任务后，都只更新这一个文件。  
这样一周后你不会得到很多零散消息，而是一个持续变清晰的项目状态。

---

## Step 4：配置每日推进任务

建议每天只做 1-3 个任务。  
任务越多，噪音越大，真正会做完的越少。

示例：

```json
{
  "cron": [
    {
      "pattern": "0 9 * * 1-5",
      "prompt": "读取 PROJECT.md 和 AGENTS.md。先检查是否还有高优先级未完成事项；如果有，今天优先继续推进这些事项，只更新状态，不新增任务。只有在高优先级事项已收敛时，才围绕当前 MVP 目标生成并推进 1-3 个低风险任务，优先做资料整理、需求草稿、落地页文案、反馈归类和下一步建议。完成后更新 PROJECT.md，并把今日推进摘要发到飞书。",
      "timezone": "Asia/Shanghai"
    }
  ]
}
```

飞书摘要建议固定成 3 段：

```markdown
## 今日推进
- 今天完成了什么

## 当前阻塞
- 卡在哪里

## 你只需要决定的事
- 今天需要你拍板的 1-2 个问题
```

这样你打开飞书时，不会先看到一大堆过程，只会看到最该处理的事。

---

## Step 5：把任务样例具体化

如果你做的是个人产品 MVP，OpenClaw 适合推进的任务可以长这样：

### 任务样例 A：整理用户反馈

```text
请整理最近 20 条试用反馈，按“核心抱怨 / 可忽略建议 / 真正会阻碍付费的点”分类，
并更新到 PROJECT.md。
```

### 任务样例 B：输出 landing page 文案草稿

```text
基于当前目标用户和问题简报，输出一版首屏文案、功能卖点和 FAQ 草稿。
要求口语化，不要 AI 套话。
```

### 任务样例 C：生成下一步优先级建议

```text
根据当前项目状态，给出今天最值得推进的 3 件事。
每件事都要说明为什么排在前面，以及暂时不做什么。
```

### 任务样例 D：补一页需求草稿

```text
把目前零散的功能想法整理成一页 PRD：
- 目标用户
- 核心场景
- 第一版功能
- 不做什么
- 验收标准
```

这些任务都很具体，也都符合“低风险高价值”的边界。

---

## Step 6：每周停下来做一次回顾

如果你只会加任务，不会删任务，这套系统很快就会空转。

每周至少回顾一次：

- 哪些自动任务确实有用
- 哪些只是把噪音变得更系统
- 哪些地方总在重复提醒，说明边界没写清
- 哪些任务应该改成“必须确认后才能执行”

建议保留 3 类有效输出：

- 真正帮你节省时间的
- 真正让项目更清晰的
- 真正把下一步变简单的

其他都可以砍掉。

---

## 一周验收标准

这节课一周后的结果，不应该是“AI 很忙”，而应该是：

- 有一份持续更新的 `PROJECT.md`
- 有一版更清晰的 landing page 或需求文档
- 有一份反馈归类或优先级建议
- 你知道下周最该推进哪 3 件事

如果一周后你只是多了很多提醒和待办，这套配置就还没调好。

---

## 常见问题

**问题 1：每天都在推很多任务，但我根本看不过来**

解决：把每日任务上限压到 1-3 个。  
OpenClaw 的目标不是让你更忙，而是减少你每天重新思考的成本。

**问题 2：它总想动代码，我其实只想先做研究和文案**

解决：把任务边界写得更硬。  
不要写“尽量不要改代码”，要写“改核心代码必须先确认”。

**问题 3：有时它给的建议不错，有时又很空**

通常说明 `PROJECT.md` 里的目标和当前状态太模糊。  
先把成功标准、当前阻塞和不做什么补清楚，再看输出质量。

---

## 这节课真正解决的事

- 不是让 OpenClaw 替你完成整个项目
- 不是把所有动作都自动化
- 而是围绕一个明确目标，让项目每天稳定往前推一小步
