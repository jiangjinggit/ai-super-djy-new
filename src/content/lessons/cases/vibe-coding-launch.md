## 适合谁

- 有产品想法但没有开发背景，或有基础但以前没有独立交付过完整产品的人
- 想用 AI 编程工具验证一个小想法的运营、产品、设计、咨询等非技术岗
- 已完成 AI 编程模块，想做第一个真实产品的人

## 前置条件

- 有一个明确的产品想法（哪怕很粗糙）
- 能用 Trae 或 Claude Code 写代码
- 会用命令行的基本操作（cd、ls、运行 npm install 之类）
- 对"第一版能用就行，不追求完美"有心理准备

---

## 先看结论

这节课你只需要记住 3 句话：

1. 先把产品范围写小，再开始生成代码。
2. 先做出能访问的 MVP，再决定要不要接支付。
3. 如果你做的是个人网站，可以直接走更短路径，不必把整套产品流程全走一遍。

> 本课解决的是“首版验证”。
> 不解决正式商用部署、备案、支付回调、生产监控这些后续问题。

---

## 完整操作步骤

### Step 1：写产品描述文档（30 分钟）

在开始写代码之前，把想法约束成一张文档。这是最能避免浪费时间的一步。

**模板**：

```
产品名称：[一个简短的名字]

解决的问题：
[谁] 在 [什么情况下] 遇到 [什么具体问题]
（例：内容创作者每周写文章，在选题时不知道什么话题受欢迎）

目标用户：
[具体描述一个典型用户]（不要写"所有人"）

第一版只做这些功能：
1. [功能 1]
2. [功能 2]（最多 3 个，再多就删掉）

第一版不做：
- [明确排除的功能]（写这个很重要，防止功能蔓延）

成功标准：
[具体描述如何判断第一版做完了，例：用户能完成一次完整的 XX 流程]
```

**检验产品描述好不好的标准**：
把这个文档发给一个不了解你想法的朋友，他能不能理解你要做什么。如果不能，说明还需要再具体。

**写完后自查 3 个问题**：
- 用户是不是足够具体，而不是“所有人”
- 第一版功能是不是控制在 3 个以内
- 成功标准是不是能一句话判断“做完了没有”

---

### Step 2：让 AI 先帮你挑错

把产品描述文档贴给 Claude 或 Trae，用这个 Prompt：

```
你是我的目标用户：[具体描述用户]。

我要做这个产品：[粘贴你的产品描述文档]

请扮演这个用户，提出 5 个你会有的疑问或疑虑。
不要给我正面鼓励，我需要真实的挑战和潜在的问题。
```

先把这些问题改清楚，再开始写代码。越早改，越省时间。

> 不要把这一步当形式。
> 很多人不是代码不会写，而是产品问题问得太晚。

---

### Step 3：选工具

这节课只讲代码路线，不再混入低代码工具。

| 工具 | 更适合什么情况 | 你的感受会是什么 |
|------|------|------|
| Trae | 第一次做 MVP、要频繁改页面和交互 | 更直观，改起来更顺手 |
| Claude Code | 任务明确、要批量改文件、调试更深入 | 更直接，更像工程协作 |

**最简单的选择方法**：
- 你想一边看页面、一边快速改交互和样式：选 `Trae`
- 你已经知道要改哪些文件、哪些命令、哪些报错：选 `Claude Code`

**模型选择原则**：
- 默认先用工具的自动选择或主推荐模型，不要一开始纠结版本号
- 复杂规划、架构拆解、棘手报错：用更强模型
- 重复改文案、批量生成、低风险改动：用便宜模型

第一次做产品，建议默认用 Trae 起步，Claude Code 作为进阶补充。

---

### Step 3.5：如果你做的是个人网站，直接走这条最短路径

如果你的目标不是做复杂产品，而是做一个**个人品牌页 / 作品集 / 服务页 / 简历站**，直接走这条更短路径：

| 步骤 | 你要做什么 | 重点检查 |
|------|------|------|
| 1 | 先用 Stitch 出首页原型 | 不要直接跳代码 |
| 2 | 看 4 件事：首屏、信息顺序、手机端、辨识度 | 页面像不像你自己 |
| 3 | 把原型交给 AI Studio 或直接交给 Trae 出代码 | 技术栈写死：React + Tailwind |
| 4 | 自己做一轮收口 | 文案、按钮、图片、SEO、手机端 |
| 5 | 再按 Step 5 部署 | 先能访问，再谈长期方案 |

如果你访问 Google 工具有困难，也可以跳过 Stitch / AI Studio，直接把原型要求写清楚后交给 Trae：

```text
帮我做一个个人网站首页，面向想了解我个人品牌和 AI 能力的访客。

要求：
- 风格：科技感强、深色、高级、克制
- 结构：首屏标题、个人介绍、核心能力模块、案例展示、联系方式、CTA
- 技术：React + Tailwind CSS
- 适配：桌面端和移动端
- 要求代码结构清晰，方便后续修改和部署
```

这个案例最适合第一次体验“AI 先做 80%，你自己收最后 20%”。

---

### Step 4：开始构建，控制范围

原则很简单：每次只做一个可以独立运行的功能，做完验证后再加下一个。

**推荐节奏**：
- 第 1 轮：先做一个最小页面或最小主流程
- 第 2 轮：修掉关键报错，让它稳定可运行
- 第 3 轮：再补样式、按钮、文案这些细节

**第一条消息怎么写**：

```
我要做一个 [产品名称]，第一步只需要：[单一功能描述]

技术要求：
- 前端：React（或 HTML+CSS，越简单越好）
- 数据存储：本地浏览器 localStorage（第一版不需要数据库）
- 不需要用户登录（第一版）

请帮我生成这个最简单的版本，能运行就行，暂时不用考虑样式好看。
```

**每次迭代的节奏**：
1. 描述一个功能 → AI 生成代码
2. 运行测试，看是否符合预期
3. 有问题：复制完整错误信息给 AI，让它分析修复
4. 符合预期：提交一个 git commit，再做下一个功能

**遇到错误的正确处理方式**：

```
我运行了你的代码，出现了以下错误：
[复制完整错误信息，包括文件名和行号]

我做了什么操作触发了这个错误：[描述操作步骤]

请分析错误原因并给出修复方案。
```

### Step 5：部署上线

这一节只解决“让用户能访问第一版”，不是“正式商用上线”。

| 路径 | 适合什么 | 推荐程度 |
|------|------|------|
| 静态前端 MVP 验证版 | 作品展示页、工具页、单页应用 | 第一优先 |
| 带简单后端的验证版 | 表单提交、简单 API、轻量数据处理 | 第二步再考虑 |

**路径 A：静态前端 MVP 验证版**

最快方案是 `Vercel`：
```bash
# 安装 Vercel CLI
npm install -g vercel

# 在项目目录里执行
vercel

# 按提示操作，几分钟就能拿到一个 https://xxx.vercel.app 地址
```

注意：Vercel 适合快速验证，但**不等于中国大陆稳定访问方案**。如果你的用户主要在中国大陆，`.vercel.app` 可能变慢甚至不可达。更稳的正式路线通常是：
- 阿里云 OSS + CDN
- 腾讯云 COS + CDN
- 备案后的国内静态托管/容器部署

**路径 B：带简单后端的验证版**

适合：表单提交、简单 API、轻量数据处理。

国内可选腾讯云函数/云开发，但它不是“所有全栈项目一键丢上去就行”。至少要准备入口文件、环境变量和部署配置。

例如用 Serverless Framework 部署时：
```bash
# 安装 Serverless 工具
npm install -g serverless

# 腾讯云登录
serverless login

# 部署
serverless deploy
```

如果你只是做海外或通用验证，`Railway` 会更省事：
1. railway.app → New Project → Deploy from GitHub
2. 连接你的 GitHub 仓库
3. 配置环境变量
4. 自动部署，一键搞定

但无论腾讯云函数还是 Railway，这一课都不覆盖：
- 数据库设计
- 多环境配置
- 监控与告警
- 正式商用的安全和容量规划

### Step 6：收款方案（上线前必须想清楚）

你不一定要一开始就接支付，但必须先知道未来走哪条路。

| 路线 | 适合谁 | 典型方案 | 关键判断 |
|------|------|------|------|
| 个人验证版 | 第一次做 MVP、先验证有没有人要 | 暂不接支付 / 手动收款 / 预约名单 | 先验证价值，不先啃支付 |
| 内容订阅版 | 知识产品、专栏、newsletter、轻服务 | 爱发电 / 小报童 | 平台抽成会变，优势是开通快 |
| 公司主体版 | 正式商业产品、要做站内支付闭环 | 微信支付 / 支付宝 | 需要营业执照、商户申请、回调开发 |
| 海外版 | 面向海外用户、英文市场 | Lemon Squeezy / Stripe | 看主体所在地、支持国家和税务处理 |

**最常用的判断方法**：
- 第一次做 MVP：先走 `个人验证版`
- 你卖的是内容或订阅：先看 `内容订阅版`
- 你已经有公司主体，准备做正式闭环：再看 `公司主体版`
- 你的用户主要在海外：直接按 `海外版` 设计

**推荐顺序**：
- 第一个产品：优先验证，不急着接正式支付
- 如果是内容订阅：先用平台型方案
- 如果产品开始稳定成交：再考虑公司主体版
- 如果目标用户主要在海外：优先按海外路线设计域名、支付和条款

费率、支持国家、结算规则经常变化，这里只做路线判断，精确条件以官方页面为准。

## 关键配置模板

### 项目规范文件（Claude Code / Trae 都能用）

如果你用 Claude Code，可以在项目根目录创建 `CLAUDE.md`；如果你用 Trae，也可以把同样内容写进项目规则或工作区规则。核心不是文件名，而是先把边界写清楚。

```markdown
# 项目说明

## 产品描述
[粘贴你的产品描述文档]

## 技术栈
- 前端：React + TypeScript
- 后端：[如果有的话]
- 数据库：[如果有的话]

## 开发规范
- 每次只改一个功能，不要同时改多个地方
- 改动前先告诉我你要做什么，再动手
- 有不确定的地方先问我，不要猜测我的意图

## 不要做
- 不要添加我没有要求的功能
- 不要重构已经运行正常的代码
- 不要在没有我确认的情况下删除文件
```

**Trae 用户最少要补 3 条规则**：
- 每次只实现一个功能，先跑通再继续
- 发生报错时优先解释原因，不要直接大改
- 不要主动引入数据库、登录、后台管理等未要求能力

你不用一开始把规则写很长，先把这 3 条写进去就够了。

## 常见问题

**问题 1：AI 生成的代码运行不起来，不知道怎么办**

把完整的错误信息复制给 AI（Terminal 里的红色文字），不要只说"运行不了"。

**问题 2：做了一大堆，但用户反馈完全不是想要的**

原因：太晚让真实用户看到。第一版主流程做好后就找 3-5 个人测试，不要等“完善”了再发。

**问题 3：部署后产品很慢，国内用户访问几乎打不开**

如果你现在只是验证产品方向，先接受“海外部署预览可访问但不稳定”；如果已经确定做中国大陆用户，就不要继续拖，直接切到国内托管或双部署。

**问题 4：做完了没人用**

这不是工具的问题。分发渠道和产品方向选择，比代码本身更重要。先想清楚“谁会为此付钱/使用它”，再开始写代码。
