## 这节课你要拿到什么

- 一套 Claude Code 5 步执行法
- 一份可直接复制的任务描述模板
- 一张“不做什么”清单
- 一个 CLI 与 Cowork 都能复用的任务收口逻辑

## 为什么需要固定流程

很多返工不是因为 Claude Code 不会做，而是因为你一开始没把范围说清。

最典型的坏启动方式是：

`这个模块有点乱，帮我整理一下。`

它太宽了，模型只能自己猜。

Cowork 也一样。你如果只说“帮我整理一下这些资料”，它很可能会给你一份看起来完整、但边界模糊的总结。

## 5 步执行法

1. 先读
2. 出计划
3. 执行
4. 验证
5. 总结

这不是形式主义，而是让任务边界逐步收紧。

核心思路只有一句话：

`先让它理解，再让它承诺，最后才让它动手。`

## 每一步都该怎么说

### 第 1 步：先读

让它先阅读相关文件并总结现状，不要急着改。

更好的说法是：

`先阅读 X、Y、Z 这几个文件，总结现状、风险点和你认为最可能的原因，先不要修改。`

### 第 2 步：出计划

让它列出：

- 要改哪些文件
- 为什么要改
- 哪些文件不会碰

这一轮最重要的不是“计划多完整”，而是它有没有主动缩小边界。

### 第 3 步：执行

在你确认计划后再开始改。

如果任务偏大，执行前再加一句：

`严格按上面的计划做，不要额外优化无关部分。`

### 第 4 步：验证

至少做一种验证：

- 测试
- 编译
- 类型检查
- 明确的人肉检查

没有验证的改动，不算完成。

### 第 5 步：总结

要求它说清：

- 改了什么
- 为什么这么改
- 还有什么风险

总结的价值不只是“汇报”，而是让你下次可以把这次任务沉淀成模板。

## 一个好任务 brief 应该长什么样

一个稳定的 brief 至少要回答 6 个问题：

| 问题 | 你要写什么 |
|------|-----------|
| 目标 | 要解决什么问题 |
| 范围 | 允许碰哪些文件 / 资料 |
| 上下文 | 相关背景、报错、业务规则 |
| 约束 | 不做什么、不能越过什么边界 |
| 验收 | 怎么算完成 |
| 输出 | 你希望最后看到什么 |

## 可复制模板

```text
目标：
[一句话说清要解决什么]

范围：
[允许改哪些文件 / 不允许碰哪些文件]

上下文：
[相关模块、报错、背景]

执行要求：
1. 先阅读相关文件并总结
2. 给出简短计划
3. 确认后再实施修改
4. 修改后运行验证
5. 输出变更说明和风险

不做什么：
[不重构 / 不引依赖 / 不改接口 / 不改无关文件]

验收标准：
[怎样算完成]
```

## CLI 与 Cowork 的 brief 差别

| 维度 | Claude Code CLI | Cowork |
|------|-----------------|--------|
| 输入对象 | 仓库、代码、命令、文档 | 文件夹、资料、连接器、浏览器内容 |
| 最该强调 | 范围、禁止事项、验证命令 | 资料来源、输出格式、引用边界 |
| 最常见风险 | 顺手多改文件 | 把猜测写成结论 |

所以你写给 CLI 的 brief，更像“工程任务单”；写给 Cowork 的 brief，更像“研究与交付任务单”。

## “不做什么”为什么这么重要

因为模型很容易“顺便优化”。

常见禁止项：

- 不重构整个模块
- 不引入新依赖
- 不改接口协议
- 不改代码风格

如果是 Cowork，可以把“不做什么”写成：

- 不引用未读取的资料
- 不把猜测写成事实
- 不擅自删掉原始材料
- 不输出成泛泛总结，要按指定格式交付

## 跟着做：把一个模糊需求改写成可执行 brief（⏱ 10 分钟）

模糊版：

`帮我处理一下这个模块。`

改写版：

```text
目标：
修复课程页面切换主题后按钮颜色不一致的问题。

范围：
允许改 src/components/lesson-page 和 src/pages/LessonPage.tsx。
不允许改内容数据和无关样式系统。

上下文：
问题只在深色模式切换后出现，浅色模式正常。

执行要求：
1. 先读相关文件并总结问题来源
2. 给出简短修复计划
3. 确认后再实施修改
4. 修改后说明如何验证
5. 总结改动和残留风险

不做什么：
不重构整个主题系统
不引入新依赖
不修改其他页面样式

验收标准：
课程页面在亮色和深色模式下按钮都显示正确颜色。
```

## 📋 案例：修一个跨文件 Bug

| 步骤 | 用时 |
|------|------|
| 写任务描述 | 5 分钟 |
| 先读文件 | 2 分钟 |
| 看计划 | 1 分钟 |
| 执行修改 | 3 分钟 |
| 验证结果 | 3 分钟 |
| 输出总结 | 1 分钟 |

重点不是速度，而是减少返工。

对比：如果你直接说“帮我修掉这个主题问题”，Claude 很可能会把主题系统整段翻出来一起改，最后你多了 5 个无关变更。

## ✅ 本课落地动作

- 用模板写 1 个真实任务
- 按 5 步流程跑一遍
- 记录你最想补强的一步：读、计划、验证，还是总结
- 再把同一个任务改写成一版适合 Cowork 的 brief，对比两者差异
