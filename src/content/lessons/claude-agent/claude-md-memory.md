## 两套上下文系统分别是什么

可以把它理解成 Claude Code 的员工手册。

它不是某一次任务的 prompt，而是默认上下文。

在 CLI 里，这份手册主要是 `CLAUDE.md`。

在 Cowork 里，类似角色由这些东西承担：

- Project
- Context files
- Global Instructions

## CLAUDE.md 适合写什么

- 项目背景
- 常用命令
- 架构说明
- 开发规范
- 禁止事项

## Cowork 的上下文更适合写什么

- 默认输出格式
- 你的工作角色
- 常用资料放在哪
- 删除前先确认这类全局偏好
- 会持续变化的资料文件

## 标准结构

### 1. 项目背景

一句话说明这个仓库是干什么的。

### 2. 常用命令

把你最常跑的命令直接写进去，别让它自己猜。

### 3. 目录与架构

说明关键目录和重要文件的作用。

### 4. 规范

例如：

- 命名习惯
- 测试要求
- 提交要求
- 注释风格

### 5. 禁止事项

这是最关键的一段之一。越具体越好。

### 6. 当前上下文

适合写当前阶段的重要目标，但要记得定期更新。

## 三类项目模板思路

| 项目类型 | 更应该强调什么 |
|---------|---------------|
| 代码项目 | 命令、架构、禁止事项 |
| 文档项目 | 语气、结构、引用规则 |
| 个人工作流 | 固定产物格式、步骤和边界 |

## 分层加载怎么理解

一般可以理解为：

- 全局 CLAUDE.md：通用习惯
- 项目 CLAUDE.md：仓库规则
- 子目录 CLAUDE.md：局部差异

如果放到 Cowork 里，可以对应理解为：

- Global Instructions：通用工作偏好
- Project Instructions：当前工作空间规则
- 文件夹里的资料与上下文文件：动态内容

越靠近当前目录，优先级通常越高。

## 跟着做：写第一版 CLAUDE.md（⏱ 15 分钟）

今天先写最小版：

- 项目一句话介绍
- 3 条常用命令
- 3 条禁止事项
- 关键目录说明

不要追求全，一周后再补。

## 常见翻车

### 翻车 1：把任务说明全塞进 CLAUDE.md

它会越来越长，而且很多内容过几天就过时。

在 Cowork 里也是一样：会变化的内容，应该放回文件或 Project 资料里，而不是把 Instructions 写成流水账。

### 翻车 2：只写愿景，不写禁区

“保持高质量”没用，“不要改接口协议”才有约束力。

### 翻车 3：写完就不管

CLAUDE.md 是活文档，不是一次性材料。
