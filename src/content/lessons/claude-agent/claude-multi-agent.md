## 先看单工具的边界

Claude Code CLI 很适合在当前项目里深度推进任务，Cowork 很适合围绕文件和资料空间工作，但它们都不天然负责全部流程：

- 定时触发
- 后台常驻
- 多渠道通知
- 大规模并行拆分

这就是为什么复杂任务会走向组合式协作。

## 三种常见组合

### Cowork + Claude Code CLI

适合：

- Cowork 先整理资料和成品草稿
- CLI 再处理仓库、脚本和自动化修订

### Claude 工具 + OpenClaw

适合：

- OpenClaw 收集和触发
- Claude 系工具处理本地项目或文件空间

### 主智能体 + 子智能体

适合：

- 把任务拆成更小的专门工作单元
- 让不同助手各自做擅长部分

## 3 个可复现场景

| 场景 | 分工 |
|------|------|
| 自动化代码质量检查 | CLI 改，子助手或人工复查 |
| 每日工作摘要 | OpenClaw 触发，Cowork 汇总资料，CLI 补脚本输出 |
| 文档自动同步 | Cowork 生成说明草稿，CLI 回写仓库文档 |

## 跟着做：先做手动版（⏱ 20 分钟）

不要一开始就全自动。

先做一个最小手动组合：

1. Cowork 或 OpenClaw 先把输入整理到共享位置
2. CLI 读取并处理需要进仓库的部分
3. 结果再回写或推送

手动跑顺之后，再谈自动触发。

## 新手的 5 阶段路径

1. 单独用好 CLI 或 Cowork 其中一条
2. 写好 CLAUDE.md 或 Cowork Instructions
3. 建立 slash commands / Project 工作习惯
4. 再接外部触发或共享目录
5. 最后再引入 subagents 或并行拆分

不要跳级。

## 常见翻车

### 翻车 1：单工具都没用稳，就想多智能体

复杂度会成倍上升。

### 翻车 2：分工不清

每个工具都想做同一件事，最后谁都做不好。

### 翻车 3：没有共享数据边界

结果文件、日志和触发条件一乱，排查非常痛苦。

## ✅ 本课落地动作
