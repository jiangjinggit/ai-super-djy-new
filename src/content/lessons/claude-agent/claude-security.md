## Claude 工作台的“危险”能力

Claude Code CLI 和 Cowork 强大的地方，往往也是风险最大的地方。

比如它可能会：

- 删除文件
- 安装依赖
- 执行 shell 命令
- 修改配置
- 对外发请求
- 重组你的本地文件夹和成品文档

所以安全不是附加项，而是默认项。

## 必须人工确认的 5 类操作

1. 推代码
2. 删除非临时文件
3. 安装新依赖
4. 修改密钥或环境配置
5. 向外部服务发送数据

如果你没有明确写出来，它就容易在任务推进中越界。

## CLI 和 Cowork 的边界要分开写

### 在 CLAUDE.md 里怎么写

写法要具体，比如：

- 不允许执行 `git push`
- 不允许安装新包
- 不允许修改 `.env`
- 遇到高风险操作必须停下来展示计划

抽象表述往往没有约束力。

### 在 Cowork 里怎么写

更适合写进 Global Instructions 或 Folder Instructions 的是：

- 不允许删除或归档原始文件，除非我确认
- 不允许把猜测写成事实
- 不允许把敏感资料发往外部服务
- 整理资料前先列出将要操作的文件范围

CLI 的风险更多是“改仓库和跑命令”，Cowork 的风险更多是“动本地文件和生成成品”。

## 任务前后检查习惯

### 任务前

- 看清范围
- 做一次备份
- 明确禁止事项

### 任务后

- 先看 `git diff`
- 再看 `git status`
- 最后再决定是否提交

如果是 Cowork，再加两步：

- 检查它到底动了哪些本地文件
- 检查最终成品里有没有把猜测写成结论

## 意外操作恢复思路

真正稳定的流程不是“永不出错”，而是“出了错能恢复”。

所以你至少要熟悉：

- 回滚单文件
- 暂存当前改动
- 放弃不想要的修改
- 恢复本地文件夹整理前的备份或快照

具体命令细节可以按你团队的 Git 规范执行。

## Cowork 特有的 3 条边界

### 1. 不要把它当受监管工作台

如果你的任务涉及强合规、高敏感或审计要求，Cowork 目前不该是默认入口。

### 2. 长任务依赖桌面端环境

它不是“提交了就丢到云端跑完”的模式。桌面端关闭、电脑睡眠，都会影响任务继续执行。

### 3. 定时任务也受本机状态影响

定时任务只在桌面端开启、电脑保持唤醒时更可靠，所以不要把它当成云端 cron 来理解。

## 常见翻车

### 翻车 1：为了省事，把所有操作都放开

短期看方便，长期看就是埋雷。

### 翻车 2：不看 diff 直接提交

这样最容易把无关改动带进来。

### 翻车 3：出了问题才补权限规则

最好在第一次正式用之前就写好。

### 翻车 4：把 Cowork 当成普通聊天窗口

它会真的读写本地文件，所以你必须像对待执行工具一样提前设边界。

## ✅ 本课落地动作

- 在 CLAUDE.md 或 Cowork Instructions 中补上权限边界
- 亲自测试 1 次危险操作是否会停下来
- 建立“CLI 任务后先 diff，Cowork 任务后先查文件变更”的固定习惯
