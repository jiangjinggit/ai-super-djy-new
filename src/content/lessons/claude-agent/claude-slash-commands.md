## 先区分三个概念

| 能力 | 作用 |
|------|------|
| Tools | 让模型具备实际执行能力 |
| Slash Commands | 把高频任务包装成可复用入口 |
| Subagents | 把任务拆给更专门的助手处理 |

很多人把这三件事混在一起，结果越配越乱。

再加一层理解：

- Tools 解决“能不能做”
- Slash Commands 解决“怎么稳定开始”
- Subagents 解决“怎么拆得更清”

## 为什么 slash commands 值得做

因为很多任务每次都差不多：

- 生成 commit 信息
- 做代码 review
- 输出今日总结
- 先分析问题再给调试建议

既然结构重复，就应该把入口标准化。

本质上，slash commands 不是“快捷键”，而是把高频 prompt 固化成团队可复用接口。

## 5 个高频命令范例

- `/commit`
- `/review`
- `/doc`
- `/daily`
- `/debug`

它们有一个共同点：

输入来源和输出格式都比较稳定。

## 什么任务适合做成 slash command

| 适合 | 不适合 |
|------|--------|
| 输入来源固定 | 每次上下文都完全不同 |
| 输出格式固定 | 需要大量临场判断 |
| 一周会重复多次 | 一个季度才做一次 |
| 你已经手动跑顺过 | 你自己都还没做清楚 |

所以，先手动跑顺，再做成命令，通常比一开始就命令化更稳。

## 一个命令最少要包含什么

建议至少有这 4 段：

1. 读什么输入
2. 产出什么结果
3. 不做什么
4. 输出语言和格式

如果少了第 3 条，命令往往会越跑越宽。

## 跟着做：写一个 /standup（⏱ 15 分钟）

目标：

让它根据当天改动和任务记录，输出一份简短站会汇报。

至少要写清：

- 读哪些内容
- 输出几段
- 用什么语气
- 不做什么

最简结构示意：

```text
/standup

目标：
根据今天的改动和任务记录，生成一份简短站会汇报。

输入：
- git log / git diff
- 今日任务清单

输出：
- 昨日完成
- 今日计划
- 风险与阻塞

不做什么：
- 不修改任何文件
- 不虚构未完成事项
```

## 设计原则

1. 一个命令只做一件事
2. 先写数据来源，再写输出格式
3. 明确不做什么
4. 保持语言和风格一致

再补两条更实用的：

5. 命令名要短，但含义要稳定
6. 先自己用 3 次，再考虑给别人用

## CLI 与 Cowork 在“命令化”上的差别

CLI 更适合把命令做成工程动作，比如：

- `/review`
- `/commit`
- `/debug`

Cowork 更适合把命令思维做成工作套路，比如：

- 统一资料汇总格式
- 固定周报结构
- 固定研究摘要模板

也就是说，Cowork 不一定靠 `/命令名` 触发，但同样可以用“固定入口模板”来降低波动。

## 常见翻车

### 翻车 1：一个命令塞太多能力

命令越杂，触发后越不可控。

### 翻车 2：只写目标，不写输入来源

模型会自己猜，结果很容易漂。

### 翻车 3：忽略“不做什么”

比如你本来只想要建议，它却直接开始改文件。

### 翻车 4：还没稳定就开始堆命令

命令太多、命名太近、边界不清，最后你自己都不知道该用哪个。

## 📋 案例：把 review 习惯固化成一个命令

原始做法：

- 每次都临时写一段 review prompt
- 有时看暂存区，有时看文件，有时又忘了写输出格式

命令化之后：

```text
/review

读取当前变更或指定文件。
优先找 bug、回归风险、边界条件和缺失测试。
先给发现，再给简短总结。
不要直接修改代码。
```

收益不是“更酷”，而是输出风格稳定了，也更接近真正可复用的工作流。
