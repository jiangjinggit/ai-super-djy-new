## 学习目标

- 建立 Claude Code 不是聊天框，而是带工具代理的认知
- 理解聊天 AI vs Claude Code vs Cowork 的本质差异
- 知道 Claude Code 适合谁，以及自己应该从哪个入口开始

## 适合谁

所有想用 Claude Code 或 Cowork 的用户，无论技术背景。

## 前置条件

无。这是第一课。

## 核心知识

### 两种 AI 工具的本质差异

普通聊天 AI（Claude.ai 对话框、ChatGPT）做的是：
- 你说一句，它回一句
- 它没有"手"——无法读你的文件、改你的代码、跑命令
- 每次对话基本上都是从零开始

Claude Code 和 Cowork 做的是：
- 它能**读你机器上的真实文件**
- 它能**执行终端命令**（CLI）或**操作本地文件夹**（Cowork）
- 它会**记住这次任务的上下文**，多步骤连续推进
- 它可以**调用外部工具**（MCP 连接器、浏览器、数据库）

一句话概括：**聊天 AI 是"顾问"，Claude Code / Cowork 是能动手的"执行者"。**

### Claude Code 怎么工作的

它不是一个聊天框，而是一套"带工具的工作代理"。每次处理任务，本质上都在做四件事：

1. **读取当前上下文**：你的提示词、当前目录、`CLAUDE.md`、规则文件、可用工具
2. **判断接下来要不要调用工具**：读文件、搜代码、跑命令、调用 MCP、启动子 Agent
3. **在权限模式约束下执行动作**
4. **把结果写回当前会话**，并在必要时更新 memory

这就是为什么同样一句"帮我修这个 Bug"，在不同项目、不同权限模式、不同 `CLAUDE.md` 下，表现会差很多。

### 两套长期记忆机制

- `CLAUDE.md` / `.claude/rules/`：你写给它的长期指令
- Auto Memory：Claude 在多次协作后，自己总结出来的经验和偏好

官方把这两者都定义为"context"，不是强制执行配置。写得越具体、越短、越一致，遵循度越高。

### Claude Code 适合谁

| 人群 | 最适合的入口 | 核心价值 |
|------|------------|---------|
| 开发者 | Claude Code CLI | 代码审查、重构、自动化脚本、仓库维护 |
| 产品经理 | Cowork | 需求文档、竞品分析、原型说明、数据整理 |
| 运营/市场 | Cowork | 内容生产、数据报告、资料归类、多文件处理 |
| 研究人员 | Cowork + CLI | 文献整理、数据分析、报告生成 |
| 创业者/独立开发 | CLI + Cowork 组合 | 从想法到产品的全链路 AI 协作 |

## Demo

同一任务分别用聊天式提问和工具式协作做对比：

**聊天式**：在 Claude.ai 对话框里问"帮我分析一下我的项目结构"——它只能让你粘贴内容，然后给出泛泛的回答。

**工具式**：在 Claude Code CLI 里说同样的话——它会真的读你的目录、列出文件、分析依赖关系、给出具体的结构说明。

## 作业

学这门课之前先问自己三个问题，写下答案保存好：

1. **我最想用 Claude Code 解决什么问题？** 越具体越好，不要说"提高效率"，要说"每周要处理 30 个竞品报告，现在手动要 3 天"。
2. **我更倾向于在终端工作，还是文件夹/界面工作？** 这决定了你应该先学 CLI 还是 Cowork。
3. **我现在最常用什么工具？** 知道自己的工具栈，才能判断哪些地方可以接入 MCP。

## 常见误区

1. **把 Claude Code 当聊天框用**：它是执行代理，不是问答机器人。给它任务，不是给它问题。
2. **以为 Cowork 就是 Claude Code 的网页版**：它们的交互方式和最适合的任务完全不同。
3. **觉得 Claude Code 什么都能做**：它有明确的能力边界，理解边界才能用好它。

## 验收标准

学员看完后能说出 Claude Code 的 3 个关键能力边界，能判断自己应该先学 CLI 还是 Cowork。
