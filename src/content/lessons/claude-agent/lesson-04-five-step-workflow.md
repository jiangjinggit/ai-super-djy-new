## 学习目标

- 理解为什么需要固定的执行流程，而不是直接说"帮我做 XXX"
- 掌握五步执行法：读→计划→执行→验证→总结，每步都有具体的提示词
- 学会写"不做什么"清单，用禁止事项锁住 AI 的自由度

## 适合谁

所有已经能启动 Claude Code 或 Cowork 的用户，尤其是经常遇到"AI 反复返工"问题的人。

## 前置条件

- 完成第 2 课的安装与首次验证任务
- 完成第 3 课，知道自己应该用哪个入口

## 核心知识

### 为什么需要固定的执行流程

大多数人第一次用 Claude Code 时是这样的：

1. 脑子里有个任务
2. 直接打开，说"帮我做 XXX"
3. 它开始做，做到一半发现方向不对
4. 你打断它，重新说
5. 反复几次，花了 2 小时，得到的结果还没手动做的准

问题不是模型不行，是**指挥方式不对**。固定流程的价值是：把自由度锁在正确的位置，减少你和 AI 的"方向对齐"成本。

### 五步执行法

**第一步：读（理解现状）**

在开始任何修改之前，先让 Claude 读相关文件，说出它的理解。

```
先读一下 src/api/user.ts 和 src/services/auth.ts，
告诉我：
1. 这两个文件分别在做什么
2. 它们之间有什么关联
先别改，我需要确认你理解正确。
```

为什么这一步不能省：Claude 经常会在没完全理解结构的情况下开始修改，导致改了一个地方、破了另一个地方。

**第二步：出计划（确认范围）**

让它说出做法，你来审批，然后再执行。

```
好，理解准确。现在给我一个修改计划：
- 你打算改哪些文件
- 每个文件改什么
- 哪些地方你不确定，需要我确认

不要现在就改，先给我计划。
```

这一步能帮你发现 90% 的"跑偏"问题——在它动手之前。

**第三步：执行（范围内推进）**

计划确认后，再给执行指令。同时，主动限制不做什么。

```
计划没问题。按这个计划改，但注意：
- 不要重构其他文件
- 不要引入新的依赖
- 不要修改接口的入参和出参
```

**第四步：验证（确认结果正确）**

修改完成后，让它帮你验证，不要全靠眼看。

```
改完了，帮我做以下验证：
1. 读一遍改过的文件，说出你改了什么
2. 运行一下测试命令，看有没有报错
3. 告诉我这次改动有没有潜在风险
```

**第五步：总结（提取可复用知识）**

任务完成后，让它输出一份简短的总结。

```
任务完成，给我一个简短总结：
- 改了什么
- 为什么这样改
- 有没有遗留问题或待确认的地方
```

这个总结有两个用途：一是给你做记录，二是如果后续任务需要接续，这段总结可以作为新任务的上下文。

### "不做什么"清单比目标更重要

一个任务描述里，禁止事项往往比目标更关键。

**通用禁止清单（可以直接复制用）：**

```
禁止事项：
- 不重构不相关的代码
- 不引入新的第三方依赖
- 不修改接口签名（入参、出参、类型）
- 不删除任何文件，除非我明确指定
- 不修改配置文件（.env、vite.config、tsconfig 等）
- 不向外部发送任何网络请求
- 遇到不确定的地方，先问我，不要自己决定
```

**经验**：把你上次"它改错了"的地方，加进你的禁止清单。这个清单越用越准。

## Demo

### 修复一个用户登录后跳转失败的 Bug——完整 5 步案例

```
# 第一步：读
读一下 src/pages/Login.tsx 和 src/router/index.ts，
告诉我登录成功后的跳转逻辑是怎么实现的。

# 第二步：出计划（等它回答后）
根据你的理解，给我一个修复计划。
只改跳转相关的逻辑，不要碰表单验证部分。

# 第三步：执行（计划确认后）
按计划修改。禁止事项：不改表单逻辑、不改路由配置。

# 第四步：验证
改完后读一遍改动，然后帮我想三个可能的边界情况，
比如"用户直接访问登录页但已经登录了"。

# 第五步：总结
给我一份两三句话的修改说明，我要粘贴进 commit message。
```

## 作业

找一个你手头最近的真实任务，用 5 步执行法跑一遍，对比一下和之前的方式有什么不同。记录下来：

1. 哪一步帮你避免了返工？
2. 禁止清单里你加了什么？
3. 总结输出是否可以直接复用？

## 常见误区

1. **省掉"读"直接让它改**：Claude 经常在没完全理解结构的情况下开始修改，导致改了一个地方、破了另一个地方。"读"是最容易被省掉、但最不该省的一步。
2. **不写禁止事项导致"顺便优化"**：不给禁止清单，Claude 会自作主张重构代码、引入新依赖、修改不相关的文件。它不是故意的，是因为你没告诉它边界在哪。
3. **不做验证就提交**：改完直接用，不让它自检也不看 diff。等到线上出问题才发现改错了，回退成本远大于多花 2 分钟验证。
4. **一个对话塞太多任务**：把修 Bug、加功能、写文档全放在一个对话里，上下文越来越乱，后面的任务质量直线下降。一个任务一个对话，用 `/clear` 隔开。

## 验收标准

学员能独立按 5 步结构（读→计划→执行→验证→总结）写出一个完整的任务指令，包含明确的禁止事项清单。
