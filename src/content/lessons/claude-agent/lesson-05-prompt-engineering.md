## 学习目标

- 掌握五个最有效的 Prompt 技巧：XML 标签、给"为什么"、Few-shot、长文档顺序、先思考再输出
- 能识别常见的烂 Prompt 并知道怎么改
- 掌握结构化 Prompt 模板：背景→任务→输入→要求→禁止→示例

## 适合谁

所有使用 Claude Code 或 Cowork 的用户，尤其是觉得"AI 总是不听话"或"输出质量不稳定"的人。

## 前置条件

- 完成第 4 课，已经理解 5 步执行法的基本流程
- 至少用 Claude Code 或 Cowork 完成过一个真实任务

## 核心知识

### 五个最有效的 Prompt 技巧

**技巧一：用 XML 标签组织结构**

Claude 对 XML 标签有很好的理解，用标签把不同类型的信息分开，可以大幅减少歧义。

```
# 不好的写法
帮我分析这段代码，找出 Bug，然后修复，用中文解释原因，别改其他地方

# 好的写法
<任务>分析下面这段代码，找出 Bug 并修复</任务>

<代码>
// 粘贴代码
</代码>

<要求>
- 用中文解释 Bug 的原因
- 只改有问题的部分
- 给我一个修改前后的对比
</要求>

<禁止>
不要改函数签名，不要引入新依赖
</禁止>
```

**技巧二：给"为什么"，不只给"什么"**

Claude 对"有动机的指令"反应比纯规则更好。解释你为什么要这样做，它会理解得更准。

```
# 不好：我需要你每次都先读文件再改
# 好：我需要你每次都先读文件再改，因为这个项目文件之间耦合很紧，
#     之前没读就改导致过几次事故
```

**技巧三：Few-shot 示例**

如果你有一个特定的输出格式要求，给它一个例子，比描述一百遍都有效。

```
<任务>给下面每个函数写注释</任务>

<示例格式>
/**
 * 计算用户的剩余额度
 * @param userId 用户 ID
 * @param month 查询月份 (YYYY-MM)
 * @returns 剩余额度（单位：元）
 */
</示例格式>

<代码>
// 你的代码
</代码>
```

**技巧四：长文档放前面，问题放后面**

Anthropic 测试发现：在给 Claude 一篇长文档后提问，把问题放在**文档后面**，性能比放前面高 30%。

```
# 低效写法
总结一下这篇文章的核心论点：[文章内容]

# 高效写法
[文章内容]

---
请总结上文的核心论点
```

**技巧五：让它思考后再输出**

对于复杂的判断类任务，让 Claude 先说思考过程，再给结论，准确率会明显更高。

```
帮我判断这段 SQL 有没有性能问题。
先列出你注意到的所有可能有问题的地方，
再根据这个分析给出你的结论和建议。
```

### 常见烂 Prompt 和改法对照表

| 烂 Prompt | 问题 | 改法 |
|----------|------|------|
| "帮我优化代码" | 太模糊，不知道优化什么 | "帮我减少这段代码里的重复逻辑，保持功能不变" |
| "用最好的方式实现" | "最好"主观，会过度工程化 | "用最简单的方式实现，不考虑扩展性" |
| "帮我写完这个功能" | 边界不清，容易越界 | "实现登录表单的客户端验证，不涉及后端逻辑" |
| "改一下" | 什么都没说 | 给具体文件名、具体行数、具体期望结果 |
| 连续几段长对话后说"继续" | 上下文可能已经失真 | 新任务开新对话，重新给上下文 |

### 结构化 Prompt 模板

```
<背景>
[项目/任务的简短背景，3 句话以内]
</背景>

<任务>
[你想让它做什么，一句话概括]
</任务>

<输入>
[相关文件路径、代码片段、或文档内容]
</输入>

<要求>
- [要求 1]
- [要求 2]
- [输出格式要求]
</要求>

<禁止>
- [不要做的事 1]
- [不要做的事 2]
</禁止>

<示例>（可选）
[期望输出的一个例子]
</示例>
```

## Demo

### 烂 Prompt 到好 Prompt 的改写前后对比

**改写前（烂 Prompt）：**

```
帮我写一个用户注册的接口
```

**改写后（好 Prompt）：**

```
<背景>
这是一个 Node.js + Express 项目，使用 TypeScript，数据库是 PostgreSQL。
目前已有登录接口 src/api/auth/login.ts 可以参考风格。
</背景>

<任务>
新增一个用户注册接口
</任务>

<输入>
参考文件：src/api/auth/login.ts
数据库模型：src/models/User.ts
</输入>

<要求>
- 接口路径：POST /api/auth/register
- 入参：email, password, nickname
- 密码用 bcrypt 加密后存储
- 注册成功返回 userId，不返回 token
- 错误处理参照 login.ts 的风格
</要求>

<禁止>
- 不要修改 login.ts
- 不要引入新的依赖（bcrypt 已经装了）
- 不要自动发送验证邮件
</禁止>

<示例>
成功响应格式：
{ "code": 200, "data": { "userId": "xxx" } }
</示例>
```

## 作业

把你上周写过的一个 Prompt 用结构化模板重写一遍。对比两个版本：

1. 原来的 Prompt 缺了哪些信息？
2. 重写后 Claude 的输出质量有没有变化？
3. 你觉得哪个部分对输出质量影响最大？

## 常见误区

1. **以为措辞比结构重要**：很多人花时间琢磨"用什么词 Claude 更听话"，但真正影响输出质量的是信息结构——背景、约束、示例是否齐全，远比措辞重要。
2. **不给上下文直接提要求**：上来就说"帮我写个函数"，不说项目背景、技术栈、代码风格。Claude 只能猜，猜错了就是返工。
3. **不用禁止项**：只说"要做什么"不说"不要做什么"，Claude 会自作主张加功能、改风格、引入依赖。禁止项是控制边界最有效的手段。
4. **长对话后说"继续"而不是开新对话**：对话越长，上下文越模糊。新任务应该开新对话，把关键上下文重新给一遍，而不是在旧对话里说"继续"。

## 验收标准

学员能把一个模糊需求重写成结构化指令，包含背景、任务、输入、要求、禁止五个部分，并能解释每个部分为什么重要。
