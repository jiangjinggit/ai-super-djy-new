## 核心知识

### 四种长期上下文载体及分工

Claude Code 现在有四种常见的长期上下文来源：

| 载体 | 作用 | 适合放什么 |
|------|------|-----------|
| `~/.claude/CLAUDE.md` | 全局偏好 | 语言偏好、输出风格、通用禁令 |
| 项目根目录 `CLAUDE.md` | 项目级规则 | 架构说明、命令、目录、禁止事项 |
| `.claude/rules/` | 更细粒度规则 | 某类文件、某种任务、某个子目录的专门规则 |
| Auto Memory | 长期协作记忆 | 你反复强调的习惯、团队偏好、历史经验 |

**关键区别：**
- `CLAUDE.md` 和 rules 是你主动写的
- Auto Memory 是 Claude 逐步总结的
- 它们都是 context，不是硬编码开关——写得越具体、越短、越一致，遵循度越高

### 项目级 CLAUDE.md 七段标准结构

一份好用的 `CLAUDE.md` 通常包含以下七类信息：

````markdown
# 项目背景

[2-3 句话：这是什么项目，用来做什么，谁在用]

# 常用命令

```bash
npm run dev       # 启动开发服务器
npm run build     # 构建生产版本
npm run test      # 运行测试
npm run lint      # 类型检查
```

# 目录结构

[列出关键目录和它们的职责]

# 编码规范

- 语言：TypeScript 严格模式
- 格式化：使用 Prettier，不要手动调格式
- 注释：只在非自明的逻辑处写注释
- 命名：函数用 camelCase，组件用 PascalCase

# 禁止事项

- 不引入新的第三方依赖（需要先问我）
- 不修改 .env 文件
- 不改动 src/api/ 目录下的接口签名
- 不删除任何文件，除非我明确要求

# 当前重点（定期更新）

[本周在做的功能/修复的问题，帮助 Claude 快速理解当前重点]

# 协作约定

- 复杂任务先出计划，再执行
- 修改后先自检，再总结
- 不确定的地方先提问，不要自己猜
````

### .claude/rules/ 细粒度拆分

当一个项目越来越大时，不要把所有东西都堆进根目录 `CLAUDE.md`。更好的做法是：

- 根目录 `CLAUDE.md` 放全局项目规则
- `.claude/rules/frontend.md` 放前端规则
- `.claude/rules/backend.md` 放后端规则
- `.claude/rules/review.md` 放审查标准

好处：冲突更少，维护更容易，也更方便团队协作。

### Auto Memory 使用边界

**适合放进 memory 的内容：**
- 你长期坚持的输出格式
- 团队反复强调的工程习惯
- 某个项目里一再出现的约束和偏好

**不适合放进 memory 的内容：**
- 本周临时需求
- 很快过期的版本号、分支名、排期
- 带有密钥、凭证、敏感数据的内容

**经验法则：**
- "这周的任务"放 `CLAUDE.md`
- "这个团队一直这样做"放 memory
- "只影响一个模块"放 rules

### Cowork 的 Global/Project Instructions 配置实战

**Global Instructions 模板（适合大多数知识工作者）：**

```
你是我的工作助手。我的工作习惯如下：

输出偏好：
- 中文输出，专业词汇保留英文
- 结构化输出（用标题、列表而不是大段文字）
- 先结论，再解释
- 要点不超过 5 条，超过则归类

操作习惯：
- 没有我的确认，不要删除任何文件
- 任何写操作执行前，先告诉我你准备做什么
- 不确定的地方，直接问我，不要猜

当前工作重点：
[每周更新这里]
```

**Project Instructions 模板（以产品经理的竞品分析项目为例）：**

```
项目背景：
这个 Project 用于竞品分析工作。主要资料在"竞品资料"文件夹。

文件组织规范：
- 竞品资料/：原始截图和素材
- 分析报告/：整理后的分析文档
- 格式模板/：输出时参照这里的格式

输出要求：
- 每份分析报告参照"格式模板"里的结构
- 分析维度：功能、用户体验、定价、市场定位
- 数据来源必须注明（截图文件名或网址）

禁止事项：
- 不要在"竞品资料"文件夹写入任何新文件
- 报告草稿保存在"分析报告/草稿"子目录
```

### 维护上下文的四个好习惯

1. **发现一次问题，立刻更新**：它做错了一件事，把禁止条件加进去，而不是每次都重新说
2. **每月回顾一次**：清掉已经过期的"当前重点"，加入新的协作约定
3. **把信息放对地方**：项目事实进 `CLAUDE.md`，长期偏好进 memory，局部规则进 rules
4. **避免互相冲突**：同一件事不要在全局、项目、rules 里写三种版本

## Demo

### 从零写一份项目级 CLAUDE.md

假设你有一个 React + TypeScript 前端项目，以下是从零开始写 CLAUDE.md 的过程：

**第一步**：让 Claude 先读项目结构

```
读一下这个项目的目录结构和 package.json，
告诉我你觉得这个项目在做什么、用了什么技术栈。
```

**第二步**：基于它的理解，开始写 CLAUDE.md

```
根据你刚才的分析，帮我生成一份 CLAUDE.md 初稿，
按照这个结构：项目背景、常用命令、目录结构、编码规范、禁止事项、当前重点、协作约定。
每个部分简洁，不要超过 5 行。
```

**第三步**：你来审核和补充

- 检查常用命令是否准确
- 补充它不知道的禁止事项（比如"不要改 .env"）
- 加入当前重点（比如"本周在做用户权限模块"）

**第四步**：保存到项目根目录，后续持续维护

## 作业

给你最常用的一个项目写一份 CLAUDE.md 或 Project Instructions：

1. 用七段标准结构写一份初稿
2. 让 Claude 读你的项目后给出补充建议
3. 合并后保存，下次任务时观察效果

## 常见误区

1. **把任务说明全塞进 CLAUDE.md**：CLAUDE.md 是"员工手册"，不是"今天的工单"。项目背景、规范、禁令放这里；具体任务在对话里说。
2. **只写愿景不写禁区**：写了"代码要高质量"但没写"不要引入新依赖"。愿景太模糊，禁区才是真正能约束行为的东西。
3. **写完就不管**：CLAUDE.md 不是一次性文档。项目在变，规则也要跟着变。至少每月回顾一次。
4. **全局和项目规则互相冲突**：全局 CLAUDE.md 说"用英文输出"，项目 CLAUDE.md 说"用中文输出"。Claude 不知道听谁的，输出就不稳定。保持一致，或者在项目级明确覆盖全局设置。
