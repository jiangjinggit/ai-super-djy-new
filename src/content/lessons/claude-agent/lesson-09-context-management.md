## 学习目标

- 理解上下文窗口为什么会很快被吃满，以及哪些操作消耗最大
- 掌握四个主动管理上下文的技巧
- 能识别上下文污染的四个信号并采取正确动作

## 适合谁

所有使用 Claude Code 或 Cowork 处理中长任务的用户，尤其是遇到过"Claude 越用越差"问题的人。

## 前置条件

- 完成第 4-8 课，已经掌握核心工作流、CLAUDE.md、Skills 和 MCP
- 至少经历过一次"长对话后 Claude 质量明显下降"的情况

## 核心知识

### 理解上下文窗口

不要死记具体 token 数字，更重要的是知道它为什么会很快被吃满：

- **大文件**：每次读一个大文件，都会迅速占用上下文
- **报错堆栈**：一个完整的错误堆栈可能就占掉大量空间
- **测试输出**：跑测试的完整输出往往很长
- **子 Agent**：子 Agent 的对话也会消耗主上下文
- **MCP**：MCP 工具的输入输出都会进入上下文
- **长日志**：日志文件、构建输出等超长文本

Claude Code 会自动做上下文整理，但这个过程天然是有损的。任务越长，早期信息越容易被压缩、摘要化、边缘化。

结论很简单：**不要把关键规则只说在对话里。**

### 四个主动管理技巧

**技巧一：任务完成后 `/clear` 开新对话**

一个任务一个对话。不要让所有任务都堆在同一个上下文里。

```
# 规则
修复一个 Bug → /clear
开始下一个功能 → 新对话，重新给上下文
```

**技巧二：用 CLAUDE.md/rules/memory 分担重复说明**

你每次都要重新解释的上下文，不应该一直堆在会话里。

判断方法：
- 重复 3 次以上的项目约束 → 写进 `CLAUDE.md`
- 长期有效的个人偏好 → 写进 memory
- 只影响某一类任务的规则 → 写进 `.claude/rules/`

**技巧三：长任务用分段交付模式**

把大任务切成多个里程碑，每个里程碑完成后让它输出总结，用总结开启下一段。

```
# 大任务：重构整个用户模块

# 对话 1：分析现有结构
分析 src/user/ 目录，给我一份重构建议报告。
[任务完成后，保存报告]

# 对话 2：执行重构（用报告作为上下文）
这是我们上次分析的重构建议：[粘贴报告]
按照第 2 条建议，先重构 UserService。
```

**技巧四：善用 `/compact` 和 `/context`**

当对话较长但任务还没完成时：
- 用 `/compact` 主动压缩上下文
- 用 `/context` 检查当前会话到底加载了哪些信息
- 发现上下文污染时，及时 `/clear`

### 上下文重置的四个信号

当你观察到以下情况，应该考虑重置上下文：

1. **重复分析**：它开始重复之前说过的分析，像是忘了已经讨论过
2. **忘记约束**：它"忘记"了你之前确认的约束条件，开始做你禁止的事
3. **建议矛盾**：它给的建议和 20 分钟前的建议明显矛盾
4. **质量下降**：回复质量明显下降，开始模糊化处理，不再给出具体的文件名和代码

## Demo

### 一个长任务的分段交付过程

**场景**：重构一个包含 20 个文件的用户模块

**对话 1：分析阶段**

```
读取 src/user/ 目录下的所有文件，给我一份分析报告：
1. 每个文件的职责
2. 文件之间的依赖关系
3. 你发现的代码问题（重复逻辑、过时写法、缺少类型）
4. 建议的重构优先级

输出成一份结构化报告，我要保存下来。
```

→ 保存报告到 `reports/user-refactor-plan.md`
→ `/clear`

**对话 2：执行第一批重构**

```
这是用户模块的重构计划：[粘贴报告的前半部分]

现在执行第一批：重构 UserService 和 UserRepository。
禁止事项：不改 UserController，不改接口签名，不引入新依赖。
```

→ 完成后让它输出变更总结
→ `/clear`

**对话 3：执行第二批重构**

```
上一轮重构的变更总结：[粘贴总结]

现在执行第二批：重构 UserController，适配上一轮的变更。
```

**关键点**：每次新对话都带上前一轮的总结作为上下文，而不是让 Claude 在一个超长对话里记住所有东西。

## 作业

回顾一次你觉得 Claude 后期"越来越差"的任务，判断问题出在哪：

1. 是上下文太长导致的？（对话超过 30 轮？读了很多大文件？）
2. 是关键规则只说在对话里没写进配置？
3. 是多个任务堆在同一个对话里？
4. 如果重新来一次，你会怎么拆分这个任务？

## 常见误区

1. **把关键规则只说在对话里不写进配置**：对话里说的规则会随着上下文压缩而丢失。真正重要的规则应该写进 `CLAUDE.md`、rules 或 memory，这样每次新对话都能自动加载。
2. **所有任务堆在同一个对话里**：修 Bug、加功能、写文档全在一个对话里，上下文越来越乱。一个任务一个对话，用 `/clear` 隔开，是最简单有效的管理方式。
3. **不用 `/compact` 和 `/context`**：很多人不知道这两个命令的存在。`/compact` 可以在任务中途主动压缩上下文，`/context` 可以检查当前加载了什么。养成定期检查的习惯。
4. **以为 Claude 永远不会忘**：Claude 的上下文窗口是有限的，超长对话里早期信息一定会被压缩。不要假设它记得你 30 分钟前说的话，重要的事情要持久化到配置文件里。

## 验收标准

学员能识别上下文污染的四个信号（重复分析、忘记约束、建议矛盾、质量下降），并能采取正确动作（/clear、分段交付、持久化规则到配置）。
