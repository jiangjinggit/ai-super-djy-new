## 核心知识

### 什么时候需要多 Agent？——判断标准

**不需要多 Agent 的情况（先做好单 Agent）：**
- 任务是线性的，一步一步来就行
- 任务范围在一个目录内
- 你还在熟悉 Claude Code 阶段

**适合多 Agent 的情况：**
- 任务可以拆成互相独立的子任务，并行处理
- 单一 Agent 的上下文窗口装不下所有需要的信息
- 需要一个"质检员"角色来检查主 Agent 的输出
- 任务有明确的分工边界（研究/撰写/审查/发布）

### Claude Code 里最值得先学的三种用法

**用法 1：Plan Mode**

复杂任务先进入 `plan`，只做分析和规划，不做改动。

```
这是一个复杂的重构任务。
先进入规划模式：
- 读相关文件
- 输出完整步骤
- 标出风险点和待确认问题
先不要改任何代码。
```

**用法 2：Subagents**

把不同类型的子任务分给不同 Agent，例如：
- 一个负责探索代码库
- 一个负责实现
- 一个负责审查风险

**用法 3：Agent Teams**

让 Claude 把一个大任务拆成多个后台子任务，再汇总结果。特别适合：
- 大仓库代码搜索
- 跨多个文档的综合分析
- 研究、撰写、审查的流水线协作

### 三种主要的多 Agent 架构

**架构 1：主从（Orchestrator + Workers）**

一个协调者 Agent 拆分任务，分发给多个执行者 Agent，最后汇总结果。

```
主 Agent：
"我需要写一份竞品分析报告，分析三家公司。
请分别启动三个子任务：
- 子任务 A：分析 A 公司产品功能
- 子任务 B：分析 B 公司产品功能
- 子任务 C：分析 C 公司产品功能
三个任务完成后，把结果汇总成一份对比报告。"
```

**架构 2：流水线（Pipeline）**

任务在多个 Agent 之间顺序流转，每个处理一段，结果传给下一个。

```
Agent 1（研究）→ Agent 2（撰写）→ Agent 3（审查）→ Agent 4（格式化）
```

**架构 3：写作+质检（Writer + Reviewer）**

一个 Agent 生成内容，另一个专门检查，形成内部复核机制。

```
写作 Agent："写一份技术方案文档"
↓
审查 Agent："读取上面的文档，检查：
1. 技术方案是否有遗漏的边界情况
2. 语言是否清晰
3. 结论是否有足够支撑"
```

### 新手四阶段路径

| 阶段 | 时间 | 重点 |
|------|------|------|
| 阶段 1 | 前两周 | 把单个 CLI 或 Cowork 用好，能稳定完成 5 步执行法 |
| 阶段 2 | 第三四周 | 加入 CLAUDE.md、rules，开始做 skill，减少重复描述 |
| 阶段 3 | 第二个月 | 接入 1-2 个 MCP，打通最常用的外部工具 |
| 阶段 4 | 稳定后 | 先学 plan 和简单 subagent，再逐渐引入 agent teams |

不要在单工具都没跑稳的时候直接上多智能体，会适得其反。

## 常见误区

- **误区 1：一上来就搞多 Agent**。单 Agent 都没用稳就上多智能体，会适得其反。先把 5 步执行法跑顺。
- **误区 2：把所有任务都拆成多 Agent**。线性任务、小范围任务用单 Agent 更高效，不要为了用而用。
- **误区 3：忽略 Plan Mode 直接执行**。复杂任务不先规划就动手，容易方向跑偏、返工成本高。
- **误区 4：跳过渐进路径**。四个阶段是有顺序的，跳阶段学习会导致基础不牢。
