## 核心知识

### 危险能力清单

Claude Code 能删文件、推代码、发请求，这些都是高价值能力——问题不是关闭它们，而是把触发条件和确认点设清楚。

| 操作 | 风险级别 | 建议策略 |
|------|---------|---------|
| 删除文件 | 高 | 必须人工确认，先备份 |
| git push | 高 | 必须人工确认，只允许特定分支 |
| 安装依赖包 | 中 | 列出清单后人工确认 |
| 写入生产数据库 | 高 | 禁止，使用测试环境 |
| 发送外部网络请求 | 中-高 | 事先说明目标地址，人工确认 |
| 修改环境配置文件 | 高 | 禁止，除非专项任务 |
| 批量重命名/移动文件 | 中 | 先列出变更清单，确认后执行 |

### 四层权限控制

真正稳定的权限控制不是一句"别乱改"，而是四层叠加：

**第一层：权限模式**
- `default`：新手默认，每步都确认
- `acceptEdits`：已验证流程，自动接受编辑
- `plan`：高风险改动，只规划不执行
- `auto`：已验证的低风险重复任务
- `dontAsk`：特定操作免确认
- `bypassPermissions`：只在隔离环境使用

**第二层：allow / ask / deny 规则**

哪些动作默认允许，哪些必须问，哪些直接禁止。

**第三层：sandbox / worktree**

把危险操作放进可隔离、可回退的环境。

**第四层：项目规则**

通过 `CLAUDE.md`、rules、hooks 定义项目级安全边界。

课程建议：
- 新手默认 `default`
- 高风险改动先 `plan`
- 已验证流程再 `acceptEdits` 或 `auto`
- `bypassPermissions` 只在隔离环境使用

### CLAUDE.md 安全边界模板

```markdown
# 安全边界

## 必须人工确认的操作（停下来等我说 OK 才能继续）
- git push（任何分支）
- 删除文件（哪怕是临时文件）
- 安装新的 npm/pip 依赖
- 修改 .env、.env.local 等环境配置文件
- 发送任何外部网络请求

## 禁止操作（不管什么情况都不能做）
- 访问 src/config/secrets.ts
- 操作 production 分支
- 执行 DROP TABLE 等破坏性 SQL

## 操作前必须完成
- 列出本次任务会修改的所有文件
- 如果涉及超过 5 个文件，先给我一个变更计划
```

### 黄金操作习惯

**任务前：备份**

```bash
git checkout -b backup/[时间]
# 或用 worktree 开一个隔离分支
```

**任务后：检查 diff**

```bash
git diff   # 看改了什么
git status # 看有没有意外的文件变动
```

**发现问题时：恢复**

```bash
# 先只恢复这次任务实际改过的文件，不要一把梭清空整个工作区
git diff --name-only
git restore path/to/file
git restore --staged path/to/file

# 如果错误已经提交，再用 git revert [hash]
```

### 核心原则

**不靠信任模型，靠最小化可恢复成本。**

真正稳定的 AI 编程习惯，不是让模型变得更可信，而是确保每一步操作都可以回退。在这个前提下，才值得逐步放开权限。

## Demo

把安全边界模板加进你的 CLAUDE.md，然后做一次完整的"备份→执行→检查"流程：

```bash
# 1. 备份
git checkout -b backup/demo-$(date +%Y%m%d)

# 2. 让 Claude 执行一个小任务（比如重命名一个文件）

# 3. 检查
git diff
git status

# 4. 确认没问题后合并，或发现问题后恢复
git restore path/to/file
```

## 常见误区

- **误区 1：完全信任 AI 不会出错**。任何 AI 都可能产生错误输出，安全靠的是流程和机制，不是信任。
- **误区 2：一刀切禁止所有危险操作**。问题不是关闭能力，而是设好触发条件和确认点。全禁等于没用 AI。
- **误区 3：跳过备份直接执行**。"这次应该没问题"是最危险的想法。备份成本极低，恢复成本极高。
- **误区 4：直接用 bypassPermissions**。这个模式只适合隔离环境，在正式项目中使用等于裸奔。
- **误区 5：只设权限不检查 diff**。权限控制是事前防线，diff 检查是事后验证，两者缺一不可。
