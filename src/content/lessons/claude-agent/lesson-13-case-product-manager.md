## 学习目标

- 学会设计产品经理场景下的 Cowork 工作区结构
- 掌握 Cowork Project Instructions 的编写方法
- 完整走通"原始资料整理→信息提炼→生成需求文档→研发评估对齐"四步工作流
- 理解 Cowork 与 CLI 协作的分工模式

## 适合谁

- 每周需要整理用户反馈、竞品动态并产出需求文档的产品经理
- 需要与研发团队频繁对齐需求的产品负责人
- 想用 AI 提升文档生产效率的非技术岗位知识工作者

## 前置条件

- 已熟悉 Cowork 的基本操作（文件读写、Project Instructions 配置）
- 了解 CLI 的基本用法
- 有实际的产品需求文档工作经验

## 核心知识

### 场景描述

你是一位产品经理，每周需要整理用户反馈、竞品动态，产出需求文档，同步给研发团队。以前这些工作常常要占掉 1-2 天，目标是用 Cowork + Claude Code 明显缩短整理和成稿时间，并减少返工。

### Workspace Project 结构

```
产品工作/
├── 原始资料/       ← 用户反馈截图、竞品截图
├── 周会纪要/       ← 会议记录
├── 需求文档/       ← 输出目录
└── 模板/          ← 各类文档模板
```

### Cowork Project Instructions

```
你是我的产品助手。每次任务开始前，先确认你读到了哪些文件。
输出的需求文档放在"需求文档"目录，参照"模板"目录里的格式。
不要修改"原始资料"和"周会纪要"里的文件。
```

### 四步工作流

**第一步：原始资料整理（Cowork）**

把用户反馈截图、竞品截图放进"原始资料"目录，会议记录放进"周会纪要"目录。确保 Project 结构清晰。

**第二步：信息提炼（Cowork）**

```
读取本周"原始资料"里的所有用户反馈文件，
归类整理：
1. 高频问题（出现 3 次以上）
2. 严重阻断问题（用户明确说影响使用）
3. 体验改进建议

每类最多 5 条，简洁描述，不需要保留原文。
```

**第三步：生成需求文档（Cowork）**

```
基于刚才整理的反馈，参照"模板/需求文档模板.md"，
生成本周需求规划文档：
- 优先级排序（高/中/低）
- 每个需求 3 行以内：背景、方案方向、成功标准
- 需要研发评估的标注"待研发确认"

保存到"需求文档/2026-W14-需求规划.md"
```

**第四步：研发评估对齐（CLI）**

把需求文档传给研发后，开发者可以用 CLI：

```
读取 需求文档/2026-W14-需求规划.md，
评估每个需求的技术实现方向和复杂度（简单/中等/复杂），
补充进文档里的"技术评估"字段。
```

### 效率对比（示例参考，不是承诺值）

| 步骤 | 以前手动 | 现在 AI 辅助 |
|------|---------|------------|
| 整理用户反馈 | 3-4 小时 | 30-60 分钟 |
| 生成需求文档初稿 | 2-3 小时 | 30-60 分钟 |
| 格式整理 | 30 分钟 | 10-20 分钟 |
| 研发评估对齐 | 异步半天到 1 天 | 30-60 分钟 |

影响效果的关键变量：
- 原始资料是否已经分门别类
- 模板是否成熟
- 团队是否接受先出初稿、再补技术评估的协作方式

## 作业

1. 按照课程中的结构创建你自己的"产品工作"Project，配置好 Project Instructions
2. 准备至少 3 份真实的用户反馈资料，放入"原始资料"目录
3. 创建一份"需求文档模板.md"放入"模板"目录，包含：需求名称、优先级、背景、方案方向、成功标准、技术评估字段
4. 完整走通四步工作流，生成一份真实的需求规划文档
5. 记录效率对比数据，判断总耗时是否明显缩短，以及瓶颈主要卡在哪一步

## 常见误区

- **误区 1：跳过 Project Instructions 直接开始**。没有 Instructions，Claude 不知道文件结构和输出规则，每次都要重复说明。
- **误区 2：让 AI 修改原始资料**。原始资料是输入源，必须保持不变。Instructions 里要明确写"不要修改原始资料"。
- **误区 3：一次性把所有步骤塞进一个 prompt**。四步工作流要分步执行，每步确认输出质量后再进入下一步。
- **误区 4：不做模板就让 AI 自由发挥**。没有模板约束，每次输出格式不一致，后续对齐成本更高。
- **误区 5：忽略研发评估环节**。需求文档不是产品经理单方面的输出，技术评估对齐是闭环的关键。
