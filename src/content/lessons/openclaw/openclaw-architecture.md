## 这节课你要拿到什么

- 工作区、Agent、渠道、节点四个对象的分工
- 什么时候该拆、什么时候该共享

## 先分清 4 个对象

| 对象 | 管什么 | 别搞混 |
| --- | --- | --- |
| 工作区 | 场景专用的规则、技能和资产 | 不是所有东西都该全局共享 |
| Agent | 执行任务的主体 | 一个 Agent 别包打天下 |
| 渠道 | 消息进出 | 渠道不等于能力 |
| 节点 | 设备级执行能力（浏览器、截图、文件） | 不只是多一个通知入口 |

真正决定系统能做什么的，不是接了几个渠道，而是工作区里加载了什么规则和技能、Agent 拿到了什么权限、节点能不能提供浏览器和文件访问。

## 什么时候拆，什么时候共享

| 情况 | 做法 |
| --- | --- |
| 多个场景都要提醒、搜索、摘要 | 放全局共享 |
| 某个场景有专用规则和技能 | 单独拆工作区 |
| 涉及外发、删改、生产环境 | 必须隔离 |
| 团队内不同角色要不同边界 | 拆不同 Agent 或工作区 |

一句话：能跨场景稳定复用的才共享，只服务一个场景的别急着全局化。

## 最常见的 4 个结构错误

| 错误 | 后果 |
| --- | --- |
| 把渠道当能力 | 消息能进来，事做不动 |
| 所有技能全局共享 | 第二个场景开始互相干扰 |
| 一个 Agent 包打天下 | 规则越来越长，谁都不稳 |
| 高风险和普通能力混放 | 排错和治理都失控 |
