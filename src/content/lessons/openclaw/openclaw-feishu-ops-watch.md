## 这节课你要拿到什么

- 飞书运营守夜台的完整配置路径
- P1/P2/P3 异常分级规则
- 飞书多维表格工单系统的字段结构

## 守夜台的核心：先分级，再通知

最怕两个极端：什么都提醒导致没人看，过滤太狠导致真正的问题漏掉。

## 配置步骤

### 1. 定义你的异常分级

先把这张表填完，再去配系统：

```markdown
### P1（立即处理，飞书群 + 可选 Telegram）
- [例如：系统不可用]
- [例如：大面积订单延迟]
- [例如：关键客户投诉]

### P2（当天跟进，飞书群摘要 + 待办）
- [例如：一般客户投诉]
- [例如：单个订单异常]

### P3（仅记录，不主动打扰）
- [例如：低优先级反馈]
- [例如：重复性已知问题]
```

### 2. 配置 Heartbeat 巡检

```json
{
  "heartbeat": {
    "interval": 1800,
    "prompt": "执行运营巡检。检查客服消息队列、订单异常日志、系统健康状态。\nP1：立即发飞书群提醒，指定责任人\nP2：写入待办摘要\nP3：只记 daily notes"
  }
}
```

运营场景建议 30 分钟巡检一次，比内容监控更频繁。

### 3. 在 AGENTS.md 里写守夜规则

```markdown
## 运营守夜台规则

### P1 处理流程
1. 立即发送飞书群消息，使用红色卡片
2. 消息包含：异常类型、影响范围、责任人、建议动作
3. 15 分钟内无人响应，升级到 Telegram
4. P1 不受静默时间限制

### P2 处理流程
1. 写入飞书群每日运营摘要
2. 同时写入多维表格工单
3. 值班人员当天内跟进

### P3 处理流程
1. 只记录到 daily notes
2. 每周汇总时统一回顾

### 升级规则
- P2 超过 24 小时未处理 -> 升级为 P1
- 同类 P3 一天内出现 5 次以上 -> 升级为 P2

### 静默规则
- 22:00-08:00 只发 P1
- 周末只发 P1
- 同一异常 1 小时内不重复提醒
```

### 4. 建飞书工单表（多维表格）

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| 异常类型 | 单选 | 退款 / 延迟 / 投诉 / 系统错误 |
| 影响范围 | 单选 | 单用户 / 单项目 / 批量 / 全局 |
| 优先级 | 单选 | P1 / P2 / P3 |
| 责任人 | 人员 | 谁必须接住这条异常 |
| 当前状态 | 单选 | 待处理 / 处理中 / 已解决 / 已忽略 |
| 触发时间 | 日期 | 系统自动填入 |
| 建议动作 | 文本 | 系统给出的第一步建议 |

飞书群提醒不是终点，这张工单表才是。

### 5. 手动测试

```bash
openclaw run --prompt "模拟一次运营巡检。假设发现 1 个 P1 级订单批量延迟、2 个 P2 级客户投诉、3 个 P3 级重复反馈。按守夜台规则分级处理并发送到飞书。"
```

确认：P1 收到红色卡片、P2 收到橙色卡片、P3 没有打扰你。

## 守夜台调不稳时看 3 个指标

| 指标 | 说明 | 修法 |
| --- | --- | --- |
| 误报率高 | P1 里有太多不紧急的 | 收紧 P1 条件 |
| 漏报率高 | 真正的问题没被抓到 | 扩大输入源，降低阈值 |
| 无人认领率高 | 异常发了但没人处理 | 每条 P1 必须指定责任人 |

## 本课落地动作

- 填完你自己的 P1/P2/P3 异常定义
- 配好 Heartbeat 巡检规则
- 在飞书里建好工单多维表格
