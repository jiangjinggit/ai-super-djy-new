## 这节课你要拿到什么

- 三件套（SOUL.md / USER.md / AGENTS.md）的分工标准
- 每个文件的可直接改的模板
- 三件套冲突时的优先级

## 三件套是什么，分别管什么

| 文件 | 管什么 | 典型内容 |
| --- | --- | --- |
| SOUL.md | 默认行为边界 | 角色定位、语气风格、决策权限、绝对禁止项 |
| USER.md | 长期稳定的个人背景 | 身份、项目、回复偏好、免打扰时段 |
| AGENTS.md | 场景化的操作规程 | 通知规则、确认机制、记忆写入、场景协议 |

一句话：SOUL.md 定边界，USER.md 给背景，AGENTS.md 管场景规则。

## SOUL.md：写边界，不是写人设

最没用的 SOUL.md 写得像品牌文案——"专业、友好、高效"。这些词几乎没有约束力。

SOUL.md 真正该回答的是：

- 指令不清时，先猜还是先确认？
- 哪些动作可以自主做？
- 哪些动作要先告知？
- 哪些动作必须停下来？

### 四段式结构

| 模块 | 该写什么 | 别写成什么 |
| --- | --- | --- |
| 角色定位 | 你是什么类型的助手 | 最懂我的超级 AI |
| 语气风格 | 回答长短、先结论还是先展开 | 自然一点、聪明一点 |
| 决策权限 | 什么能自主做、什么先告知、什么必须确认 | 高风险操作谨慎处理 |
| 绝对禁止项 | 删除、外发、安装等红线 | 注意安全 |

判断标准：**是不是具体到能观察、能复盘、能修正。**

### SOUL.md 起步模板

```markdown
# SOUL.md

## 角色定位
你是一个长期在线的个人工作台助手，优先帮助我处理信息汇总、提醒、监控和知识整理。

## 沟通风格
- 先给结论，再给依据
- 信息不够时先确认，不擅自猜测
- 默认简洁，除非我要求展开

## 决策权限
- 可自主执行：公开信息搜索、摘要、分类、日报整理
- 先告知再执行：新建日程、修改文档、发送团队内部通知
- 必须确认：删除内容、对外发送、改生产配置、访问敏感文件

## 绝对禁止项
- 不删除任何文件或数据
- 不向未授权渠道发送消息
- 不读取或暴露密钥
- 不安装未知依赖或启动未知进程
```

### 写完必须测

至少测 3 类：
1. 模糊任务：它会不会擅自猜？
2. 风险任务：它会不会越权？
3. 主动任务：它会不会把低价值信息当重要提醒？

## USER.md + AGENTS.md：规则要写成条件

| 差写法 | 好写法 |
| --- | --- |
| 重要消息及时提醒我 | 只有来自指定发件人、日历冲突、或命中关键字的才立即推送 |
| 关键信息记住就行 | 跨天仍有效的偏好写 MEMORY.md，当天状态只写 daily notes |
| 高风险动作先确认 | 删除、外发、改配置、访问敏感文件时必须确认 |

### USER.md 模板

```markdown
# USER.md
- 我的主要角色：
- 当前最重要的 1-3 个项目：
- 回复偏好：先结论 / 详细展开 / 表格优先
- 免打扰时间：
```

### AGENTS.md 模板

```markdown
# AGENTS.md

## 通知规则
- 日历冲突立即推送
- 常规邮件并入早报
- 低优先级信息只记日志

## 确认规则
- 删除、外发、改配置必须确认

## 记忆规则
- 跨天仍有效 → MEMORY.md
- 当天状态 → daily notes

## 场景协议
- 内容监控结果先入素材池，再决定是否推送
```

## 三件套冲突时的优先级

冲突时按这个顺序：

1. **SOUL.md 禁止项**（最高）
2. **AGENTS.md 场景规则**
3. **USER.md 默认偏好**
4. 系统默认行为（最低）

## 常见翻车

| 翻车 | 原因 |
| --- | --- |
| SOUL.md 写完没约束力 | 写的是感觉，不是可观察的边界 |
| USER.md 和 AGENTS.md 混写 | 背景信息和场景规则堆在一起 |
| 写完不测 | 后面出问题无法定位是规则问题还是部署问题 |

## ✅ 本课落地动作

- 写出 SOUL.md 四段式基础版，至少补 3 条明确权限边界
- 写出 USER.md 和 AGENTS.md 基础版
- 明确 3 条立即推送规则 + 3 条必须确认的动作
- 用 3 个真实问题测试 SOUL.md 是否越界
